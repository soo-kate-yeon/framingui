# デザイントークンのベストプラクティス

デザイントークンは「値の置き場」ではなく、UI 判断を共有するための契約です。ここでは運用で効くルールだけを整理します。

## 1. 命名は意味ベースで

`blue-500` のような見た目命名より、`bg-primary` のような役割命名を優先します。

## 2. レイヤーを分ける

- Primitive: 生の値
- Semantic: 役割
- Component: コンポーネント用途

この 3 層に分けると変更影響を局所化しやすくなります。

## 3. 変更単位を小さくする

大きなリネームは避け、互換期間を設けて段階的に移行します。

## 4. 配布形式を統一

- CSS Variables
- JSON
- TypeScript typings

同一ソースから複数形式へ出力できるようにすると運用が安定します。

## 5. 品質ゲートを作る

- 重複トークン検出
- 未使用トークン検出
- コントラストチェック

CI に組み込むと、負債を早期に防げます。

## まとめ

デザイントークン運用は、作成より保守が本体です。命名とレイヤー設計を最初に整えると、後続の機能開発が一気に楽になります。

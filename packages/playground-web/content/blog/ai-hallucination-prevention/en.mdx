## The AI Hallucination Problem You Didn't Know You Had

You ask Claude Code to create a button. It generates:

```tsx
<button className="bg-blue-600 hover:bg-blue-700 text-white px-6 py-3 rounded-lg">
  Click me
</button>
```

You ask again for a different component. Now it gives you:

```tsx
<button className="bg-indigo-500 hover:bg-indigo-600 text-white px-4 py-2 rounded-md">
  Submit
</button>
```

**Two buttons. Different blues. Different padding. Different border radius.**

This is AI hallucination in UI—and it's silently destroying your design consistency.

## What is AI Hallucination in UI?

AI hallucination traditionally refers to LLMs making up facts. But in UI code generation, there's a subtler, more insidious form:

**Design hallucination** — when AI invents visual design decisions without any grounding in your actual design system.

### The Three Types of UI Hallucination

1. **Color Hallucination** — AI picks random Tailwind colors (`blue-600`, `indigo-500`, `sky-400`)
2. **Spacing Hallucination** — Inconsistent padding/margins (`p-4`, `px-5 py-3`, `gap-6`)
3. **Pattern Hallucination** — Each component gets a unique, one-off implementation

The result? **Every AI-generated component looks different**, even when they should be identical.

## Why This Happens (The Technical Truth)

LLMs don't "see" your design system. They're trained on millions of open-source repositories with wildly different styles.

When you prompt:
```
Create a primary button
```

The AI's internal reasoning is:
- "Primary" usually means blue
- Blue could be `blue-500`, `blue-600`, `indigo-600`, `sky-500`...
- I'll pick `blue-600` because it appears most frequently in training data
- For padding, `px-6 py-3` is common... but so is `px-4 py-2`

**There's no grounding.** The AI is guessing based on statistical patterns, not your actual design intent.

## The Hidden Cost of Design Hallucination

### 1. Death by a Thousand Inconsistencies

Each AI-generated component introduces subtle variations:
- 15 different shades of blue across your app
- 8 different button sizes
- 12 different border radius values

Over time, your app looks like it was designed by a committee that never met.

### 2. The Refactoring Tax

Every hallucinated component needs manual fixing:
```tsx
// AI generated this
<div className="bg-blue-50 border border-blue-200 rounded-lg p-4">

// You need to change it to
<div className="bg-[var(--color-background-secondary)] border border-[var(--color-border-default)] rounded-[var(--radius-default)] p-[var(--spacing-4)]">
```

**Time lost:** 2-5 minutes per component. Multiply by 100+ components = hours of wasted time.

### 3. Design Debt Compounds

Three months later, you want to change your primary blue:
- Without design tokens: Find and replace 50+ hardcoded `blue-600` values
- With design tokens: Change one CSS variable

**Technical debt from AI hallucination is permanent unless you have guardrails.**

## The FramingUI Solution: Design Tokens as AI Guardrails

FramingUI solves AI hallucination with a simple principle:

**If AI can only generate code using design tokens, it can't hallucinate colors or styles.**

### How It Works

1. **Semantic Tokens Replace Raw Values**
   - Instead of `bg-blue-600` → `var(--bg-primary)`
   - Instead of `p-4` → `var(--spacing-4)`
   - Instead of `rounded-lg` → `var(--radius-lg)`

2. **Type-Safe Component API**
   - Button doesn't accept `className="bg-red-500"`
   - It accepts `variant="primary" | "secondary" | "destructive"`

3. **AI Instructions Baked Into Project Context**
   - `.claude/project-knowledge.md` teaches AI your design system
   - Every prompt automatically references your tokens

## Setting Up AI Hallucination Prevention

### Step 1: Install FramingUI

```bash
pnpm add @framingui/ui @framingui/core
```

### Step 2: Create Project Knowledge for Claude Code

Create `.claude/project-knowledge.md`:

```markdown
# FramingUI Design System Rules

## CRITICAL: Anti-Hallucination Rules

1. **NEVER hardcode colors** — Use `var(--*)` tokens only
2. **NEVER invent Tailwind colors** — No `bg-blue-600`, `text-red-500`, etc.
3. **ALWAYS import from @framingui/ui** — Never recreate components
4. **USE semantic variants** — `variant="primary"` not `className="bg-blue-600"`

## Available Design Tokens

### Colors (Semantic)
- `--bg-primary` — Primary brand color
- `--bg-secondary` — Secondary actions
- `--bg-destructive` — Errors/deletions
- `--bg-muted` — Muted backgrounds
- `--bg-accent` — Highlights
- `--text-primary` — Main text
- `--text-muted` — Secondary text

### Spacing (Consistent Scale)
- `--spacing-1` through `--spacing-16`
- Use: `p-[var(--spacing-4)]`

### Border Radius
- `--radius-sm`, `--radius-md`, `--radius-lg`, `--radius-xl`

## Component Library

Available from `@framingui/ui`:
- Layout: `Card`, `Sheet`, `Dialog`, `Tabs`, `Separator`
- Forms: `Button`, `Input`, `Label`, `Select`, `Switch`, `Checkbox`
- Data: `Table`, `Avatar`, `Badge`
- Feedback: `Toast`, `Tooltip`, `Progress`
```

## Measuring Success: Hallucination Audit

Run this command to check for design hallucinations:


## Common Mistakes (And How to Avoid Them)

### Mistake 1: Mixing Tokens with Hardcoded Values

```tsx
// ❌ BAD: Half tokens, half hardcoded
<div className="bg-[var(--bg-primary)] p-4 rounded-lg">
```

```tsx
// ✅ GOOD: All tokens
<div className="bg-[var(--bg-primary)] p-[var(--spacing-4)] rounded-[var(--radius-lg)]">
```

### Mistake 2: Not Teaching AI Your System

AI won't know to use tokens unless you tell it. Always include project knowledge or Cursor rules.

### Mistake 3: Accepting AI's First Output

Even with tokens, audit first few generations. Once AI learns your patterns, it'll be consistent.

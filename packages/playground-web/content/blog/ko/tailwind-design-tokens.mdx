Tailwind 프로젝트를 6개월 정도 운영하다 보면 대부분의 코드베이스에서 비슷한 증상이 나타나요. 파란색 계열 클래스가 스무 가지 넘게 섞여 있고, 버튼 패딩은 파일마다 제각각이고, 간격 값은 레이아웃마다 달라져 있죠. Tailwind가 문제가 아니에요. 결정의 부재가 문제입니다.

Tailwind는 유틸리티 툴킷이고, 디자인 시스템은 결정의 집합이에요. 이 둘은 다른 개념이거든요. Tailwind 없이 디자인 시스템을 운영할 수 있고, 디자인 시스템 없이 Tailwind를 쓸 수도 있습니다. 하지만 후자의 경우 시간이 갈수록 비일관성 부채가 쌓여요. 이 글에서는 Tailwind의 속도를 유지하면서 디자인 토큰으로 구조를 더하는 방법을 알려드릴게요.

## Tailwind만으로는 왜 부족할까요

Tailwind는 색상마다 22가지 shade를 제공해요. 이건 기능이지, 제약이 아니에요. 문제는 명확한 규칙이 없으면 같은 유형의 컴포넌트를 만들 때도 팀원마다 다른 shade를 선택한다는 거예요.

개발자 A는 primary 버튼에 `bg-blue-600`을 쓰고, 개발자 B는 다른 화면에서 `bg-blue-500`을 써요. 각자의 화면에서는 멀쩡해 보이지만, 나란히 놓으면 설계된 제품이 아니라 조립된 느낌을 주죠.

간격도 마찬가지예요. `px-4`, `px-5`, `px-6` 모두 유효한 Tailwind 클래스예요. 어떤 걸 어떤 상황에 쓸지 규칙이 없으면 같은 앱의 버튼들이 제각각의 패딩을 갖게 돼요.

다크 모드가 추가되면 문제가 더 심해져요. `bg-blue-600`을 라이트 모드에 써뒀다면, 모든 컴포넌트마다 다크 모드 equivalent를 수동으로 결정해야 하거든요. 이 둘을 연결해주는 시스템이 없으니까요.

## 디자인 토큰이 실제로 해결하는 것

디자인 토큰은 이름 붙은 결정이에요. `--color-brand-primary`가 특정 값에 매핑되는 거죠. 원시 클래스 대신 토큰을 참조하면 의도가 명확해져요.

```css
/* 이전: 이 파란색이 뭘 의미하지? */
bg-blue-600

/* 이후: 이건 primary 브랜드 액션 컬러다 */
bg-[var(--color-brand-primary)]
```

브랜드 색상을 바꿔야 한다면 값 하나만 업데이트하면 돼요. 그 토큰을 참조하는 모든 컴포넌트가 자동으로 바뀌죠. grep-and-replace도 없고, 놓친 인스턴스도 없어요.

시맨틱 네이밍은 의도도 전달해요. 개발자가 `bg-surface-elevated`를 보면 이게 모달이나 팝오버용이라는 걸 알 수 있어요. `bg-gray-50`을 보면 추측해야 하지만요.

## 두 레이어 아키텍처

가장 깔끔한 패턴은 Tailwind의 유틸리티 레이어 위에 토큰 레이어를 얹는 거예요.

**레이어 1: 베이스 팔레트**

```css
:root {
  --color-blue-600: #2563EB;
  --color-blue-700: #1D4ED8;
  --color-gray-50:  #F9FAFB;
  --color-gray-900: #111827;
}
```

**레이어 2: 시맨틱 별칭**

```css
:root {
  --color-brand-primary:       var(--color-blue-600);
  --color-brand-primary-hover: var(--color-blue-700);
  --color-surface-base:        var(--color-gray-50);
  --color-text-primary:        var(--color-gray-900);
}

[data-theme="dark"] {
  --color-surface-base:  var(--color-gray-900);
  --color-text-primary:  var(--color-gray-50);
}
```

**레이어 3: Tailwind config**

```javascript
// tailwind.config.js
export default {
  theme: {
    extend: {
      colors: {
        brand: {
          primary:      'var(--color-brand-primary)',
          primaryHover: 'var(--color-brand-primary-hover)',
        },
        surface: {
          base: 'var(--color-surface-base)',
        },
        text: {
          primary: 'var(--color-text-primary)',
        },
      },
    },
  },
}
```

이제 `bg-brand-primary`를 쓰면 Tailwind 클래스 경험을 유지하면서 아래에 토큰 시스템이 작동하는 거예요.

## 간격과 타이포그래피

색상부터 시작하는 개발자가 많지만, 같은 원리가 간격과 타이포그래피에도 적용돼요.

`px-4`와 `px-5` 중에 "더 나아 보이는 것"을 고르는 대신, 스케일을 정의하세요.

```css
:root {
  --spacing-sm: 0.75rem;  /* 12px */
  --spacing-md: 1rem;     /* 16px */
  --spacing-lg: 1.5rem;   /* 24px */
}
```

이걸 Tailwind에 매핑해요.

```javascript
theme: {
  extend: {
    spacing: {
      sm: 'var(--spacing-sm)',
      md: 'var(--spacing-md)',
      lg: 'var(--spacing-lg)',
    },
  },
}
```

이제 컴포넌트는 `px-4` 대신 `px-md`를 써요. 6개월 뒤에 코드를 보는 사람도 이게 medium 패딩이라는 결정임을 알 수 있죠. 임의적인 값이 아니라요.

## 다크 모드가 단순해져요

시맨틱 토큰을 쓰면 다크 모드는 CSS 변수 교체가 되지, 클래스 이름 감사가 아니에요.

```css
:root {
  --color-surface-base:     #FFFFFF;
  --color-surface-elevated: #F9FAFB;
  --color-text-primary:     #111827;
  --color-text-secondary:   #6B7280;
}

[data-theme="dark"] {
  --color-surface-base:     #111827;
  --color-surface-elevated: #1F2937;
  --color-text-primary:     #F9FAFB;
  --color-text-secondary:   #9CA3AF;
}
```

`bg-surface-base text-text-primary`를 쓰는 컴포넌트는 테마 속성이 바뀌면 자동으로 적응해요. JSX 전체에 `dark:` 접두사를 뿌릴 필요가 없죠.

## 전체 재작성 없이 마이그레이션하기

한꺼번에 다 리팩터링할 필요는 없어요. Tailwind의 `extend` 키 덕분에 기존 클래스는 그대로 작동하면서 토큰 기반 클래스를 나란히 도입할 수 있거든요.

코드베이스에서 가장 많이 쓰이는 색상을 찾는 것부터 시작하세요. 거기에 시맨틱 토큰을 정의하고, Tailwind config에 새 클래스 이름을 추가해요. 그 시점부터 새 컴포넌트에는 시맨틱 클래스를 써요.

다른 이유로 기존 파일을 수정할 때, 그 파일의 색상 클래스도 마이그레이션하면 돼요. 몇 달에 걸쳐 코드베이스가 자연스럽게 수렴해요. 별도의 리팩터링 스프린트가 필요 없어요.

## AI 도구와 더 잘 맞아요

Claude Code나 Cursor 같은 AI 도구가 UI 컴포넌트를 생성할 때, 일반적인 Tailwind 클래스를 패턴 매칭해요. 컨텍스트가 없으면 학습 데이터에서 흔한 `bg-blue-600`을 선택하죠. 당신의 브랜드 색상이라서가 아니라요.

디자인 시스템이 CSS 변수로 표현되고 Tailwind에 매핑되면, AI 도구에 명확한 지침을 줄 수 있어요. 원시 색상 클래스 말고 `bg-brand-primary`를 쓰라고요. 토큰 이름이 hex 값으로는 전달할 수 없는 의도를 담고 있거든요. 생성된 컴포넌트가 처음부터 시스템에 더 가까이 착지해요.

Tailwind의 속도는 생각과 코드 사이의 마찰을 없애는 데서 나와요. 디자인 토큰은 엔트로피가 쌓이는 걸 막는 구조를 더해줘요. 이 둘은 경쟁하는 게 아니라 서로를 보완하는 관계예요.

# AI 에이전트를 위한 디자인 토큰: 완벽 가이드

AI 에이전트는 코드를 작성할 수 있습니다. 하지만 일관되게 디자인할 수 있을까요?

답은 전적으로 디자인 토큰을 어떻게 구조화하느냐에 달려 있습니다. AI 에이전트에게 raw Tailwind 클래스를 주면, 컴포넌트마다 새로운 blue를 만들어낼 겁니다. 제대로 된 디자인 토큰을 주면, 디자인 시스템과 완벽하게 일치하는 UI를 만들어냅니다.

이 가이드는 AI 에이전트가 실제로 이해할 수 있는 디자인 토큰을 설계하는 방법과 hallucination을 방지하는 방법을 보여줍니다.

## 문제: AI는 당신의 디자인 시스템을 "보지" 못한다

Claude Code나 Cursor에게 버튼을 생성하라고 하면, AI는 당신의 브랜드 색상을 알지 못합니다. 학습 데이터를 기반으로 추측할 뿐입니다:

```tsx
// AI hallucination: 임의의 값을 만들어냄
<button className="bg-blue-600 hover:bg-blue-700 px-6 py-3 rounded-lg">
  제출
</button>

// 다음 컴포넌트: 다른 blue, 다른 spacing
<div className="bg-indigo-500 p-4 rounded-md">
  카드 내용
</div>
```

**모든 컴포넌트가 다르게 보이는 이유는 AI에게 단일 진실 공급원이 없기 때문입니다.**

## 해결책: AI가 읽을 수 있는 데이터로서의 디자인 토큰

디자인 토큰은 인간의 디자인 의도와 AI 코드 생성 사이의 간극을 연결합니다. 단순한 CSS 변수가 아니라 **AI가 쿼리하고, 이해하고, 일관되게 적용할 수 있는 구조화된 데이터**입니다.

### 토큰을 "AI가 읽을 수 있게" 만드는 것은?

1. **시맨틱 네이밍** — `blue-600` 대신 `color-primary`
2. **타입 정보** — 이게 색상인가? spacing 값인가? shadow인가?
3. **사용 컨텍스트** — 이 토큰은 언제 사용해야 하는가?
4. **계층적 구조** — Primitives → Semantic → Component 토큰

## 아키텍처: AI를 위한 3계층 토큰 시스템

AI 에이전트를 위한 가장 효과적인 토큰 아키텍처는 세 개의 레이어를 사용합니다:

### 1계층: Primitive 토큰 (원시 값)

디자인 시스템의 원재료입니다:

```typescript
// tokens/primitives.ts
export const primitives = {
  color: {
    blue: {
      50: 'oklch(0.97 0.02 250)',
      100: 'oklch(0.93 0.04 250)',
      500: 'oklch(0.55 0.15 250)',
      900: 'oklch(0.25 0.10 250)',
    },
    neutral: {
      50: 'oklch(0.98 0 0)',
      100: 'oklch(0.95 0 0)',
      900: 'oklch(0.20 0 0)',
      950: 'oklch(0.15 0 0)',
    }
  },
  spacing: {
    1: '0.25rem',   // 4px
    2: '0.5rem',    // 8px
    3: '0.75rem',   // 12px
    4: '1rem',      // 16px
    6: '1.5rem',    // 24px
    8: '2rem',      // 32px
  },
  fontSize: {
    xs: '0.75rem',
    sm: '0.875rem',
    base: '1rem',
    lg: '1.125rem',
    xl: '1.25rem',
    '2xl': '1.5rem',
  }
}
```

**AI에게 효과적인 이유:** AI는 사용 가능한 값의 전체 범위를 보고 스케일 시스템을 이해할 수 있습니다.

### 2계층: Semantic 토큰 (목적 기반)

primitive를 디자인 의도에 매핑합니다:

```typescript
// tokens/semantic.ts
export const semantic = {
  color: {
    // 배경
    backgroundPrimary: primitives.color.neutral[50],
    backgroundSecondary: primitives.color.neutral[100],
    backgroundInverse: primitives.color.neutral[950],
    
    // 텍스트
    textPrimary: primitives.color.neutral[900],
    textSecondary: primitives.color.neutral[600],
    textInverse: primitives.color.neutral[50],
    
    // 인터랙티브
    interactivePrimary: primitives.color.blue[500],
    interactivePrimaryHover: primitives.color.blue[600],
    interactiveDisabled: primitives.color.neutral[300],
    
    // 피드백
    feedbackSuccess: primitives.color.green[500],
    feedbackError: primitives.color.red[500],
    feedbackWarning: primitives.color.amber[500],
  },
  
  spacing: {
    componentGapSmall: primitives.spacing[2],
    componentGapMedium: primitives.spacing[4],
    componentGapLarge: primitives.spacing[6],
    
    insetSmall: primitives.spacing[2],
    insetMedium: primitives.spacing[4],
    insetLarge: primitives.spacing[6],
  },
  
  typography: {
    headingLarge: {
      fontSize: primitives.fontSize['2xl'],
      fontWeight: '700',
      lineHeight: '1.2',
    },
    headingMedium: {
      fontSize: primitives.fontSize.xl,
      fontWeight: '600',
      lineHeight: '1.3',
    },
    bodyDefault: {
      fontSize: primitives.fontSize.base,
      fontWeight: '400',
      lineHeight: '1.5',
    },
    bodySmall: {
      fontSize: primitives.fontSize.sm,
      fontWeight: '400',
      lineHeight: '1.4',
    }
  }
}
```

**AI에게 효과적인 이유:** 토큰 이름이 구현(blue, gray)이 아닌 *의도*(primary, secondary, disabled)를 설명합니다. AI는 컴포넌트의 목적을 기반으로 어떤 토큰을 사용할지 판단할 수 있습니다.

### 3계층: Component 토큰 (특정 사용 사례)

컴포넌트별 오버라이드와 조합:

```typescript
// tokens/components.ts
export const components = {
  button: {
    // Primary 변형
    primaryBackground: semantic.color.interactivePrimary,
    primaryBackgroundHover: semantic.color.interactivePrimaryHover,
    primaryText: semantic.color.textInverse,
    primaryPadding: `${semantic.spacing.insetSmall} ${semantic.spacing.insetMedium}`,
    primaryBorderRadius: primitives.borderRadius.default,
    
    // Secondary 변형
    secondaryBackground: 'transparent',
    secondaryBorder: semantic.color.interactivePrimary,
    secondaryText: semantic.color.interactivePrimary,
    
    // 상태
    disabledBackground: semantic.color.interactiveDisabled,
    disabledText: semantic.color.textSecondary,
  },
  
  card: {
    background: semantic.color.backgroundSecondary,
    border: semantic.color.borderDefault,
    padding: semantic.spacing.insetLarge,
    borderRadius: primitives.borderRadius.large,
    shadow: primitives.shadow.medium,
  },
  
  input: {
    background: semantic.color.backgroundPrimary,
    border: semantic.color.borderDefault,
    borderFocus: semantic.color.interactivePrimary,
    text: semantic.color.textPrimary,
    placeholder: semantic.color.textSecondary,
    padding: semantic.spacing.insetSmall,
  }
}
```

**AI에게 효과적인 이유:** 버튼을 생성할 때, AI는 `components.button.*`를 쿼리해서 정확한 값을 얻습니다. 추측 없음. Hallucination 없음.

## 예시: AI 에이전트가 토큰 사용하기

AI 에이전트(MCP 또는 직접 API를 통해)가 토큰을 사용해 버튼 컴포넌트를 생성하는 방법:

### 디자인 토큰 없이 (Hallucination)

```tsx
// AI가 임의의 값을 만들어냄
function Button({ children }) {
  return (
    <button className="bg-blue-600 hover:bg-blue-700 text-white px-6 py-3 rounded-lg font-semibold">
      {children}
    </button>
  )
}
```

### 디자인 토큰과 함께 (Grounded)

```tsx
// AI가 토큰 시스템을 쿼리해서 일관된 코드 생성
import { components } from '@/tokens/components'

function Button({ children, variant = 'primary' }) {
  return (
    <button 
      style={{
        background: components.button.primaryBackground,
        color: components.button.primaryText,
        padding: components.button.primaryPadding,
        borderRadius: components.button.primaryBorderRadius,
      }}
      onMouseEnter={(e) => {
        e.currentTarget.style.background = components.button.primaryBackgroundHover
      }}
    >
      {children}
    </button>
  )
}
```

**AI가 생성한 모든 버튼이 동일하게 보이는 이유는 동일한 토큰을 참조하기 때문입니다.**

## Hallucination 방지: 아키텍처 패턴

### 패턴 1: 토큰 검증 레이어

토큰 스키마에 대해 검증하여 AI가 값을 만들어내지 못하도록 방지:

```typescript
// tokens/validate.ts
import { primitives, semantic, components } from './index'

export function validateToken(tokenPath: string): boolean {
  const [tier, category, ...rest] = tokenPath.split('.')
  
  const tokenMap = {
    primitives,
    semantic,
    components,
  }
  
  let current = tokenMap[tier]
  for (const key of [category, ...rest]) {
    if (!current?.[key]) {
      throw new Error(`Invalid token: ${tokenPath}`)
    }
    current = current[key]
  }
  
  return true
}

// AI 프롬프트에서 사용
const tokenPath = 'semantic.color.interactivePrimary'
if (validateToken(tokenPath)) {
  // 안전하게 사용 가능
}
```

### 패턴 2: 토큰 자동완성 컨텍스트

모든 프롬프트에 구조화된 토큰 참조를 AI에게 제공:

```typescript
// tokens/context.ts
export function getTokenContext() {
  return `
사용 가능한 디자인 토큰:

색상:
- semantic.color.interactivePrimary → 주요 액션 색상
- semantic.color.textPrimary → 본문 텍스트
- semantic.color.backgroundPrimary → 기본 배경

간격:
- semantic.spacing.insetSmall → 8px padding
- semantic.spacing.insetMedium → 16px padding
- semantic.spacing.componentGapMedium → 16px gap

컴포넌트:
- components.button.primaryBackground
- components.button.primaryPadding
- components.card.padding

규칙:
1. 하드코딩된 색상 절대 사용 금지 (예: #3b82f6, blue-600)
2. 항상 위 목록의 토큰 참조
3. 확실하지 않으면 어떤 토큰을 사용해야 하는지 질문
  `.trim()
}
```

이를 AI 에이전트의 시스템 프롬프트에 추가하면, 사용 가능한 토큰을 항상 인식하게 됩니다.

### 패턴 3: 토큰 우선 코드 생성

토큰 사용을 강제하도록 프롬프트 구조화:

```
primary 버튼 컴포넌트를 생성하세요.

요구사항:
1. 스타일링에 components.button.* 토큰 사용
2. 하드코딩된 값 금지
3. components.button.primaryBackgroundHover를 사용한 hover 상태 포함

사용 가능한 토큰:
${getTokenContext()}
```

## 실전 예시: FramingUI MCP 통합

FramingUI는 AI 에이전트를 위한 일급 MCP(Model Context Protocol) 지원을 제공합니다. 디자인 시스템에서 AI 생성 코드로 토큰이 흐르는 방법:

### 1. 토큰 정의 (디자인 시스템)

```typescript
// design-system/tokens.ts
export const tokens = {
  semantic: {
    color: {
      interactivePrimary: 'var(--color-interactive-primary)',
      textPrimary: 'var(--color-text-primary)',
    }
  }
}
```

### 2. MCP 서버가 토큰 노출

```typescript
// mcp-server/handlers.ts
mcp.tool('get_design_tokens', async () => {
  return {
    tokens: tokens,
    schema: tokenSchema,
  }
})
```

### 3. AI 에이전트가 토큰 쿼리

```typescript
// AI 에이전트가 MCP를 사용해 토큰 쿼리
const { tokens } = await mcp.callTool('get_design_tokens')

// AI가 토큰 값을 사용해 코드 생성
const buttonCode = `
<button style={{
  background: '${tokens.semantic.color.interactivePrimary}',
  color: '${tokens.semantic.color.textPrimary}',
}}>
  클릭하세요
</button>
`
```

**결과:** Hallucination 제로. 모든 컴포넌트가 실제 디자인 시스템을 참조합니다.

## AI 친화적 토큰 베스트 프랙티스

### 1. 설명적인 시맨틱 이름 사용

❌ **나쁨:** `color1`, `spacing-md`, `blue-main`

✅ **좋음:** `color-interactive-primary`, `spacing-component-gap-medium`, `color-brand-primary`

### 2. 타입 정보 포함

```typescript
export type ColorToken = `var(--color-${string})`
export type SpacingToken = `var(--spacing-${string})`

export const tokens: {
  color: Record<string, ColorToken>
  spacing: Record<string, SpacingToken>
} = {
  color: {
    primary: 'var(--color-interactive-primary)',
  },
  spacing: {
    medium: 'var(--spacing-4)',
  }
}
```

TypeScript는 AI(와 개발자)가 토큰을 올바르게 사용하도록 보장합니다.

### 3. 토큰 사용법 문서화

```typescript
/**
 * 주요 인터랙티브 색상
 * @usage 버튼, 링크, 활성 상태
 * @contrast backgroundPrimary에서 WCAG AA 충족
 * @example <Button bg={tokens.color.interactivePrimary}>저장</Button>
 */
export const interactivePrimary = 'var(--color-interactive-primary)'
```

AI는 JSDoc 주석을 읽고 각 토큰을 언제 사용해야 하는지 이해할 수 있습니다.

### 4. Fallback 값 제공

```typescript
export const tokens = {
  color: {
    primary: 'var(--color-interactive-primary, oklch(0.55 0.15 250))',
  }
}
```

CSS 변수가 실패해도, fallback이 UI가 깨지는 것을 방지합니다.

## 토큰 아키텍처 체크리스트

디자인 토큰 시스템의 AI 준비도를 감사하는 체크리스트:

- [ ] **3계층 구조** (primitives → semantic → components)
- [ ] **시맨틱 네이밍** (`blue600`이 아닌 `interactivePrimary`)
- [ ] **타입 안전성** (토큰 카테고리에 대한 TypeScript 타입)
- [ ] **사용법 문서화** (모든 토큰에 JSDoc 주석)
- [ ] **검증 레이어** (잘못된 토큰 참조 방지)
- [ ] **MCP 통합** (AI 에이전트가 프로그래밍 방식으로 토큰 쿼리 가능)
- [ ] **프롬프트의 토큰 컨텍스트** (AI가 사용 가능한 토큰 인지)
- [ ] **Fallback 값** (fallback이 있는 CSS 변수)
- [ ] **다크 모드 지원** (테마에 맞춰 토큰 적응)
- [ ] **단일 진실 공급원** (하나의 토큰 정의 파일)

## 흔한 함정

### 함정 1: 너무 많은 컴포넌트 토큰

모든 CSS 속성에 대해 컴포넌트 토큰을 만들지 마세요:

❌ **과도한 토큰화:**
```typescript
button: {
  primaryBackgroundColorLightModeDefault: '...',
  primaryBackgroundColorLightModeHover: '...',
  primaryBackgroundColorDarkModeDefault: '...',
  // 50개 이상의 토큰...
}
```

✅ **적절한 추상화 수준:**
```typescript
button: {
  primaryBackground: semantic.color.interactivePrimary,
  primaryBackgroundHover: semantic.color.interactivePrimaryHover,
  primaryText: semantic.color.textInverse,
}
```

### 함정 2: 계층 혼용

컴포넌트 코드에서 primitive를 직접 참조하지 마세요:

❌ **나쁨:**
```tsx
<Button bg={primitives.color.blue[500]} />
```

✅ **좋음:**
```tsx
<Button bg={components.button.primaryBackground} />
```

### 함정 3: 모호한 이름

의도를 전달하지 않는 이름을 피하세요:

❌ **모호함:** `colorMain`, `spacingDefault`, `sizeNormal`

✅ **명확함:** `color-interactive-primary`, `spacing-component-gap`, `size-button-default`

## 성공 측정

토큰 아키텍처를 검증하기 위해 이 지표들을 추적하세요:

1. **Hallucination 비율** — 하드코딩된 값 vs 토큰을 사용하는 AI 생성 컴포넌트 비율
2. **토큰 커버리지** — 토큰을 사용하는 코드베이스 비율
3. **디자인 일관성 점수** — 컴포넌트 전체의 고유 color/spacing 값 (낮을수록 좋음)
4. **AI 반복 시간** — "이 색상 고쳐줘" 피드백 라운드 수

**목표:** <5% hallucination 비율, >95% 토큰 커버리지, 앱 전체에서 <20개 고유 색상.

## 결론

디자인 토큰은 더 이상 인간 개발자만을 위한 것이 아닙니다. 인간의 디자인 의도와 AI 코드 생성 사이의 인터페이스입니다.

토큰 아키텍처를 제대로 설계하면, AI 에이전트는 신뢰할 수 있는 디자인 시스템 시민이 되어—매번 완벽하게 일관된 UI를 생성합니다.

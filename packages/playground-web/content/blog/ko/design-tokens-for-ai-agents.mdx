# AI 에이전트를 위한 디자인 토큰 아키텍처

AI에게 버튼을 만들어달라고 하면, 그 다음 카드를, 그 다음 입력 필드를 만들어달라고 해보세요. 색상을 확인해보면 세 컴포넌트 모두 다른 파란색을 쓰고 있을 겁니다. 이건 AI의 문제가 아니라 토큰 아키텍처의 문제예요.

Claude Code나 Cursor 같은 AI 도구는 당신의 디자인 결정을 알 방법이 없거든요. 수백만 개의 공개 저장소에서 패턴을 학습했고, 그 중에 가장 그럴듯한 값을 골라 씁니다. 해결책은 AI가 실제로 쿼리할 수 있는 기계 판독 가능한 정보 소스를 주는 거예요.

## Figma 변수만으론 충분하지 않은 이유

Figma 변수는 AI 코드 생성기에 보이지 않아요. 개발자가 인스펙터 패널에서 색상값을 확인하고 `#4F46E5`를 하드코딩하면, AI도 그 패턴을 학습합니다. 다음 컴포넌트를 생성할 때 AI는 같은 방식으로 추측해서, 비슷한 오픈소스 프로젝트에서 자주 봤던 hex 값을 골라 씁니다.

결과적으로 코드베이스에는 `blue-600`, `indigo-500`, `#3b82f6`이 모두 "주요 색상"으로 공존하게 됩니다. AI 생성을 반복할수록 드리프트는 쌓여가죠.

코드에 토큰을 정의하면 달라져요. AI가 구조화된 토큰 파일을 임포트해서 읽을 수 있으면 추측을 멈추고 참조하기 시작합니다. 값 자체보다 아키텍처가 훨씬 더 중요한 이유예요.

## 3계층 토큰 구조

AI 가독성을 위한 가장 효과적인 패턴은 프리미티브, 시맨틱 토큰, 컴포넌트 토큰의 세 레이어를 사용하는 거예요.

### 1계층: Primitive

프리미티브는 원시값입니다. 전체 색상 팔레트, 타입 스케일, 간격 단위가 여기 해당돼요.

```typescript
// tokens/primitives.ts
export const primitives = {
  color: {
    blue: {
      500: 'oklch(0.55 0.15 250)',
      600: 'oklch(0.48 0.16 250)',
    },
    neutral: {
      50:  'oklch(0.98 0 0)',
      900: 'oklch(0.20 0 0)',
      950: 'oklch(0.15 0 0)',
    },
  },
  spacing: {
    2: '0.5rem',   // 8px
    4: '1rem',     // 16px
    6: '1.5rem',   // 24px
  },
}
```

AI가 전체 스케일을 보고 `blue.600`이 `blue.500`보다 어두운 것을 이해할 수 있어요.

### 2계층: Semantic 토큰

시맨틱 토큰은 프리미티브를 의도에 매핑합니다. AI에게 가장 중요한 레이어예요. 이름이 값의 *내용*이 아니라 *언제* 쓸지를 알려주거든요.

```typescript
// tokens/semantic.ts
export const semantic = {
  color: {
    interactivePrimary:      primitives.color.blue[500],
    interactivePrimaryHover: primitives.color.blue[600],
    interactiveDisabled:     primitives.color.neutral[300],
    backgroundPrimary:       primitives.color.neutral[50],
    backgroundInverse:       primitives.color.neutral[950],
    textPrimary:             primitives.color.neutral[900],
    textSecondary:           primitives.color.neutral[600],
    textInverse:             primitives.color.neutral[50],
  },
  spacing: {
    insetSmall:  primitives.spacing[2],
    insetMedium: primitives.spacing[4],
    insetLarge:  primitives.spacing[6],
  },
}
```

AI가 주요 액션의 색상이 필요할 때, hex 값 목록에서 고르는 게 아니라 `interactivePrimary`라는 이름을 보고 판단합니다. 이게 hallucination을 막는 핵심 메커니즘이에요.

### 3계층: Component 토큰

컴포넌트 토큰은 AI에게 특정 상황에 대한 정확한 답을 줘요.

```typescript
// tokens/components.ts
export const components = {
  button: {
    primaryBackground:      semantic.color.interactivePrimary,
    primaryBackgroundHover: semantic.color.interactivePrimaryHover,
    primaryText:            semantic.color.textInverse,
    primaryPaddingX:        semantic.spacing.insetMedium,
    primaryPaddingY:        semantic.spacing.insetSmall,
    disabledBackground:     semantic.color.interactiveDisabled,
    disabledText:           semantic.color.textSecondary,
  },
  card: {
    background: semantic.color.backgroundPrimary,
    padding:    semantic.spacing.insetLarge,
  },
}
```

버튼을 생성할 때 AI는 `components.button.*`을 쿼리하고 정확한 값을 얻습니다. 추측도 없고, 임의 색상도 없어요.

## CSS 변수를 올바르게 쓰는 방법

FramingUI 토큰은 hex 값을 담은 정적 JS 객체가 아닌 CSS 변수예요. 올바른 패턴은 이렇습니다.

```typescript
export const tokens = {
  color: {
    primary:     'var(--foreground-accent)',
    textPrimary: 'var(--foreground-primary)',
    bgCanvas:    'var(--background-page)',
  },
}
```

CSS 변수가 없을 때 UI가 깨지지 않도록 항상 폴백을 포함하세요.

```typescript
export const tokens = {
  color: {
    primary: 'var(--foreground-accent, oklch(0.55 0.15 250))',
  },
}
```

## 프롬프트 레벨에서 Hallucination 방지

토큰 아키텍처는 인프라고, 프롬프트 패턴은 이를 활성화하는 방법이에요.

AI 에이전트의 시스템 프롬프트나 CLAUDE.md에 토큰 참조 블록을 추가하세요.

```
UI 컴포넌트를 생성할 때는 항상 @/tokens에서 임포트하세요.
주요 액션에는 semantic.color.interactivePrimary를 사용하세요.
본문 텍스트에는 semantic.color.textPrimary를 사용하세요.
기본 패딩에는 semantic.spacing.insetMedium을 사용하세요.
hex 값이나 Tailwind 색상 클래스를 하드코딩하지 마세요.
```

이 지침과 AI가 읽을 수 있는 토큰 파일이 결합되면, 매 생성 세션마다 일관된 코드가 만들어집니다.

## MCP로 토큰을 AI에 연결하기

FramingUI MCP 서버는 디자인 토큰을 Claude Code에 직접 노출합니다. 프로젝트 루트의 `.mcp.json`에 설정하면 돼요.

```json
{
  "mcpServers": {
    "framingui": {
      "type": "stdio",
      "command": "npx",
      "args": ["-y", "@framingui/mcp-server@latest"]
    }
  }
}
```

가장 빠른 방법은 init 명령어예요. 프로바이더 부트스트랩, CSS 임포트, MCP 설정, AI 워크플로우 가이드를 한 번에 잡아줍니다.

```bash
npx -y @framingui/mcp-server@latest init
```

Claude Code를 재시작하면 AI가 구조화된 토큰에 접근해서, 실제 디자인 시스템을 참조하는 코드를 생성합니다.

## 자주 만나는 토큰 함정

**과도한 토큰화.** 모든 컴포넌트의 모든 CSS 속성에 토큰을 만들면 시스템이 감당하기 힘들어집니다. 버튼 토큰이 50개면 AI도 뭘 골라야 할지 혼란스럽죠. 배경, 텍스트 색상, 패딩, 비활성 상태처럼 핵심 결정에만 컴포넌트 토큰을 쓰세요.

**계층 혼용.** 컴포넌트 코드에서 프리미티브를 직접 참조하면 시맨틱 레이어를 우회하게 됩니다. 버튼 컴포넌트에 `interactivePrimary` 대신 `blue[500]`이 등장하면, AI는 의도 신호를 잃고 추측을 시작해요.

**모호한 이름.** `colorMain`, `spacingDefault`, `sizeNormal` 같은 이름은 AI에게 언제 쓰는 값인지 아무것도 알려주지 않습니다. `interactivePrimary`, `insetMedium`, `buttonHeightDefault`처럼 컨텍스트가 담긴 이름을 써야 추가 설명 없이도 AI가 올바른 선택을 해요.

## TypeScript로 강제하기

TypeScript 타입은 AI와 개발자 모두 존재하지 않는 토큰을 참조하지 못하게 막아줍니다.

```typescript
export type ColorToken = `var(--${string})`
export type SpacingToken = `var(--spacing-${string})`

export const tokens: {
  color:   Record<string, ColorToken>
  spacing: Record<string, SpacingToken>
} = {
  color: {
    primary: 'var(--foreground-accent)',
  },
  spacing: {
    medium: 'var(--spacing-4)',
  },
}
```

AI가 타입 시그니처와 맞지 않는 토큰을 참조하면 TypeScript가 즉시 잡아냅니다. 피드백 루프가 빠르게 유지되죠.

## 전후 비교

토큰 아키텍처 전의 AI 생성 버튼은 이렇게 생겼어요.

```tsx
// hallucinated values — 나머지 앱과 맞지 않음
<button className="bg-blue-600 hover:bg-blue-700 text-white px-6 py-3 rounded-lg">
  제출
</button>
```

아키텍처를 갖춘 후라면 이렇게 됩니다.

```tsx
import { components } from '@/tokens/components'

<button
  style={{
    background: components.button.primaryBackground,
    color:      components.button.primaryText,
    padding:    `${components.button.primaryPaddingY} ${components.button.primaryPaddingX}`,
  }}
>
  제출
</button>
```

AI가 생성하는 모든 버튼이 이제 동일한 토큰을 참조합니다. 디자인 시스템을 따르는 개발자가 만든 코드와 동일한 결과물이 나오는 거예요. 둘 다 같은 정보 소스를 읽고 있으니까요.

토큰 아키텍처 작업에 몇 시간을 투자하면, 매번 수동으로 수정해야 하는 AI 생성 UI에서 벗어날 수 있습니다.

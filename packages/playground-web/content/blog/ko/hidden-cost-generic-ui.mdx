## 1,000만 원짜리 색상 실수

Claude Code에게 대시보드를 만들어달라고 요청했습니다. 20개의 컴포넌트가 생성되었고, 전부 괜찮아 보였습니다 — 버튼 색상이 일관되지 않다는 걸 발견하기 전까지는요. 어떤 건 `bg-blue-500`, 다른 건 `bg-indigo-600`, 몇 개는 `#4F46E5` 같은 하드코딩된 hex 값을 사용하고 있었습니다.

다음 3시간 동안 수동으로 고쳐야 했습니다. 그리고 다음 주에 컴포넌트 10개를 더 생성하자 문제가 다시 발생했습니다.

**이건 버그가 아닙니다. AI가 범용 UI 라이브러리와 동작하는 방식의 근본적인 설계 결함입니다.**

## AI가 색상을 자꾸 틀리는 이유

AI가 shadcn UI, Tailwind, Material UI 같은 라이브러리를 사용해 UI 코드를 생성할 때, 세 가지 중대한 문제에 직면합니다:

### 1. **시맨틱 컨텍스트 부재**

범용 컴포넌트 라이브러리는 임의의 값을 사용합니다:

```tsx
// AI는 문서에서 이렇게 봅니다
<Button className="bg-blue-500 hover:bg-blue-600">
  Primary Action
</Button>
```

AI는 `blue-500`이 당신의 "primary" 색상인지 모릅니다. 그냥 예시로만 봅니다. 다음번에 버튼을 생성할 때는:
- `bg-blue-600` (더 어두운 파란색)
- `bg-indigo-500` (다른 색조)
- `#3B82F6` (하드코딩)
- `bg-primary` (존재하지 않는 토큰 추측)

**결과:** 코드베이스 전체에서 일관되지 않은 색상.

### 2. **모든 곳에 하드코딩된 값**

일반적인 UI 라이브러리를 보세요:

```tsx
// 전형적인 shadcn UI 컴포넌트
<div className="rounded-md border border-gray-200 bg-white p-4">
  <h3 className="text-lg font-semibold text-gray-900">Title</h3>
  <p className="text-sm text-gray-600">Description</p>
</div>
```

모든 값이 하드코딩되어 있습니다:
- 테두리는 `gray-200`
- 배경은 `white`
- 제목은 `gray-900`
- 텍스트는 `gray-600`

AI가 비슷한 컴포넌트를 생성할 때, 변형을 환각합니다:
- 때로는 `gray-100`, 때로는 `slate-200`
- 때로는 `gray-600` 대신 `text-gray-700`

**결과:** AI가 생성하는 모든 컴포넌트마다 디자인 시스템이 퇴화합니다.

### 3. **컨텍스트 윈도우 한계**

AI는 대규모 코드베이스에서 모든 색상 선택을 기억할 수 없습니다. 컨텍스트에 기존 컴포넌트 50개를 포함해도, AI는:
- 정확한 색조를 잘못 기억
- 다른 컴포넌트의 패턴을 혼합
- 시스템에 존재하지 않는 "중간" 값을 보간

**결과:** 코드 리뷰에서 잡기 어려운 점진적인 스타일 이탈.

## "무료" 컴포넌트의 숨겨진 비용

범용 UI 라이브러리는 무료이고 유연해 보입니다. 하지만 AI로 빌드하면 숨겨진 비용이 발생합니다:

| 문제 | 시간 비용 | 빈도 |
|------|----------|------|
| 일관되지 않은 색상 수정 | 2-4시간 | 매 스프린트 |
| 스타일 이슈 코드 리뷰 | 1-2시간 | 매 PR |
| 하드코딩된 값 리팩토링 | 8-16시간 | 분기마다 |
| 디자이너-개발자 정렬 | 4-8시간 | 매월 |

**한 달에 20-40시간** — 즉 엔지니어링 시간으로 200만~1,000만 원 — 을 색상 일관성 관리에만 쓰게 됩니다.

## 시맨틱 토큰이 해결하는 방법

FramingUI는 다른 접근 방식을 취합니다: **모든 색상, 간격, 스타일이 시맨틱 토큰입니다.**

하드코딩된 값 대신:

```tsx
// ❌ 범용 라이브러리 (하드코딩)
<div className="bg-blue-500 text-white">
  Primary content
</div>
```

시맨틱 토큰을 사용합니다:

```tsx
// ✅ FramingUI (시맨틱)
<div className="bg-primary text-on-primary">
  Primary content
</div>
```

### AI에 왜 효과적인가

1. **명확한 의도**
   - `bg-primary`는 AI에게 "이게 primary 브랜드 색상"이라고 알려줌
   - "무엇이든 의미할 수 있는" blue-500이 아님

2. **자체 문서화**
   - `text-error`는 명백히 에러 상태용
   - `bg-surface-raised`는 명확히 elevated surface
   - AI가 추측할 필요 없음

3. **자동으로 일관성 유지**
   - AI가 모든 곳에 `bg-primary` 사용 → 모든 primary 색상 일치
   - 토큰 값 변경 → 모든 것이 업데이트됨

4. **테마 인식**
   - 토큰이 자동으로 라이트/다크 모드 적응
   - AI가 테마 변형을 신경 쓸 필요 없음

## 실제 예시: Before & After

### Before (범용 UI + AI)

AI가 로그인 폼 생성:

```tsx
// 첫 번째 컴포넌트
<Button className="bg-blue-500 hover:bg-blue-600 text-white">
  Sign In
</Button>

// 두 번째 컴포넌트 (나중에 생성)
<Button className="bg-indigo-600 hover:bg-indigo-700 text-white">
  Create Account
</Button>

// 세 번째 컴포넌트 (환각)
<Button className="bg-primary-600 hover:bg-primary-700 text-white">
  Continue
</Button>
```

**결과:** 세 개의 "primary" 버튼, 세 가지 다른 색상. 수동 수정 필요.

### After (FramingUI + AI)

AI가 시맨틱 토큰으로 생성:

```tsx
// 모든 컴포넌트
<Button variant="primary">Sign In</Button>
<Button variant="primary">Create Account</Button>
<Button variant="primary">Continue</Button>

// 내부적으로 사용: bg-primary text-on-primary
```

**결과:** 완벽한 일관성. 수동 수정 제로.

## 시맨틱 토큰의 경제학

계산해봅시다:

**시맨틱 토큰 없이:**
- 일관성 문제 수정 시간: 월 30시간
- 시급 10만원 기준 비용: 월 300만원
- 연간 비용: 3,600만원

**시맨틱 토큰 사용:**
- 초기 설정: 4-8시간 (40~80만원)
- 지속적인 유지보수: 월 ~2시간 (월 20만원)
- 연간 비용: 320만원

**절약: 연간 3,280만원** (소규모 팀 기준).

AI로 빌드하는 5명 팀의 경우, ROI는 엄청납니다.

## 시맨틱 토큰 시작하기

FramingUI로 쉽게 시작할 수 있습니다:

1. **FramingUI 설치** (shadcn UI 포크)
2. **하드코딩된 클래스 대신 시맨틱 variant 사용**
3. **AI에게 컴포넌트 생성하도록 하기** — 기본적으로 일관성 유지됨

예시:

```tsx
import { Button } from '@/components/ui/button'

// AI가 안정적으로 생성 가능
<Button variant="primary">Primary Action</Button>
<Button variant="secondary">Secondary Action</Button>
<Button variant="destructive">Delete</Button>
```

모든 variant가 시맨틱 토큰에 매핑됩니다:
- `primary` → `bg-primary`, `text-on-primary`
- `destructive` → `bg-error`, `text-on-error`
- `secondary` → `bg-secondary`, `text-on-secondary`

## 결론

범용 UI 라이브러리는 사람이 신중하게 코드를 작성할 때 훌륭하게 작동합니다. 하지만 **AI는 신중하게 작동하지 않습니다** — 확률적으로 작동합니다.

숨겨진 비용은 라이브러리 자체가 아닙니다. 라이브러리가 AI를 위해 설계되지 않았기 때문에 **AI가 생성한 일관성 문제를 수정하는 데 쓰는 수백 시간**입니다.

시맨틱 토큰은 단순히 있으면 좋은 게 아닙니다. **도움이 되는 AI와 기술 부채를 만드는 AI의 차이**입니다.

FramingUI는 두 세계의 장점을 제공합니다:
- ✅ 사랑하는 모든 컴포넌트 (shadcn UI 포크)
- ✅ AI 일관성을 위한 시맨틱 토큰
- ✅ 자동 테마 지원
- ✅ 런타임 오버헤드 제로

AI의 색상 실수 고치는 걸 멈추세요. 시맨틱 토큰으로 빌드를 시작하세요.

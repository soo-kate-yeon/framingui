AI로 만든 SaaS 제품 쇼케이스를 스크롤하다 보면 같은 인터페이스가 반복됩니다. 둥근 카드, 파란 primary 버튼, 흰 배경에 회색 본문 텍스트. 나쁘지 않은 UI예요. 하지만 옆에 있는 제품과 구분이 안 됩니다.

미적 문제가 아니에요. 구조적 문제거든요. 왜 이런 일이 일어나는지 이해하면 해결책이 보입니다.

## AI는 의도가 아닌 빈도로 생성합니다

"사용자 대시보드 만들어줘"라고 하면 모델은 훈련 데이터에서 가장 자주 등장하는 것을 기반으로 생성해요. 그러니까 둥근 모서리가 나옵니다. 거의 모든 현대 UI가 쓰니까요. 파란 primary 색상이 나옵니다. 오픈소스 저장소의 SaaS 제품에서 가장 흔하니까요. Tailwind 유틸리티 클래스가 나옵니다. 공개된 코드에서 너무 흔하니까요.

모델은 브랜드 성격, 색상 팔레트, 간격 철학, 컴포넌트 라이브러리를 전혀 모릅니다. 수천 개의 코드베이스를 평균 내서 그 분포의 통계적 중심을 만들어냅니다. 그 중심은 정의상 획일적입니다.

`text-gray-700` 같은 Tailwind 클래스의 하드코딩된 색상을 생각해보세요. 사람 디자이너는 그 색상이 브랜드 시스템에서 무엇을 나타내는지, 언제 적절한지, 주변 요소와 어떤 관계인지 알아요. AI는 통계적으로 본문 텍스트와 상관관계가 있는 색상 값을 봅니다. 의미론적이 아닌 기계적으로 적용합니다. 시각적 결과는 맞는 것처럼 보이지만 의미가 없어요.

## 디자인 시스템을 붙여넣어도 안 되는 이유

직관적인 해결책은 디자인 토큰이나 컴포넌트 문서를 프롬프트에 붙여넣는 거예요. 도움은 됩니다. 하지만 심각한 한계가 있어요.

디자인 토큰 문서는 컨텍스트를 많이 소모합니다. 의미론적 이름, 사용 가이드라인, 제약이 포함된 제대로 된 토큰 세트는 실제 요청을 쓰기도 전에 수천 토큰을 씁니다. 대화가 길어질수록 앞부분의 토큰에 대한 모델의 주의가 줄어들어요.

더 결정적인 문제는 수동 유지보수예요. 새 대화는 항상 처음부터 시작합니다. 디자인 시스템이 변하면 모든 프롬프트 템플릿을 업데이트해야 해요. 그리고 토큰이 완벽하게 문서화되어 있어도 모델은 각 생성 결정에 어떻게 적용할지 추론해야 하는데, 이 추론에서 오류가 생깁니다.

필요한 건 프롬프트에 더 많은 텍스트가 아니에요. 디자인 시스템과의 실시간 프로그래밍적 연결입니다.

## MCP 솔루션

FramingUI는 Model Context Protocol을 통해 디자인 시스템을 Claude Code에 연결합니다. 설정은 `.mcp.json`에 항목 하나면 돼요:

```json
{
  "mcpServers": {
    "framingui": {
      "type": "stdio",
      "command": "npx",
      "args": ["-y", "@framingui/mcp-server@latest"]
    }
  }
}
```

이 연결이 활성화되면 Claude는 생성 과정에서 디자인 시스템을 쿼리할 수 있어요. 어떤 테마를 쓸 수 있는지, 어떤 컴포넌트가 있는지, 컴포넌트가 지원하는 변형이 무엇인지, 그 변형이 사용하는 CSS 변수가 무엇인지 물어볼 수 있습니다. 답은 훈련 데이터 보간이 아니라 실제 디자인 시스템 카탈로그에서 옵니다.

MCP 접근이 있는 모델이 프로필 카드를 생성할 때 둥근 모서리와 파란 버튼이 적절하다고 추측하지 않아요. 컴포넌트 카탈로그를 쿼리하고, 사용 가능한 카드 변형을 가져오고, 테마가 정의한 의미론적 CSS 변수를 참조하는 코드를 생성합니다. `#ffffff` 대신 `var(--background-page)`. `bg-blue-500` 대신 `var(--foreground-accent)`.

## 실제로 무엇이 달라지는가

가장 즉각적인 변화는 하드코딩된 값의 제거예요. AI 생성 컴포넌트에서 실제 디자인 시스템과 관계없는 색상 hex 코드나 임의의 Tailwind 유틸리티가 사라집니다. 모든 값이 테마가 제어하는 CSS 변수로 거슬러 올라갑니다.

두 번째 변화는 변형의 정확성이에요. 모델이 훈련 데이터에서 추론하는 대신 실제 컴포넌트 정의를 쿼리하기 때문에, 실제로 존재하는 변형을 참조합니다. FramingUI의 버튼 컴포넌트에는 문서화된 변형과 문서화된 props가 있어요. 모델이 패턴 추측이 아니라 명세를 읽었기 때문에 그 변형을 정확하게 생성합니다.

세 번째 변화는 세션 간 일관성이에요. 디자인 시스템이 프롬프트 텍스트가 아니라 MCP 서버에 존재하기 때문에, 모든 대화가 같은 신뢰할 수 있는 소스로 시작합니다. 첫 번째 세션과 열 번째 세션이 같은 토큰 세트를 참조하는 컴포넌트를 생성해요. 생성된 컴포넌트 간 시각적 드리프트가 크게 줄어듭니다.

## 전제 조건: 기계 가독형 디자인 시스템

디자인 시스템이 Figma 파일이나 문서 웹사이트로만 존재한다면 이 중 어느 것도 작동하지 않아요. MCP 서버가 접근할 수 있는 형태여야 합니다.

FramingUI는 여섯 개의 프로덕션 준비 테마를 제공합니다: `classic-magazine`, `dark-boldness`, `minimal-workspace`, `neutral-workspace`, `pebble`, `square-minimalism`. 각 테마는 MCP 사용을 위해 완전히 구조화되어 있어요. 의미론적 토큰 정의, 컴포넌트 카탈로그, AI가 생성 시점에 쿼리할 수 있는 변형 메타데이터가 포함됩니다.

실용적인 경로는 시각적 방향과 가장 잘 맞는 테마를 선택하고, 인증(`framingui-mcp login`)하고, MCP 서버가 편집기를 그 구조화된 디자인 시스템에 연결하게 하는 거예요. 그러면 AI가 모든 미학의 통계적 평균이 아니라 선택한 미학을 반영하는 컴포넌트를 생성합니다.

획일화된 UI는 AI 생성의 불가피한 결과가 아니에요. 컨텍스트 없는 생성의 결과입니다. 기계 가독형 디자인 시스템으로 컨텍스트를 제공하면, 출력이 여러분의 제품에 고유하게 속하는 모습으로 달라집니다.

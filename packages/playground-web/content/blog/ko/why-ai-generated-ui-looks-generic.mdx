## 이것도 눈치채셨죠

트위터를 여세요. 인디 해커 쇼케이스를 스크롤하세요. 모든 AI 생성 SaaS가 똑같이 생겼습니다:

- 같은 둥근 모서리
- 같은 파란색 액센트
- 같은 카드 레이아웃
- 같은 버튼 스타일
- 같은 "깔끔하고 현대적인" 미학

나쁜 UI가 아닙니다—사실 꽤 괜찮습니다. 하지만 **획일화**되어 있습니다. 세 제품을 나란히 놓으면, 어느 것이 어느 것인지 구분할 수 없습니다.

## 진짜 문제: AI에게 브랜드 컨텍스트가 없다

"사용자 대시보드 만들어줘"라고 AI에 프롬프트하면 이런 일이 일어납니다:

```
프롬프트 → AI 훈련 데이터 → 통계적 패턴 매칭 → 일반적인 출력
```

AI는 **의도**가 아닌 **빈도**를 기반으로 생성합니다. 훈련 데이터에서 가장 자주 나타나는 것을 만들어냅니다:
- 둥근 모서리 (모두가 쓰니까)
- 파란색 primary 색상 (SaaS에서 가장 흔함)
- Tailwind 유틸리티 클래스 (오픈소스에서 인기)
- 카드 기반 레이아웃 (어디서나 볼 수 있는 패턴)

**AI가 모르는 것:**
- 당신 브랜드의 성격 (장난스러움 vs. 전문적)
- 당신 색상 팔레트 (따뜻함 vs. 차가움)
- 당신 간격 철학 (밀도 높음 vs. 여유로움)
- 당신 컴포넌트 라이브러리 (커스텀 vs. 일반)

결과: 모든 출력이 "일반적인 SaaS UI 템플릿 #47"처럼 보입니다.

## TL;DR

- **문제**: AI 도구는 브랜드 아이덴티티가 아닌 훈련 데이터 패턴을 기반으로 생성합니다
- **왜 이런 일이**: AI는 디자인 시스템 컨텍스트가 없어서 일반적인 패턴으로 기본 설정됩니다
- **해결책**: 구조화된 디자인 토큰으로 AI에게 디자인 시스템 컨텍스트를 제공합니다
- **framingui의 도움**: 디자인 시스템을 기계 가독형으로 만들어 AI가 브랜드에 맞는 UI를 생성합니다
- **결과**: AI가 생성하는 모든 컴포넌트를 수동으로 수정하지 않아도 일관되고 브랜드화된 인터페이스

## 획일화된 AI UI의 해부학

디자인 컨텍스트 없이 AI가 생성하는 것을 살펴봅시다.

### 예시 1: 일반적인 대시보드

**프롬프트:**
```
사용자 분석 대시보드 만들어줘
```

**AI 출력 (v0, Claude, ChatGPT):**

```tsx
export function Dashboard() {
  return (
    <div className="min-h-screen bg-gray-50">
      <div className="p-6">
        <h1 className="text-2xl font-bold text-gray-900 mb-6">대시보드</h1>

        <div className="grid grid-cols-3 gap-6">
          <div className="bg-white p-6 rounded-lg shadow-sm">
            <h3 className="text-gray-600 text-sm">총 사용자</h3>
            <p className="text-3xl font-bold text-gray-900">2,543</p>
          </div>

          <div className="bg-white p-6 rounded-lg shadow-sm">
            <h3 className="text-gray-600 text-sm">매출</h3>
            <p className="text-3xl font-bold text-gray-900">$45,231</p>
          </div>

          <div className="bg-white p-6 rounded-lg shadow-sm">
            <h3 className="text-gray-600 text-sm">성장률</h3>
            <p className="text-3xl font-bold text-green-600">+12.5%</p>
          </div>
        </div>
      </div>
    </div>
  );
}
```

**이게 뭐가 문제일까요?**

기술적으론 아무것도. 작동합니다. 하지만 자세히 보세요:

- **하드코딩된 색상**: `gray-50`, `gray-900`, `green-600` (브랜드 색상 아님)
- **임의의 간격**: `p-6`, `gap-6`, `mb-6` (간격 스케일 아님)
- **일반적인 타이포그래피**: `text-2xl`, `text-3xl` (타입 시스템 아님)
- **패턴 남용**: 둥근 모서리 + 그림자 = 어디서나 봄

만약 당신 브랜드가:
- **따뜻하고 친근함**: 이건 차갑게 느껴짐 (회색 팔레트)
- **대담하고 에너지**: 이건 밋밋하게 느껴짐 (낮은 대비)
- **미니멀하고 세련됨**: 이건 어수선하게 느껴짐 (그림자 + 테두리)

### 예시 2: 같은 버튼, 다른 프로젝트

AI는 완전히 다른 브랜드에서 거의 동일한 버튼을 생성합니다:

```tsx
// 이커머스 스토어 프롬프트
<button className="bg-blue-500 hover:bg-blue-600 text-white px-4 py-2 rounded">
  장바구니 담기
</button>

// 헬스케어 앱 프롬프트
<button className="bg-blue-500 hover:bg-blue-600 text-white px-4 py-2 rounded">
  예약하기
</button>

// 핀테크 대시보드 프롬프트
<button className="bg-blue-500 hover:bg-blue-600 text-white px-4 py-2 rounded">
  송금하기
</button>
```

**같은 버튼으로:**
- 운동화 구매
- 병원 예약
- 돈 이체

말이 되나요? 당연히 아닙니다. 하지만 AI는 모릅니다—훈련 데이터에서 **통계적으로 가장 가능성 높은 버튼**을 생성합니다.

## 왜 이런 일이: AI 훈련의 문제

### AI는 오픈소스 코드베이스에서 학습합니다

대부분의 AI 훈련 데이터 출처:
- **오픈소스 프로젝트**: Tailwind 기반이 많고 일반적인 스타일링
- **튜토리얼 코드**: "초보자 친화적" = 최소한의 커스터마이징
- **UI 컴포넌트 라이브러리**: 넓은 호소력을 위해 설계, 특정 브랜드용 아님

AI에게 UI 생성을 요청하면, 본질적으로 이 모든 소스를 **평균**하는 것입니다. 결과? 가장 공통분모—획일화된 UI.

### AI는 디자인 의도를 이해하지 못합니다

이 디자인 토큰을 보세요:

```css
--color-primary: #3B82F6;
```

**인간 디자이너가 아는 것:**
- 이 파란색은 "신뢰와 전문성"을 전달
- primary CTA에만 사용 (화면당 최대 1개)
- 중립 회색과 균형 맞춰 사용
- 고우선순위 액션에 사용

**AI가 보는 것:**
- 헥스 색상 값
- 통계적 상관관계: "버튼과 함께 나타남"
- 패턴: "배경에 자주 사용"

AI는 토큰을 **의미론적**이 아닌 **기계적**으로 적용합니다. 그 파란색을 *왜* 선택했는지, *언제* 사용해야 하는지 이해하지 못합니다.

### 컨텍스트 윈도우 한계

전체 디자인 시스템을 프롬프트에 붙여넣어도 한계에 부딪힙니다:

- **토큰 낭비**: 디자인 토큰이 수천 토큰의 컨텍스트를 소비
- **컨텍스트 소멸**: AI가 대화 앞부분의 토큰을 "잊음"
- **수동 유지보수**: 새 채팅마다 디자인 시스템 다시 붙여넣기 필요
- **버전 드리프트**: 모든 대화에서 업데이트하지 않으면 AI가 오래된 토큰 값 사용

## 해결책: 기계 가독형 디자인 시스템

획일화된 AI 출력을 고치려면 **세 가지**가 필요합니다:

### 1. 구조화된 디자인 토큰

사람 읽는 문서를 기계 가독형 데이터로 변환합니다.

**전 (사람용 문서):**

```markdown
## 색상

콜투액션 버튼에는 primary 블루(#3B82F6)를 사용하세요.
제목에는 gray-900, 본문 텍스트에는 gray-600을 사용하세요.
```

**후 (기계 가독형 토큰):**

```json
{
  "colors": {
    "primary": {
      "value": "#3B82F6",
      "usage": "call-to-action",
      "semantic": "trust, professionalism",
      "constraints": ["max-one-per-viewport"]
    },
    "text": {
      "heading": { "value": "#1a1a1a", "usage": "h1, h2, h3" },
      "body": { "value": "#666666", "usage": "paragraphs, labels" }
    }
  },
  "spacing": {
    "unit": 4,
    "scale": [0, 4, 8, 12, 16, 24, 32, 48, 64],
    "semantic": {
      "component-padding": "spacing-4",
      "section-gap": "spacing-6",
      "page-margin": "spacing-8"
    }
  }
}
```

이제 AI가:
- 토큰을 프로그래밍적으로 쿼리 (컨텍스트 윈도우 낭비 없음)
- 의미론적 의미 이해 ("call-to-action" = primary 색상)
- 제약 준수 ("max-one-per-viewport")
- 모든 생성에서 토큰을 일관되게 적용

### 2. 컴포넌트 지식 그래프

AI에게 컴포넌트가 존재하는 것뿐 아니라, *언제*와 *왜* 사용하는지 알려줍니다.

**전 (컴포넌트 라이브러리 문서):**

```markdown
## 버튼

사용 가능한 변형: primary, secondary, ghost
```

**후 (컴포넌트 지식):**

```json
{
  "components": {
    "Button": {
      "variants": [
        {
          "name": "primary",
          "visualImpact": "high",
          "purpose": "주어진 컨텍스트의 메인 콜투액션",
          "constraints": ["max-one-per-viewport"],
          "tokens": {
            "bg": "button.bg.primary",
            "fg": "button.fg.primary"
          },
          "slotAffinity": {
            "modal_footer": 0.9,
            "card_actions": 0.85,
            "header": 0.2
          }
        },
        {
          "name": "ghost",
          "visualImpact": "low",
          "purpose": "저강조 보조 액션",
          "tokens": {
            "bg": "transparent",
            "fg": "button.fg.ghost"
          }
        }
      ]
    }
  }
}
```

이제 AI가 이해합니다:
- **Primary 버튼**: 높은 시각적 임팩트, 절제해서 사용 (화면당 최대 1개)
- **Ghost 버튼**: 낮은 시각적 임팩트, 보조 액션에 사용
- **슬롯 친화도**: Primary 버튼은 모달 푸터에 어울림 (0.9 점수), 헤더에는 아님 (0.2 점수)

결과: AI가 모달을 생성할 때 "저장"은 primary 버튼으로, "취소"는 ghost로—직접 지정하지 않아도.

### 3. AI 가독형 접근 (MCP)

Model Context Protocol로 AI를 디자인 시스템에 직접 연결합니다.

**기존 워크플로우:**

```
1. 문서에서 디자인 토큰 복사
2. AI 프롬프트에 붙여넣기
3. AI가 컴포넌트 생성
4. 토큰 사용 수동 검증
5. AI가 놓친 하드코딩 값 수정
6. 모든 컴포넌트마다 반복
```

**MCP 워크플로우:**

```
1. AI가 MCP로 디자인 시스템 쿼리
2. AI가 올바른 토큰으로 컴포넌트 생성
3. 끝
```

**MCP 통합 예시:**

```json
// .claude/mcp.json
{
  "servers": {
    "design-system": {
      "command": "npx",
      "args": ["@framingui/mcp-server"],
      "env": {
        "TEKTON_PROJECT_PATH": ".",
        "TEKTON_TOKEN_PATH": "./tokens.json"
      }
    }
  }
}
```

이제 프롬프트하면:

```
사용자 프로필 카드 만들어줘
```

AI가 자동으로:
1. 사용 가능한 컴포넌트 쿼리 (`Card`, `Avatar`, `Badge`, `Button`)
2. 토큰 값 가져오기 (색상, 간격, 타이포그래피)
3. 컴포넌트 제약 확인 (버튼 배치 규칙)
4. 실제 디자인 토큰으로 브랜드에 맞는 코드 생성

```tsx
import { Card, Avatar, Badge, Button } from '@/components';
import { tokens } from '@/design-system/tokens';

export function ProfileCard({ user }) {
  return (
    <Card
      variant="elevated"
      style={{
        padding: tokens.spacing[6],
        background: tokens.bg.surface.elevated,
      }}
    >
      <div style={{ display: 'flex', gap: tokens.spacing[4] }}>
        <Avatar
          src={user.avatar}
          size="lg"
          style={{ borderRadius: tokens.radius.full }}
        />
        <div>
          <h3 style={{
            fontSize: tokens.fontSize.lg,
            fontWeight: tokens.fontWeight.semibold,
            color: tokens.fg.primary,
          }}>
            {user.name}
          </h3>
          <Badge
            variant="success"
            style={{
              marginTop: tokens.spacing[2],
              background: tokens.badge.bg.success,
              color: tokens.badge.fg.success,
            }}
          >
            {user.status}
          </Badge>
        </div>
      </div>

      <div style={{
        marginTop: tokens.spacing[4],
        display: 'flex',
        gap: tokens.spacing[2],
      }}>
        <Button variant="primary">프로필 보기</Button>
        <Button variant="ghost">메시지</Button>
      </div>
    </Card>
  );
}
```

**하드코딩 값 제로. 모든 토큰은 디자인 시스템에서.**

## 실제 예시: 일반 vs. 브랜드화

### 예시 1: 이커머스 상품 카드

**일반 AI 출력:**

```tsx
<div className="bg-white p-4 rounded-lg shadow">
  <img src="/product.jpg" className="w-full h-48 object-cover rounded" />
  <h3 className="mt-4 text-lg font-semibold">상품명</h3>
  <p className="text-gray-600">$99.99</p>
  <button className="mt-4 w-full bg-blue-500 text-white py-2 rounded">
    장바구니 담기
  </button>
</div>
```

**브랜드화 AI 출력 (디자인 컨텍스트 포함):**

```tsx
<Card
  variant="product"
  style={{
    padding: tokens.spacing[4],
    background: tokens.bg.surface.base,
    border: `1px solid ${tokens.border.subtle}`,
    borderRadius: tokens.radius.lg,
  }}
>
  <ProductImage
    src="/product.jpg"
    style={{
      aspectRatio: '4/3',
      borderRadius: tokens.radius.md,
      objectFit: 'cover',
    }}
  />
  <ProductDetails style={{ marginTop: tokens.spacing[4] }}>
    <Heading
      level={3}
      style={{
        fontSize: tokens.fontSize.lg,
        fontWeight: tokens.fontWeight.bold,
        color: tokens.fg.primary,
      }}
    >
      상품명
    </Heading>
    <Price
      style={{
        fontSize: tokens.fontSize.xl,
        fontWeight: tokens.fontWeight.semibold,
        color: tokens.fg.accent,
      }}
    >
      $99.99
    </Price>
  </ProductDetails>
  <Button
    variant="primary"
    fullWidth
    style={{
      marginTop: tokens.spacing[4],
      background: tokens.button.bg.primary,
      color: tokens.button.fg.primary,
      padding: `${tokens.spacing[3]} ${tokens.spacing[4]}`,
      borderRadius: tokens.radius.md,
    }}
  >
    장바구니 담기
  </Button>
</Card>
```

**차이점:**
- 브랜드 간격 스케일 사용 (임의의 `mt-4` 아님)
- 가격에 액센트 색상 사용 (일반적인 `gray-600` 아님)
- 브랜드 테두리 스타일 사용 (그림자 대신 섬세한 테두리)
- 의미론적 컴포넌트 사용 (`ProductImage`, `Price`) 일반 div 아님

### 예시 2: 대시보드 카드

**일반 AI 출력:**

```tsx
<div className="bg-white p-6 rounded-lg shadow-sm">
  <h3 className="text-sm text-gray-600 mb-2">총 매출</h3>
  <p className="text-3xl font-bold text-gray-900">$124,500</p>
  <p className="text-sm text-green-600 mt-2">↑ 지난달 대비 12.5%</p>
</div>
```

**브랜드화 AI 출력 (핀테크 브랜드 - 전문적인 다크 테마):**

```tsx
<StatCard
  variant="elevated"
  style={{
    padding: tokens.spacing[6],
    background: tokens.bg.surface.elevated,
```

## framingui가 UI 일관성을 만드는 방법

framingui는 세 계층으로 획일화된 AI 문제를 해결합니다:

### 계층 1: 결정론적 토큰 생성

WCAG 준수가 내장된 알고리즘으로 디자인 토큰을 생성합니다.

```typescript
import { generateTokensFromArchetype } from '@framingui/token-generator';

const archetype = {
  name: "fintech-professional",
  primaryColor: { l: 0.45, c: 0.13, h: 240 }, // 딥 블루
  neutralTone: "tinted", // 약간 틴트된 회색
  contrast: "high", // 핀테크 = 접근성 우선
  borderRadius: "minimal", // 전문적 = 미묘함
};

const tokens = await generateTokensFromArchetype(archetype, {
  wcagLevel: 'AAA', // 표준 AA보다 엄격
  generateDarkMode: true,
});

// 결과: WCAG 준수가 보장된 100개 이상 토큰
// 모든 토큰이 브랜드 성격을 따름
```

### 계층 2: 컴포넌트 지식 시스템

컴포넌트를 AI가 이해하는 메타데이터로 변환합니다.

```typescript
export const StatCardKnowledge = {
  name: "StatCard",
  category: "data-display",
  purpose: "컨텍스트와 함께 핵심 지표 표시",
  visualImpact: "medium",

  variants: [
    {
      name: "elevated",
      purpose: "페이지 배경에서 돋보이게",
      tokens: {
        bg: "bg.surface.elevated",
        border: "border.default",
      },
      slotAffinity: {
        dashboard_grid: 0.95, // 대시보드 그리드에 완벽
        modal: 0.1, // 모달에는 거의 없음
      }
    }
  ],

  composition: {
    requiredSlots: ["label", "value"],
    optionalSlots: ["change", "icon"],
    constraints: ["value-must-be-numeric"]
  }
};
```

이제 AI가 알게 됩니다:
- StatCard는 대시보드 그리드에 어울림 (0.95 친화도)
- 라벨과 값이 필요 (필수 슬롯)
- 변화 지표를 보여줄 수 있음 (선택 슬롯)

```bash

    }
  }
}

# AI가 이제 접근:
# - 모든 디자인 토큰
# - 컴포넌트 지식
# - 사용 제약
# - 슬롯 친화도 점수
```

**결과:**


AI가 생성하는 모든 컴포넌트가:

- 제약 준수 (primary CTA 최대 1개)

## FAQ

### 기존 디자인 시스템과 framingui를 쓸 수 있나요?

네! framingui는 다음과 통합됩니다:
- **Figma Tokens**: Figma Tokens 플러그인에서 토큰 JSON 가져오기
- **Style Dictionary**: Style Dictionary 출력에 MCP 연결
- **Tailwind CSS**: `tailwind.config.js`에서 토큰 추출
- **커스텀 토큰**: 색상/간격/타이포그래피 값이 있는 모든 JSON 포맷

### 모든 AI 도구와 작동하나요?

framingui의 MCP 서버는 다음과 작동합니다:
- **Claude Code** (공식 지원)
- **Cursor** (실험적 MCP 지원)
- **Cline / Windsurf** (커뮤니티 지원)
- Model Context Protocol을 구현하는 모든 도구

MCP 지원이 없는 AI 도구의 경우:
- 토큰 JSON을 프롬프트에 복사 (수동이지만 효과적)
- framingui의 코드 검증 도구로 AI 출력 확인
- MCP 지원 도구로 초기 컴포넌트 생성 후 다른 곳에서 반복

### AI가 여전히 디자인 실수를 하나요?

디자인 시스템 컨텍스트가 있는 AI는 실수가 적지만, 완벽하지 않습니다:

**제거된 실수:**
- ✅ 하드코딩된 색상과 간격
- ✅ 일관되지 않은 컴포넌트 사용
- ✅ 접근성 위반 (WCAG 토큰 내장)
- ✅ 잘못된 컴포넌트 변형

**여전히 가능한 실수:**
- ⚠️ 레이아웃 논리 오류 (예: flex vs. grid 선택)
- ⚠️ 컴포넌트 구성 문제 (중첩 오류)
- ⚠️ 비즈니스 논리 버그 (디자인과 무관)

framingui는 **디자인 일관성**을 해결합니다. 모든 가능한 버그는 아닙니다.

### 비용은 얼마인가요?

framingui는 **오픈소스이며 무료**입니다:
- MIT 라이선스
- 사용 제한 없음
- API 키 불필요 (MCP가 로컬에서 실행)
- 모든 AI 도구와 작동 (벤더 종속 없음)

### 디자인 시스템 준수를 자동으로 강제할 수 있나요?

네! framingui는 빌드 타임 검증을 포함합니다:

```typescript
// esbuild.config.js
import { tektonPlugin } from '@framingui/esbuild-plugin';

export default {
  plugins: [
    tektonPlugin({
      strict: true, // 위반 시 빌드 실패
      threshold: 100, // 100% 토큰 준수 요구
    })
  ]
};
```

이것이 방지하는 것:
- 하드코딩 색상/간격이 프로덕션에 도달
- AI (또는 인간) 개발자가 디자인 시스템 우회
- 시간 지나며 디자인 드리프트

디자인 시스템을 위반하면 빌드가 실패합니다. AI 생성 코드와 인간 작성 코드 모두에 작동합니다.

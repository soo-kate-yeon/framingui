## AI 일관성 문제

Cursor로 대시보드를 만들고 있습니다. Claude에게 요청합니다:

> "이름, 아바타, 상태 배지가 있는 사용자 프로필 카드를 만들어줘"

Claude가 멋진 코드를 생성합니다:

```tsx
<div className="bg-white rounded-lg p-6 shadow-md">
  <img src="/avatar.jpg" className="w-16 h-16 rounded-full" />
  <h3 className="text-xl font-bold text-gray-900">John Doe</h3>
  <span className="bg-green-500 text-white px-3 py-1 rounded-full text-sm">
    Active
  </span>
</div>
```

완벽합니다! 코드베이스에 복사합니다.

이틀 후, 다시 요청합니다:

> "아이콘, 메시지, 타임스탬프가 있는 알림 카드를 만들어줘"

Claude가 생성합니다:

```tsx
<div className="bg-slate-50 rounded-xl p-4 border border-gray-200">
  <div className="flex items-center gap-3">
    <Bell className="w-5 h-5 text-blue-600" />
    <p className="text-base text-slate-800">New message received</p>
  </div>
  <time className="text-xs text-gray-500">2 minutes ago</time>
</div>
```

역시 좋아 보입니다! 하지만 잠깐...

- 프로필 카드: `bg-white`, 알림: `bg-slate-50`
- 프로필: `rounded-lg`, 알림: `rounded-xl`
- 프로필: `p-6`, 알림: `p-4`
- 프로필: `text-gray-900`, 알림: `text-slate-800`

**AI가 완전히 다른 디자인 언어로 두 개의 카드를 생성했습니다.**

## 왜 이런 일이 벌어질까: AI가 UI 코드를 생성하는 방법

### 패턴 매칭 문제

Claude, GPT-4, Gemini 같은 LLM은 디자인 시스템을 "기억"하지 않습니다. 코드 생성 방식:

1. 학습 데이터(GitHub, 문서, 튜토리얼)에 대한 **패턴 매칭**
2. 통계적으로 가장 가능성 높은 다음 토큰 **예측**
3. 독립적으로 "합리적으로 보이는" 것에 대한 **최적화**

"카드 컴포넌트"를 요청하면 AI는 본 수천 개의 카드 구현에서 샘플링합니다:

- Material Design 카드 (Google의 패턴)
- Tailwind UI 예시
- shadcn/ui 컴포넌트
- 무작위 GitHub 저장소
- Stack Overflow 스니펫

각각은 다른 것을 사용합니다:
- 색상 팔레트 (`blue-500` vs. `indigo-600` vs. `#4F46E5`)
- 간격 스케일 (`p-4` vs. `p-6` vs. `px-5 py-3`)
- Border radius 값 (`rounded-md` vs. `rounded-lg` vs. `rounded-xl`)
- Typography 스케일 (`text-sm` vs. `text-base` vs. `text-lg`)

**AI는 어느 것이 당신의 패턴인지 알 방법이 없습니다.**

### 컨텍스트 윈도우 제한

디자인 시스템을 채팅에 붙여넣어도 두 가지 문제에 직면합니다:

**1. 컨텍스트 감쇠**

대화에서 디자인 시스템이 뒤로 갈수록 AI 생성에서 "가중치"가 줄어듭니다:

```
메시지 1: [디자인 토큰 - 2000 토큰]
메시지 2-5: [기타 구현 세부사항]
메시지 6: "새 카드 컴포넌트 만들어줘"
         ↑
         AI가 토큰을 "잊어버림"
```

**2. 토큰 예산**

모든 메시지가 토큰을 소비합니다. 전체 디자인 시스템(색상, 간격, 타이포그래피, 컴포넌트) 붙여넣기는:
- 포괄적인 design tokens에 2,000~5,000 토큰 사용
- 실제 구현 컨텍스트에 쓸 공간 감소
- 비용이 빠르게 증가 (특히 Claude Opus / GPT-4)

### 암시적 가정 문제

AI는 명시하지 않으면 "합리적인 기본값"으로 공백을 채웁니다:

**당신:** "버튼 만들어줘"

**AI가 가정:**
- Primary color: 아마도 파란색 (`blue-600`)
- Padding: 아마도 `px-4 py-2` (Tailwind 기본값)
- Border radius: 아마도 `rounded-md`
- Font weight: 아마도 `font-medium`

이런 가정은 **학습 데이터의 통계적 빈도**에 기반하며, 당신의 디자인 결정이 아닙니다.

## 실제 일관성 문제 사례

### 사례 1: 색상 표류

Claude Code로 2주 동안 분석 대시보드 개발:

**1주차 컴포넌트:**
- Primary action에 `blue-600` 사용
- 배경에 `gray-100`
- Success 상태에 `green-500`

**2주차 컴포넌트 (같은 프롬프트, 새 채팅):**
- Primary action에 `indigo-600` 사용
- 배경에 `slate-50`
- Success 상태에 `emerald-500`

**결과:** 앱이 두 명의 다른 사람이 디자인한 것처럼 보임.

### 사례 2: 간격 혼란

Cursor로 SaaS 온보딩 플로우 개발:

```tsx
// Step 1 (월요일 생성)
<div className="p-6 space-y-4">

// Step 2 (화요일 생성)  
<div className="p-8 space-y-6">

// Step 3 (수요일 생성)
<div className="p-4 space-y-3">
```

같은 플로우, 같은 디자이너 의도, **세 가지 다른 간격 시스템**.

### 사례 3: 타이포그래피 정글

ChatGPT에게 폼 컴포넌트 생성 요청:

- Input label: `text-sm font-medium text-gray-700`
- Error message: `text-xs text-red-600`
- Helper text: `text-sm text-gray-500`
- Placeholder: `text-base text-gray-400`

지원 텍스트에 네 가지 다른 폰트 크기. 시스템 없이 감으로.

## 현재 솔루션이 작동하지 않는 이유

### ❌ 솔루션 1: "AI에게 일관성 유지하라고 말하기"

```
프롬프트: "모든 컴포넌트에서 일관된 간격과 색상 사용"
```

**문제:** "일관된"이 모호합니다. AI는 하나의 응답 내에서는 내부적으로 일관되지만 여러 생성에 걸쳐서는 아닙니다.

### ❌ 솔루션 2: "매번 디자인 시스템 붙여넣기"

```
프롬프트: [2000 토큰의 디자인 시스템]
"이제 이 토큰을 사용해서 버튼 만들어줘"
```

**문제:**
- 지루하고 느림
- 토큰 한도 소진
- 1~2명 개발자를 넘어 확장되지 않음
- 긴 대화에서는 여전히 컨텍스트 감쇠 발생

### ❌ 솔루션 3: "AI 내장 메모리 사용"

Claude Projects와 ChatGPT Memory가 도움이 되지만:
- 제한된 메모리 크기 (포괄적인 디자인 시스템 보유 불가)
- 프로그래밍 방식 접근 없음 ("primary color가 뭐지?" 쿼리 불가)
- 일관성 없는 회상 (색상 팔레트는 기억하지만 간격 스케일은 잊을 수 있음)

### ❌ 솔루션 4: "모델 파인튜닝"

기술적으로 가능하지만:
- ML 전문지식 필요
- 비쌈 (수천 달러)
- 디자인 시스템 업데이트마다 재학습 필요
- 99% 팀에게는 과함

## Design Token 솔루션

Design tokens은 AI에게 디자인 시스템에 대한 **프로그래밍 방식 접근**을 제공하여 문제를 해결합니다.

### 작동 방식: MCP (Model Context Protocol)

채팅에 토큰을 붙여넣는 대신, 디자인 시스템을 AI 도구에 **연결**합니다:

```
┌──────────────┐         MCP          ┌─────────────────┐
│   Cursor     │ ◄─────────────────► │  Design Token   │
│   Claude     │  "primary가 뭐야?"   │     Server      │
│   ChatGPT    │ ◄─────────────────► │  (FramingUI)    │
└──────────────┘    반환: #4338CA      └─────────────────┘
```

**이전 (복사-붙여넣기 방식):**
```
사용자: "브랜드 색상으로 버튼 만들어줘"
AI: *추측* → bg-blue-600 생성
```

**이후 (MCP 방식):**
```
사용자: "브랜드 색상으로 버튼 만들어줘"
AI: *토큰 서버 쿼리* → primary: '#4338CA' 수신
AI: 생성 → bg-[#4338CA] 또는 커스텀 Tailwind 클래스 bg-brand-primary 사용
```

### FramingUI 방식

FramingUI는 design tokens을 쿼리 가능한 리소스로 노출하는 MCP 서버를 제공합니다:

**1. 토큰 한 번 정의**

```typescript
// tokens/config.ts
export const tokens = {
  colors: {
    brand: {
      primary: '#4338CA',
      secondary: '#7C3AED',
    },
    neutral: {
      50: '#F9FAFB',
      900: '#111827',
    }
  },
  spacing: {
    sm: '0.5rem',
    md: '1rem',
    lg: '1.5rem',
  },
  typography: {
    fontFamily: {
      sans: 'Inter, system-ui',
    },
    fontSize: {
      sm: '0.875rem',
      base: '1rem',
      lg: '1.125rem',
    }
  }
}
```

**2. AI가 자동으로 토큰 쿼리**

Cursor나 Claude Code에게 컴포넌트 생성을 요청하면:
1. UI 컴포넌트 작업 중임을 감지
2. 토큰 서버 쿼리: "사용 가능한 색상이 뭐지?"
3. 토큰 구조 수신
4. 일반 토큰이 아닌 당신의 토큰을 사용하여 코드 생성

**3. 매번 일관된 출력**

```tsx
// 첫 생성 (1일차)
<button className="bg-brand-primary text-white px-md py-sm rounded-md">
  Primary Button
</button>

// 두 번째 생성 (30일차, 다른 채팅 세션)
<button className="bg-brand-primary text-white px-md py-sm rounded-md">
  Another Button
</button>
```

같은 클래스, 같은 디자인 언어, **복사-붙여넣기 제로**.

## 토큰 인식 AI 설정하기 (5분)

### 1단계: FramingUI 설치

```bash
npm install @framingui/tokens @framingui/mcp-server
```

### 2단계: 토큰 정의

```typescript
// lib/tokens.ts
import { createTokens } from '@framingui/tokens'

export const tokens = createTokens({
  colors: {
    brand: {
      primary: '#4338CA',
      secondary: '#7C3AED',
    },
    // ... 색상 시스템
  },
  spacing: {
    xs: '0.25rem',
    sm: '0.5rem',
    md: '1rem',
    lg: '1.5rem',
    xl: '2rem',
  }
})
```

### 3단계: MCP 서버 설정

```json
// claude_desktop_config.json (Claude Code용)
{
  "mcpServers": {
    "framingui": {
      "command": "npx",
      "args": ["@framingui/mcp-server", "--tokens", "./lib/tokens.ts"]
    }
  }
}
```

Cursor의 경우 `.cursorrules`에 추가:

```
이 프로젝트는 FramingUI design tokens을 사용합니다.
UI 코드를 생성하기 전에 항상 사용 가능한 토큰을 쿼리하세요.
하드코딩된 값 대신 토큰 변수를 사용하세요.
```

### 4단계: 테스트

**프롬프트:** "primary 버튼 만들어줘"

**AI 출력 (MCP 이전):**
```tsx
<button className="bg-blue-600 px-4 py-2">Click me</button>
```

**AI 출력 (MCP 이후):**
```tsx
<button className="bg-brand-primary px-md py-sm">Click me</button>
```

## 미래: 완전 자율 디자인 시스템

다음 단계는 디자인 시스템을 **읽는** 것뿐만 아니라 **유지보수**하는 AI입니다:

**현재 (2026):**
- AI가 토큰 읽기 → 일관된 코드 생성 ✅
- 토큰을 수동으로 업데이트
- 새 컴포넌트를 수동으로 생성

**가까운 미래 (2026-2027):**
- AI가 토큰 업데이트 제안: "이번 주에 `#4F46E5`를 5번 사용했습니다. `brand.accent`로 추가할까요?"
- AI가 컴포넌트 변형 생성: "`ghost` 버튼 변형이 필요한 것 같습니다. 기존 primary 버튼 기반 구현을 제시합니다."
- AI가 일관성 감사: "12개 컴포넌트가 하드코딩된 색상을 사용합니다. 토큰 사용하도록 리팩터링할까요?"

**장기 (2027+):**
- 사용 패턴에 따라 자율적으로 진화하는 디자인 시스템
- 표류를 감지하고 자체 수정하는 AI
- 자연어 디자인 시스템 쿼리: "deprecated spacing 값을 사용하는 모든 컴포넌트 보여줘"

## AI + Design Tokens 모범 사례

### 1. 컴포넌트가 아닌 토큰부터 시작

AI가 컴포넌트를 생성하기 전에 색상 팔레트, 간격 스케일, 타이포그래피 시스템을 정의하세요. 그렇지 않으면 AI 출력에서 디자인 시스템을 역설계하는 것입니다.

### 2. 시맨틱 네이밍 사용

```typescript
// ❌ 나쁨 (AI가 의도를 추론할 수 없음)
colors: {
  blue500: '#3B82F6',
  blue600: '#2563EB',
}

// ✅ 좋음 (AI가 목적을 이해)
colors: {
  brand: {
    primary: '#2563EB',
    primaryHover: '#1D4ED8',
  }
}
```

### 3. 사용 컨텍스트 제공

```typescript
// ✅ AI 컨텍스트를 위한 주석 포함
export const tokens = {
  spacing: {
    sm: '0.5rem',  // 컴팩트 UI, 모바일용
    md: '1rem',    // 대부분 컴포넌트의 기본 간격
    lg: '1.5rem',  // 여유 있는 레이아웃, 마케팅 페이지용
  }
}
```

### 4. 린팅과 통합

AI의 환각을 자동으로 감지:

```javascript
// eslint-plugin-design-tokens
{
  "rules": {
    "design-tokens/no-hardcoded-colors": "error",
    "design-tokens/no-arbitrary-spacing": "warn"
  }
}
```

### 5. 정기적으로 감사

```bash
# 코드베이스에서 하드코딩된 색상 찾기
npx @framingui/audit colors

# 출력:
# ❌ src/components/Button.tsx:12 - 하드코딩됨 #3B82F6
# ❌ src/pages/Dashboard.tsx:45 - 하드코딩됨 rgb(59, 130, 246)
# ✅ 47개 컴포넌트가 토큰을 올바르게 사용
```

## 결론

AI는 UI 코드 생성에 놀랍습니다—하지만 가드레일 없이는 일관성 유지에 형편없습니다.

**문제는 AI의 지능이 아닙니다. AI의 컨텍스트 부족입니다.**

Design tokens + MCP는 AI가 매번 디자인 시스템과 일치하는 코드를 생성하는 데 필요한 컨텍스트를 제공합니다.

**토큰 없이:** AI는 스타일 가이드를 계속 잊어버리는 주니어 개발자입니다.

**토큰 있이:** AI는 디자인 시스템을 완벽히 아는 시니어 개발자입니다.

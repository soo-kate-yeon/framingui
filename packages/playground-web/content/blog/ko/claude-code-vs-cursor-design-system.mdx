디자인 시스템을 만들다 보면 모든 AI 코딩 도구의 특정 약점이 드러나요. 일반적인 코드 생성과 내 아키텍처에 실제로 맞는 코드 생성 사이의 간극이에요. Cursor와 Claude Code는 그 간극을 다른 방식으로 좁히는데, 디자인 시스템 작업에서 그 차이가 중요해요.

## 핵심 차이

**Cursor**는 속도에 최적화되어 있어요. 인라인으로 코드를 제안하고, 키 하나로 변경을 수락하고, 에디터 흐름을 절대 끊지 않죠. 디자인 시스템에서의 약점은 이거예요: 제안하기 전에 전체 코드베이스를 읽지 않아요. 최근에 연 파일들과 학습 데이터의 일반적인 패턴에 의존해요.

**Claude Code**는 일관성에 최적화되어 있어요. 뭔가를 제안하기 전에 전체 프로젝트를 읽어요 — 토큰 구조, 네이밍 규칙, 기존 컴포넌트 API. 약점은 느리다는 거예요. 요청을 작성하고, 응답을 기다리고, diff를 검토하고, 적용해요. 인라인 제안은 없어요.

디자인 시스템에서는 대체로 속도보다 일관성이 더 중요해요. 기존 패턴에 맞는 컴포넌트가 30초 더 빨리 생성된 것보다 가치 있거든요.

## Claude Code가 구조적 우위를 갖는 이유

디자인 시스템은 아직 건드리지 않은 파일들 전반에 걸쳐 일관성이 필요해요. Button에 `size` variant를 추가할 때 Claude Code는 기존 컴포넌트를 읽고 네이밍 규칙에 맞춰요 — `small`/`medium`/`large`가 아닌 `sm`/`md`/`lg`로요. HSL에서 OKLCH로 색상 토큰 구조를 리팩토링할 때 Claude Code는 그 토큰을 참조하는 모든 파일에 걸쳐 변경을 조율해요.

MCP(Model Context Protocol)도 마찬가지예요. Claude Code는 파일시스템, 데이터베이스, Git, 커스텀 서버 등 MCP 서버에 네이티브로 연결해요. 디자인 토큰 소스에 연결해서 정적 파일 대신 현재 토큰 상태를 직접 쿼리할 수 있어요. Cursor에도 커뮤니티 MCP 지원이 있지만 1급 기능은 아니에요.

FramingUI의 MCP 서버를 사용하는 팀이라면 Claude Code가 생성하는 동안 사용 가능한 테마, 컴포넌트, 토큰 값을 직접 쿼리할 수 있어요 — 디자인 시스템의 현재 상태와 정확히 일치하는 결과를 만들어내죠.

## Cursor가 실용적 우위를 갖는 이유

명확한 스펙에서 잘 정의된 컴포넌트를 구현할 때 Cursor의 탭 자동완성이 체감상 더 빠른 건 사실이에요. 함수 시그니처를 타이핑하면 Cursor가 본문을 채워주고 Tab을 누르면 끝이에요. 같은 API를 공유하는 폼 인풋 10개를 만들 때 이 루프는 상당한 시간 절약으로 이어져요.

Cursor Composer 모드의 멀티 파일 편집도 특정 작업에는 강해요 — 예를 들어 폴더 안의 모든 컴포넌트에 `data-testid` prop을 추가하는 것처럼요. 변경 사항을 설명하고 파일을 선택하면 적용돼요.

## 구체적인 작업 비교

**시나리오: 새 디자인 시스템 설정**

Claude Code: 요구사항을 설명하면 시맨틱 토큰 구조를 제안하고 트레이드오프를 설명하고 토큰 파일을 만들고 TypeScript 타입이 있는 기본 컴포넌트를 구축하고 문서를 작성해요. 기본적으로 일관성이 보장돼요.

Cursor: 시작이 더 빨라요. 파일을 만들면 Cursor가 공격적으로 자동완성해줘요. 근데 적극적으로 관리하지 않으면 파일 간에 토큰 이름이 어긋날 수 있고, 아키텍처 결정은 직접 해야 해요.

**시나리오: 기존 컴포넌트에 variant 추가**

Claude Code: 요청하면 기존 variants를 읽고 패턴에 맞춰요. diff 검토 포함 2분.

Cursor: 파일을 열고 타이핑을 시작하면 Cursor가 variant 로직을 인라인으로 제안해요. 30초.

독립적인 변경에는 Cursor가 이겨요. 큰 기존 시스템과 일관성을 유지해야 하는 변경에는 Claude Code가 이겨요.

**시나리오: 20개 폼 컴포넌트를 일관된 API로 리팩토링**

Claude Code: 요청 하나로 20개 컴포넌트 전부를 일관성 있게 생성하거나 업데이트해요.

Cursor: 첫 번째 컴포넌트는 빠르게 나와요. 두 번째에서 조금 어긋나요. 열 번째쯤 되면 prop 이름을 수동으로 맞추고 있어요.

여러 파일에 걸친 일관성이 목표일 때 Claude Code가 압도적으로 이겨요.

## 실용적인 워크플로우

두 도구를 모두 쓰는 팀들은 자연스럽게 역할 분담이 생겨요.

Claude Code는 아키텍처와 시스템 레벨 결정을 담당해요 — 토큰 구조 설계, 컴포넌트 API 표면 계획, 컴포넌트 작업 스프린트 후 일관성 검토, 대규모 리팩토링 실행.

Cursor는 컴포넌트 구현을 담당해요 — 아키텍처가 정해진 후 개별 컴포넌트를 만드는 빠르고 반복적인 작업.

이건 임의적인 분담이 아니에요. 각 도구가 실제로 잘하는 것에서 자연스럽게 나오는 거예요. 버튼 하나 추가할 때마다 Claude Code를 쓰거나, 토큰 아키텍처 재설계에 Cursor를 쓰는 건 도구의 강점에 역행하는 거예요.

## Claude Code 설정하기

FramingUI MCP 서버를 Claude Code와 함께 쓴다면 `.mcp.json` 설정은 이렇게 돼요:

```json
{
  "mcpServers": {
    "framingui": {
      "type": "stdio",
      "command": "npx",
      "args": ["-y", "@framingui/mcp-server@latest"]
    }
  }
}
```

`npx -y @framingui/mcp-server@latest init`을 실행하면 이 설정이 자동으로 완료돼요. Claude Code를 재시작하면 사용 가능한 테마, 컴포넌트, 토큰 값을 직접 쿼리할 수 있어서 Claude의 학습 데이터 근사값이 아닌 실제 디자인 시스템에 맞는 코드가 나와요.

두 도구는 경쟁자가 아니에요. 서로를 보완하는 관계예요. 둘 다 사용해서 만든 디자인 시스템이 어느 하나만 쓴 것보다 더 잘 설계되고 더 빠르게 만들어져요.

AI 코딩 세션은 매번 제로에서 시작해요. 스키마 붙여넣기, 폴더 구조 설명, 에러 로그 복사, 인증 흐름 다시 설명. 효과는 있는데, 매번 수동으로 추가하는 마찰이에요.

Model Context Protocol(MCP)은 그 마찰을 없애줘요. AI 어시스턴트가 파일, 데이터베이스, API 같은 실제 개발 환경에 직접 연결해서 읽을 수 있게 해주는 오픈 표준이에요. 채팅으로 정보를 중계할 필요가 없어요.

## 구조

MCP는 클라이언트-서버 모델을 써요. **MCP 서버**는 로컬에서 실행되면서 정의된 인터페이스로 리소스를 노출하고, **MCP 클라이언트**(Claude Code, Cursor, 기타 호환 도구)는 그 리소스를 요청하고, **프로토콜** 자체는 요청과 응답이 어떻게 구조화되는지 정의하는 표준화된 JSON-RPC 메시지예요.

```plaintext
┌──────────────┐        MCP Protocol         ┌──────────────┐
│              │◄───────────────────────────►│              │
│  AI 클라이언트│  요청: "routes.ts 읽어줘"   │  MCP 서버    │
│ (Claude Code)│  응답: 파일 내용            │ (파일시스템) │
│              │                             │              │
└──────────────┘                             └──────────────┘
```

핵심은 표준화예요. MCP 이전에는 파일시스템을 읽으려는 AI 도구마다 커스텀 통합이 필요했어요. 데이터베이스를 쿼리하려면 자체 플러그인이 필요했고요. MCP는 인터페이스를 범용으로 만들어요 — MCP 서버는 어떤 MCP 클라이언트에서든 작동해요.

## 실제로 어떤 차이가 나나

구체적인 예시에서 차이가 확실히 드러나요. AI에게 API에 인증 미들웨어를 추가해달라고 해요.

MCP 없이: 학습 데이터가 대부분 그런 패턴이라 범용적인 Express.js JWT 미들웨어를 생성해요. 프로젝트는 커스텀 세션 플러그인을 쓴 Fastify를 쓰고 있어요. 생성된 코드가 안 맞아요. 설정을 설명하고, AI가 재생성하고, 반복하죠.

MCP 있으면: AI가 생성 전에 라우트 정의를 읽어요. 기존 Fastify 설정, 플러그인 구조, 인증이 적용될 위치를 봐요. 첫 번째 초안이 아키텍처에 맞아요.

품질 향상은 세션 전체에 걸쳐 복리로 쌓여요. 모든 후속 질문이 더 정확한 답을 받아요. 설명이 아니라 실제 파일로 작업하니까요.

## Claude Code와 파일시스템 연결하기

Claude Code는 MCP를 기본 지원해요. 설정은 프로젝트 루트의 `.mcp.json`에 저장돼요 (버전 관리에 체크인해두면 팀원들이 자동으로 같은 설정을 써요):

```json
{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/프로젝트/경로"]
    }
  }
}
```

Claude Code를 재시작하면 파일 목록 조회, 내용 읽기, 실제 코드 구조 기반 제안이 가능해져요. 수동 붙여넣기-설명 루틴이 사라져요.

## 데이터베이스 연결하기

스키마 설명은 AI 지원 개발에서 가장 큰 컨텍스트 오버헤드 중 하나예요. 데이터베이스 MCP 서버를 쓰면 AI가 스키마를 직접 쿼리해요:

```bash
npm install -g @modelcontextprotocol/server-postgres
```

```json
{
  "mcpServers": {
    "database": {
      "command": "mcp-server-postgres",
      "args": ["--connection", "postgresql://localhost/mydb"]
    }
  }
}
```

"orders 테이블에 어떤 외래 키가 있어?"가 설명 작업이 아니라 직접 쿼리가 돼요. AI가 읽기 전용 SQL을 실행하고 실제 스키마 기반 답변을 줘요.

## 여러 서버 동시 연결

AI 클라이언트는 여러 MCP 서버를 동시에 연결할 수 있어요. 코드를 위한 파일시스템 서버, 스키마 쿼리를 위한 데이터베이스 서버, 저장소 컨텍스트를 위한 Git 서버가 함께 있는 식이에요. 각 서버는 한 도메인에 집중하고, AI가 여러 관심사에 걸친 복잡한 질문에 답할 때 이들을 조율해요.

## 커스텀 서버 만들기

MCP의 진짜 힘은 확장성이에요. 내부 API나 사내 도구, 비표준 형식의 문서가 있다면 커스텀 서버로 노출할 수 있어요.

프로젝트 문서를 노출하는 최소 예시:

```typescript
import { Server } from '@modelcontextprotocol/sdk';

const server = new Server({ name: 'docs-server', version: '1.0.0' });

server.resource('docs://architecture', async () => ({
  contents: await readFile('./docs/architecture.md'),
  mimeType: 'text/markdown',
}));

server.listen();
```

다른 서버와 같은 방식으로 설정하면 돼요. 이제 AI가 관련 섹션을 붙여넣길 기다리는 대신 `docs://architecture`를 직접 요청할 수 있어요.

## 디자인 시스템용 MCP

프론트엔드 개발자에게는 디자인 시스템 MCP 서버가 토큰 정의와 생성된 코드 사이의 간격을 좁혀줘요. FramingUI가 정확히 이 목적으로 `@framingui/mcp-server`를 제공해요.

설정은 명령어 하나예요:

```bash
npx -y @framingui/mcp-server@latest init
```

이 명령이 올바른 설정으로 프로젝트의 `.mcp.json`에 서버를 추가해요:

```json
{
  "mcpServers": {
    "framingui": {
      "type": "stdio",
      "command": "npx",
      "args": ["-y", "@framingui/mcp-server@latest"]
    }
  }
}
```

연결되면 Claude Code가 UI 컴포넌트를 생성하기 전에 디자인 토큰을 쿼리해요. 생성된 코드가 Tailwind 기본값 대신 여러분의 색상 토큰, 스페이싱 스케일, 타이포그래피 정의를 써요. 컴포넌트 첫 초안이 수동 수정 없이도 디자인 시스템과 맞아요.

## 보안 고려사항

MCP 서버는 여러분 사용자 계정의 권한으로 실행돼요. 안전하게 유지하는 몇 가지 방법: 파일시스템 접근을 특정 프로젝트 디렉토리로 제한하고, AI 쿼리 접근에는 읽기 전용 데이터베이스 연결을 쓰고, 실행하는 MCP 서버 코드를 맹목적으로 신뢰하지 말고 검토하세요. 공식 `@modelcontextprotocol` 서버는 오픈소스이고 검증이 잘 돼 있어요. 서드파티 서버는 파일이나 데이터베이스 접근을 허용하기 전에 검토가 필요해요.

## MCP vs 다른 컨텍스트 방식

채팅에 컨텍스트를 복사-붙여넣기하는 건 수동이고 얼마나 빠뜨릴지 한도가 없어요. RAG(검색 증강 생성)는 대형 정적 지식 베이스에 유용하지만 인덱싱 인프라가 필요하고 오래된 내용을 제공할 수 있어요. MCP는 실시간 구조화된 접근을 제공해요 — AI가 지난주 인덱스 스냅샷이 아니라 파일과 데이터베이스의 현재 상태를 봐요.

트레이드오프는 설정 오버헤드예요. MCP는 서버를 실행해야 해요. 가끔 AI를 쓴다면 복사-붙여넣기로 충분해요. 매일 AI 지원 개발을 한다면 설정 비용이 몇 시간 안에 회수돼요.

새 프로젝트를 시작할 때마다 같은 의식을 치르게 되죠. `tailwind.config.js` 편집, 라이트/다크 모드 CSS 변수 정의, TypeScript 타입 수동 동기화, 그리고 다음 주 디자이너가 색상 하나 바꿀 때 아무것도 안 깨지길 기도하기. FramingUI는 이 과정 전체를 없앱니다.

## 세 개의 진실 공급원 문제

기존 테마 설정은 세 개의 파일을 동시에 관리하도록 강제해요.

Tailwind 설정 파일에 원시 색상 값이 들어가고, 별도의 CSS 변수 파일이 그 값을 의미론적 이름에 매핑하고, TypeScript 선언 파일이 자동완성을 두 파일과 동기화하려고 분투합니다. 디자이너가 브랜드 컬러 하나를 바꾸면 세 파일을 모두 수정해야 하는데, 어딘가 빠뜨리지 않았길 바라는 수밖에 없어요.

설정이 많아 보이고 뭔가 되는 것 같지만, 결과는 하나예요. 세 파일 중 어느 곳에 오타가 하나라도 있으면 조용히 실패하거나 스타일이 깨지는 취약한 시스템이 나옵니다.

## FramingUI가 작동하는 방식

FramingUI는 단일 런타임 계약으로 프로젝트를 연결해요. 명령어 하나가 모든 것을 연결합니다.

```bash
npx -y @framingui/mcp-server@latest init
```

이 명령어는 프레임워크(Next.js 또는 Vite)를 감지하고, `@framingui/ui`와 `tailwindcss-animate`를 설치하고, 전역 스타일시트에 CSS 임포트를 추가하고, 앱 루트에 `FramingUIProvider`를 마운트하고, 로컬 `framingui-theme` 모듈을 생성하고, `.mcp.json`에 MCP 서버를 등록합니다. 열 단계가 명령어 하나로 끝나요.

런타임 계약은 명확하고 최소한이에요:

- 전역 스타일시트에 `@import '@framingui/ui/styles'`가 정확히 한 번 존재
- 앱 루트에 `FramingUIProvider`가 정확히 한 번 마운트
- 로컬 `framingui-theme` 모듈이 테마 설정을 provider에 전달
- 오버레이 애니메이션을 위한 `tailwindcss-animate` 설치

테마를 바꾸려면 `framingui-theme` 모듈 하나만 수정하면 돼요. 나머지는 자동으로 업데이트됩니다.

## 토큰은 CSS 변수예요

FramingUI는 디자인 토큰을 CSS 커스텀 속성으로 노출합니다. hex 값이 하드코딩된 JavaScript 객체가 아니에요. 테마 파일에는 `{ background: '#ffffff' }` 같은 게 아니라 `var(--background-page)`, `var(--foreground-accent)` 같은 참조가 들어가죠.

AI가 코드를 생성할 때 이 차이가 중요해요. Claude가 MCP 서버를 통해 디자인 시스템을 쿼리하면, 추측이 필요한 원시 색상 값이 아니라 의미론적 의미를 가진 실제 CSS 변수 이름을 받습니다. primary 버튼 배경에 어떤 토큰이 적용되는지 물으면 명확한 답이 나와요. 어림짐작도, 환각도 없습니다.

## 설정 파일 없이 테마 추가하기

기본 `square-minimalism` 테마에서 `dark-boldness`로 바꾸고 싶다고 해봐요. 기존 방식이라면 Tailwind, CSS 변수, TypeScript 선언을 모두 업데이트해야 합니다. FramingUI에서는 `framingui-theme` 모듈의 값 하나만 바꾸면 돼요.

FramingUI는 여섯 개의 프로덕션 테마를 제공합니다: `classic-magazine`, `dark-boldness`, `minimal-workspace`, `neutral-workspace`, `pebble`, `square-minimalism`. 각 테마는 완전히 라이선스가 있고 인증 후 MCP 서버를 통해 접근할 수 있어요.

```bash
framingui-mcp login
```

인증은 `~/.framingui/credentials.json`에 저장된 API 키를 사용합니다. 인증 후 Claude Code는 테마 미리보기, 컴포넌트 목록 조회, 프로젝트가 참조하는 라이선스 테마로 스크린 블루프린트 생성을 할 수 있어요.

## MCP 서버가 제공하는 것

MCP 서버는 Claude Code와 다른 MCP 호환 편집기에 16개 도구를 노출합니다. 도구는 네 범주로 나뉘어요: 블루프린트 및 테마 탐색, 아이콘 라이브러리 브라우징, 4단계 스크린 생성 워크플로우, 환경 검증입니다.

스크린 생성 워크플로우가 FramingUI를 복사-붙여넣기 방식과 구분 짓는 핵심이에요.

1. `get-screen-generation-context` — 자연어 설명에 매칭되는 템플릿과 컴포넌트 조회
2. `validate-screen-definition` — 생성 전 구조화된 스크린 정의 JSON 검증
3. `generate_screen` — 검증된 정의에서 React 컴포넌트 코드 생성
4. `validate-environment` — 프로젝트에 필요한 패키지와 Tailwind 설정 확인

환경 검증 단계가 특히 주목할 만해요. 패키지 설치 여부만 확인하는 게 아니라 런타임 계약 조건을 확인합니다. 올바른 패키지가 있어도 연결 설정이 잘못되면 런타임에 실패할 수 있거든요.

## Tailwind v4 프로젝트라면

Tailwind v4 프로젝트는 `@framingui/ui`를 위한 수동 `@source` 항목이 필요 없어요. `@import '@framingui/ui/styles'` 한 줄이 내부적으로 스캐닝 계약을 포함합니다. 추가 항목을 넣어도 문제가 생기진 않지만, 설정을 최소화하려면 그냥 두는 게 나아요.

설치 후 버튼 패딩이나 다이얼로그 애니메이션이 이상해 보인다면, 패키지를 살피기 전에 런타임 계약을 먼저 확인하세요. 스타일시트 임포트, provider 마운트, 테마 모듈, animate 패키지—이 네 조건이 대부분의 런타임 문제를 커버합니다.

## 설치 경로

새 프로젝트라면 전체 설정 순서는 이렇습니다:

```bash
npx -y @framingui/mcp-server@latest init
framingui-mcp login
```

Claude Code를 재시작하고 스크린 생성을 요청하면 돼요. MCP 서버가 나머지를 처리합니다.

MCP 워크플로우 없이 React 컴포넌트만 필요한 기존 프로젝트라면:

```bash
pnpm add @framingui/ui @framingui/core @framingui/tokens tailwindcss-animate
```

테마 설정이 제품 개발에서 가장 어려운 부분이 될 필요는 없어요. FramingUI는 그것을 가장 빠른 부분으로 만듭니다.

## 작은 팀의 딜레마

3~5명으로 SaaS 제품을 만들고 있습니다. 디자이너 1명(있으면 다행), 개발자 2명, 그리고 나머지를 모두 담당하는 파운더. 1분 1초가 중요하고, 돈 한 푼도 아껴야 합니다.

누군가 제안합니다. "디자인 시스템을 도입하면 어떨까요?"

당신의 즉각적인 반응: **"그럴 시간이 없어요. 디자인 시스템은 전담 팀이 있는 큰 회사나 하는 거죠."**

이 글은 실제 수치로 그 생각이 틀렸음을 증명합니다.

## 핵심 요약

- 작은 팀(2~5명)은 기본 디자인 시스템 도입 후 **월 15~25시간 절약**
- 평균 투자금 회수 기간: 초기 설정 시간의 **2~4주**
- ROI는 시간이 지날수록 증가: 대부분의 팀에서 첫 해 300~500%
- Figma + Storybook + 문서화가 모두 필요하지 않습니다—design tokens + 컴포넌트 라이브러리로 시작하세요
- **FramingUI** 같은 도구를 사용하면 설정 시간이 몇 주에서 몇 시간으로 단축됩니다

## 디자인 시스템이 없을 때의 실제 비용

디자인 시스템 없이 개발할 때 무슨 일이 벌어지는지 추적해봅시다. 실제 팀들의 데이터입니다.

### 시나리오: 새 기능 개발

**1주차: 초기 구현**

- 디자이너가 기존과 살짝 다른 버튼 스타일로 목업 작성 (어느 게 "맞는" 건지 논의하는 데 **2시간**)
- 개발자가 다른 곳에 쓰인 `#4338CA` 대신 `#4F46E5`로 구현 (코드 리뷰에서 수정하는 데 **30분**)
- 간격이 나머지 앱과 맞지 않음—어떤 카드는 `16px`, 다른 건 `20px`, 이번 건 `18px` (Figma 댓글에서 논쟁하는 데 **1시간**)

**2주차: 반복**

- 프로덕트 매니저가 브랜드 전체의 primary button 색상 변경 요청
- 개발자가 **12개 파일**의 버튼 업데이트 (**3시간**)
- 3곳을 누락; 버그 리포트 도착 (찾아서 고치는 데 **1시간**)

**3주차: 계약직 온보딩**

- 새 개발자가 질문: "spacing scale이 뭔가요?" 아무도 모름. (기존 코드를 역분석하는 데 **2시간**)
- "어떤 파란색을 써야 하나요?" 코드베이스에 파란색이 7가지. (Slack 논의 **1시간**)

**월간 비용: 3명이 낭비하는 시간 약 11시간**. 팀의 시간당 요율에 따라 **약 $1,100~$2,200** (스타트업 기준 $100~$200/시간).

연간: **$13,200~$26,400의 낭비**.

## 디자인 시스템의 실제 비용

설정 시간과 유지보수에 대해 현실적으로 살펴봅시다.

### 초기 설정 (최신 도구 사용)

**옵션 A: 처음부터 직접 구축**
- Design tokens 정의 (colors, spacing, typography): **8시간**
- 핵심 컴포넌트 구축 (Button, Input, Card 등): **24시간**
- 문서화: **8시간**
- **총: 40시간 (약 1주)**

**옵션 B: 기반 활용 (FramingUI + shadcn/ui)**
- FramingUI design tokens 설치 및 설정: **2시간**
- 토큰을 브랜드에 맞게 커스터마이징: **3시간**
- shadcn/ui 컴포넌트 가져오기 및 조정: **3시간**
- 기본 문서화: **2시간**
- **총: 10시간 (약 1.5일)**

### 지속적인 유지보수

- 월간 업데이트/추가: **2~4시간**
- 분기별 리뷰: **4시간**
- **연간: 약 30~40시간**

## ROI 계산

3인 팀(디자이너 1명, 개발자 2명)의 첫 해 비용 vs 절감액을 비교해봅시다.

### 디자인 시스템 없이

| 비용 항목 | 시간/년 | 비용 (@$150/시간) |
|----------|---------|------------------|
| 반복적인 논의 | 48 | $7,200 |
| 일관성 수정 | 36 | $5,400 |
| 온보딩 비효율 | 24 | $3,600 |
| 디자인-개발 핸드오프 문제 | 40 | $6,000 |
| **합계** | **148시간** | **$22,200** |

### 디자인 시스템 사용 시 (옵션 B)

| 비용 항목 | 시간/년 | 비용 (@$150/시간) |
|----------|---------|------------------|
| 초기 설정 | 10 | $1,500 |
| 유지보수 | 35 | $5,250 |
| 잔여 비효율 (20%) | 30 | $4,500 |
| **합계** | **75시간** | **$11,250** |

**첫 해 절감액: $10,950**  
**ROI: 97%**

핵심은 **절감액이 누적된다**는 점입니다.

## 2년차 이후: 복리 효과

2년차에는 이미 설정 비용을 지불했습니다. 유지보수는 비슷하지만, 이점은 커집니다:

### 2년차 이후의 새로운 이점

1. **더 빠른 기능 개발**: 컴포넌트가 미리 구축되고 테스트됨
   - 평균 구현 시간 20% 단축
   - 월 2개 기능 출시 팀 기준: **월 약 15시간 절약**

2. **확장 용이**: 4번째 또는 5번째 팀원 추가
   - 온보딩 시간 50% 단축 (**신규 채용당 약 12시간 절약**)
   - 새 개발자가 2주가 아닌 첫날부터 생산적

3. **AI 기반 개발**: MCP를 사용하는 Cursor/Claude Code 같은 도구
   - AI가 자동으로 브랜드에 맞는 코드 생성
   - 수동 토큰 교체에서 **월 약 8시간 절약**

**2년차 가치: $36,000 이상의 시간 절감**

## 작은 팀의 스위트 스팟: 무엇을 만들 것인가

모든 걸 다 할 필요는 없습니다. **80/20**에 집중하세요.

### 최소 기능 디자인 시스템 (10시간)

**1. Design Tokens (3시간)**
```typescript
// tokens/colors.ts
export const colors = {
  brand: {
    primary: '#4338CA',
    secondary: '#7C3AED',
  },
  neutral: {
    50: '#F9FAFB',
    100: '#F3F4F6',
    // ... 전체 스케일
    900: '#111827',
  },
  semantic: {
    success: '#10B981',
    error: '#EF4444',
    warning: '#F59E0B',
  }
}

// tokens/spacing.ts
export const spacing = {
  xs: '0.25rem',   // 4px
  sm: '0.5rem',    // 8px
  md: '1rem',      // 16px
  lg: '1.5rem',    // 24px
  xl: '2rem',      // 32px
}
```

**2. 핵심 컴포넌트 (5시간)**

가장 많이 쓰는 5개 컴포넌트에 집중:
- Button (primary, secondary, ghost variants)
- Input
- Card
- Modal
- Select/Dropdown

**3. 문서화 (2시간)**

README 하나에:
- Design tokens 사용법
- 컴포넌트 예시
- 새 컴포넌트를 만들어야 할 때 vs 기존 컴포넌트 사용할 때

### 아직 만들지 말아야 할 것

- ❌ Storybook (8명 이상으로 성장하면 추가)
- ❌ 자동화된 visual regression 테스트 (시기상조)
- ❌ 포괄적인 문서 사이트 (README로 충분)
- ❌ 자동 동기화되는 Figma 라이브러리 (좋긴 하지만 필수는 아님)

## 실제 사례 연구

### 사례 1: B2B SaaS (4인 팀)

**이전:**
- 개발자 3명, 디자이너 1명
- 프로젝트 관리 도구 개발
- 8개월 차, 코드베이스에 14가지 버튼 스타일

**이후 (FramingUI로 6주):**
- 설정: 2주에 걸쳐 12시간 (디자이너 + 개발자 1명)
- 3개월 후 결과:
  - 기능 개발 18% 빨라짐 (Jira 추정치로 측정)
  - 디자인 관련 버그 67% 감소
  - 계약직 2명을 기존의 1/3 시간에 온보딩

**ROI: 첫 해 340%**

### 사례 2: 모바일 앱 스타트업 (3인 팀)

**이전:**
- 개발자 2명, 파운더/디자이너 1명
- React Native 앱, 디자인 시스템 없음
- 모든 화면이 처음부터 시작하는 느낌

**이후 (FramingUI + React Native Paper 사용):**
- 설정: 1주에 걸쳐 8시간
- 6개월 후 결과:
  - 기존 컴포넌트로 12개 새 화면 구축 (없었다면 8개 예상)
  - 브랜드 리디자인이 예상 40시간이 아닌 4시간 소요 (토큰 변경으로 컴포넌트 자동 업데이트)
  - Series A 유치—투자자들이 "제품 완성도"를 핵심 요소로 언급

**ROI: 첫 해 520%**

## 의사결정 프레임워크: 당신은 디자인 시스템을 만들어야 할까요?

### 만들어야 하는 경우:

✅ 같은 코드베이스에서 일하는 개발자가 2명 이상  
✅ 향후 6개월 동안 5개 이상의 새 기능을 만들 계획  
✅ "어떤 파란색을 쓰지?"라고 말한 적이 있음  
✅ 채용하거나 계약직을 온보딩할 예정  
✅ AI 코딩 도구(Cursor, Claude Code)를 효과적으로 사용하고 싶음

### 만들지 말아야 하는 경우 (아직):

❌ Product/market fit 이전이고 2주마다 피봇 중  
❌ 혼자서 MVP를 만드는 중 (Tailwind + shadcn/ui 그대로 사용)  
❌ 제품이 순전히 내부 도구이고 사용자가 본인뿐

### 고려해볼 만한 경우:

⚠️ 곧 펀딩을 받을 예정 (데모에서 완성도가 중요)  
⚠️ 첫 디자이너 채용 예정 (먼저 기반을 설정)  
⚠️ 유사한 제품/화이트라벨이 3~5개 (재사용성 = 높은 ROI)

## 이번 주에 시작하는 방법

### 1일차: 점검 (2시간)

1. 가장 많이 쓰는 컴포넌트 10개 나열
2. 코드에 존재하는 primary color의 다양한 shade 개수 세기
3. 현재 spacing 값 문서화 (코드베이스에서 `px-`, `p-`, `m-` grep)

### 2~3일차: Tokens 설정 (4시간)

**옵션 1: FramingUI (권장)**

```bash
npm install @framingui/tokens
```

```typescript
// lib/tokens.ts
import { createTokens } from '@framingui/tokens'

export const tokens = createTokens({
  colors: {
    brand: {
      primary: '#4338CA',
      // 여기에 색상 추가
    }
  },
  // ... 나머지 디자인 결정사항
})
```

**옵션 2: CSS Variables로 직접 구축**

```css
/* styles/tokens.css */
:root {
  --color-primary: #4338CA;
  --color-secondary: #7C3AED;
  --spacing-md: 1rem;
  --font-sans: 'Inter', system-ui;
}
```

### 4~5일차: 상위 3개 컴포넌트 마이그레이션 (4시간)

처음부터 다시 만들지 마세요. 기존 컴포넌트를 토큰 사용하도록 리팩터링하세요:

```tsx
// Before
<button className="bg-blue-600 px-4 py-2">
  Click me
</button>

// After
import { tokens } from '@/lib/tokens'

<button style={{
  backgroundColor: tokens.colors.brand.primary,
  padding: `${tokens.spacing.sm} ${tokens.spacing.md}`
}}>
  Click me
</button>

// 또는 Tailwind + FramingUI 사용
<button className="bg-brand-primary px-md py-sm">
  Click me
</button>
```

### 2주차: 문서화 & 전파 (2시간)

저장소에 `DESIGN_SYSTEM.md` 생성:

````markdown
# 우리의 디자인 시스템

## Colors
- Primary: `tokens.colors.brand.primary`
- 용도: CTA, 링크, active 상태

## Spacing
- 임의의 값 대신 `tokens.spacing.*` 사용
- `px-[13]`이나 임의 숫자 사용 금지

## Components
- Button: `<Button variant="primary">` (`/components/Button.tsx` 참고)
- Input: `<Input />` (`/components/Input.tsx` 참고)

## 확실하지 않을 때
#design-system Slack 채널에 질문하거나 기존 컴포넌트를 먼저 확인하세요.
````

## 흔한 반론 (그리고 왜 틀렸는지)

### "우리는 디자인 시스템 쓰기엔 너무 작아요"

당신이 **완벽한** 크기입니다. 큰 회사는 50명에게 동의를 얻느라 고생합니다. 당신은 1주일에 기반을 설정하고 모두가 정렬될 수 있습니다.

### "Product/market fit 이후에 할게요"

나무 심기에 가장 좋은 시기는 20년 전이었습니다. 두 번째로 좋은 시기는 지금입니다. 기본 디자인 시스템(10시간)은 로드맵을 지연시키지 않지만 이후 모든 것을 빠르게 만듭니다.

### "디자이너가 Figma 파일만 관리하면 돼요"

Figma 파일은 코드의 source of truth가 아닙니다. 개발자는 추측하고, 하드코딩하고, 적당해 보이는 Tailwind 클래스를 사용합니다. 프로그래밍 방식의 토큰이 필요합니다.

### "Tailwind/Material-UI 그대로 쓸 거예요"

일반 라이브러리는 MVP에 적합하지만, 브랜드 차별화가 필요한 순간(펀딩, 마케팅, 사용자 신뢰를 위해 필요함)이 오면 커스터마이징이 필요합니다. 지금 토큰으로 시작하세요; 나중에 필요하면 컴포넌트를 교체하세요.

## 결론

3~5인 팀에게 디자인 시스템은 오버헤드가 아니라 **인프라**입니다.

데이터베이스, 인증 시스템, 배포 파이프라인 없이 SaaS 앱을 만들지 않을 것입니다. 디자인 시스템도 같은 범주의 기초 도구입니다.

**최소 투자: 10시간**  
**일반적인 투자금 회수: 2~4주**  
**첫 해 ROI: 100~500%**

질문은 "디자인 시스템을 만들 여유가 있을까?"가 아닙니다.

"만들지 *않을* 여유가 있을까?"입니다.

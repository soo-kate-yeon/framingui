# 작은 팀에서 디자인 시스템이 실제로 돈이 되는 이유

소규모 팀은 디자인 시스템을 전담 플랫폼 팀이 있는 대기업의 전유물이라고 생각하는 경향이 있어요. 이 가정이 꽤 비쌉니다. 진짜 질문은 디자인 시스템을 만들 여유가 있냐가 아니라, 없어서 지금 얼마를 낭비하고 있느냐예요.

이 글에서는 그 비용이 어디서 발생하는지, 3~5인 팀에게 현실적인 최소 투자가 뭔지 살펴봅니다.

## 디자인 시스템 없이 시간이 어디로 사라지는가

낭비는 눈에 잘 띄지 않아요. 한 번에 한 시간씩 날아가는 게 아니라 여기저기서 조금씩 새거든요.

디자이너가 기존과 약간 다른 버튼 변형으로 목업을 만들었어요. 어느 게 표준인지 확실하지 않아서 팀이 Figma 댓글에서 시간을 씁니다. 개발자는 토큰 파일 대신 인스펙터를 확인하다가 기존에 쓰던 `#4338CA` 대신 `#4F46E5`로 구현해요. 코드 리뷰에서 오류가 발견되고, 수정에 또 시간이 들죠.

2주 뒤, 제품 결정으로 버튼 색상을 전체적으로 바꿔야 합니다. 개발자가 12개 파일을 뒤지고 업데이트했는데, 3곳을 빠뜨렸어요. 버그 리포트가 들어옵니다.

계약직이 합류하면 "spacing scale이 어떻게 되나요?", "파란색은 어떤 걸 쓰나요?" 같은 질문이 쏟아집니다. 정답을 아는 사람이 없어서, 새 사람은 기존 코드를 역분석하는 데 몇 시간을 씁니다. 그래도 결국 근사치일 뿐이에요.

이 사건들 하나하나는 치명적으로 보이지 않아요. 그런데 3인 팀에서 1년 동안 쌓이면, 토큰 파일 하나와 문서화된 컴포넌트 몇 개로 없앨 수 있었던 문제에 엔지니어링과 디자인 시간의 상당 부분을 쓰는 거예요.

## 최소한의 디자인 시스템에 드는 비용

디자인 시스템이라면 Figma 라이브러리, Storybook, 비주얼 회귀 테스트, 문서 사이트가 필요하다는 오해가 있어요. 작은 팀에게는 처음에 그 중 어느 것도 필요하지 않습니다. 기반은 훨씬 단순해요.

**토큰 정의** (약 3시간): 시맨틱 별칭이 있는 색상 팔레트, 간격 스케일, 타입 크기. 결과물은 다른 모든 것이 임포트하는 TypeScript 파일 하나예요.

**핵심 컴포넌트 5개** (약 5시간): Button, Input, Card, Modal, Select. 새로 만드는 게 아니라 이미 있는 컴포넌트를 토큰 파일을 쓰도록 리팩터링하는 거예요.

**한 페이지짜리 README** (약 2시간): 토큰 사용법, 컴포넌트 목록, 새 컴포넌트를 추가할 때와 기존 것을 변형할 때의 기준.

총 약 10시간의 집중 작업이에요. FramingUI를 시작점으로 쓰면 이게 6시간 정도로 줄어들어요. 토큰 구조, shadcn/ui 기반 컴포넌트, MCP 통합이 이미 갖춰져 있으니까요.

```bash
pnpm add @framingui/ui @framingui/core @framingui/tokens tailwindcss-animate
```

토큰은 CSS 커스텀 프로퍼티로 제공됩니다. 컴포넌트에서 직접 사용하면 돼요.

```css
/* 하드코딩 대신 토큰을 사용 */
background: var(--foreground-accent);
padding: var(--spacing-4);
```

이게 기반 전부예요. 배포 파이프라인도, 문서 인프라도 필요 없습니다.

## 투자 회수 타임라인

설정 후 2~3주는 손익분기점이에요. 10시간을 투자했고, 색상 논쟁과 코드 리뷰 시간 감소로 회수하는 중입니다.

2개월차부터는 복리 효과가 보이기 시작해요. 디자인 결정이 토큰 수준에서 이미 내려져 있으니 새 기능 구현 시간이 줄어들고, 브랜드 색상이 바뀌면 값 하나 수정으로 모든 컴포넌트가 반영됩니다.

AI 개발 케이스는 따로 언급할 가치가 있어요. 디자인 토큰 없이 Claude Code나 Cursor를 쓰면, AI는 매번 그럴듯하지만 틀린 값을 생성합니다. AI가 읽을 수 있는 토큰 파일과 MCP 통합이 있으면, AI가 실제 디자인 시스템을 참조하는 코드를 만들어냅니다. FramingUI MCP 서버는 한 번 설정하면 끝이에요.

```bash
npx -y @framingui/mcp-server@latest init
```

이후 AI가 생성한 컴포넌트는 개발자가 직접 작성할 때와 동일한 토큰을 참조합니다. 재생성하고, 색상을 확인하고, 수정하는 루프가 눈에 띄게 짧아져요.

## 만들지 말아야 할 것

10시간 버전이 대부분의 소규모 팀에게 적절한 범위예요. 지금 당장 투자할 가치가 없는 것들이 있어요.

Storybook은 규모가 커지면 가치 있지만, 8~10명이 되기 전에는 유지보수 부담만 더해집니다. 비주얼 회귀 테스트는 아직 제품-시장 적합성을 찾는 단계라면 시기상조예요. 5인 팀에서 문서 사이트는 저장소의 마크다운 파일로 충분한 역할을 합니다. Figma 자동 동기화는 개발자들이 토큰 파일을 직접 읽는다면 필수가 아니에요.

팀이 성장하면 순서대로 추가하세요. 일찍 만든다고 디자인 시스템이 더 효과적이 되는 게 아니라, 유지비만 더 비싸집니다.

## 결국 어떤 결정인가

디자인 시스템은 디자인 인프라가 아니에요. 타입 안전한 API 클라이언트나 공유 에러 처리 모듈과 같은 범주의 엔지니어링 인프라입니다. 10개 파일을 건드리는 기능을 테스트 없이 출시하지 않는 것처럼, 디자인 시스템은 UI 일관성을 위한 동등한 안전망이에요.

6개월 후에도 살아있을 제품을 만드는 팀이라면, 투자 시간은 스프린트 절반 정도예요. 회수는 다음 스프린트부터 시작되고 멈추지 않습니다.

실용적인 시작점을 제안하면: 코드베이스를 열고, 주요 색상으로 쓰인 서로 다른 hex 값의 수를 세어보고, 버튼 변형의 수를 세어보세요. 둘 중 하나라도 2를 넘는다면, 디자인 시스템 문제가 이미 존재하는 겁니다. 토큰 파일이 그 문제를 해결하는 첫 번째 단계예요.

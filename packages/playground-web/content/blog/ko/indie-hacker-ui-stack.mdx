Material-UI를 쓰는 솔로 개발자는 처음부터 만드는 것보다 빨리 출시해요. 그런데 몇 달이 지나면 또 다른 문제가 생기죠. 랜딩 페이지가 "부트스트랩 템플릿"처럼 보인다는 거예요. 방문자 입장에서는 이미 어디선가 본 UI거든요.

아래 레이어드 스택은 두 문제를 함께 해결해요. Tailwind CSS가 유틸리티를 담당하고, FramingUI가 디자인 토큰 레이어를 추가하고, shadcn/ui가 직접 소유하는 컴포넌트를 제공해요. 설정은 몇 시간이면 충분하고, 그 이후엔 스스로 굴러가요.

## 세 도구가 함께여야 하는 이유

Tailwind는 1,000가지 스타일링 옵션을 줘요. 그 자유로움이 일관성을 무너뜨리죠. FramingUI는 그 선택지를 브랜드가 실제로 사용하는 집합으로 좁혀줘요 — 특정 neutral 스케일, 두 가지 브랜드 색상, 하나의 간격 리듬 — 그리고 그 결정을 CSS 변수로 고정해요. shadcn/ui는 Radix UI 프리미티브를 기반으로 접근성이 갖춰진 컴포넌트를 코드베이스에 직접 복사해서 쓸 수 있게 해줘요.

shadcn의 핵심 특성은 소스를 직접 소유한다는 거예요. 싸워야 할 npm 패키지가 없어요. `Button`의 모양을 바꾸고 싶으면 `components/ui/button.tsx`를 직접 수정하면 돼요. 테마 오버라이드 시스템도, `!important` 꼼수도 필요 없어요.

이 조합이 만드는 또 다른 이점은 AI 도구예요. Cursor나 Claude가 코드를 생성할 때 임의의 팔레트 값 대신 토큰 이름 — `--foreground-accent`, `--background-page` — 이 컨텍스트에 있으니까, 기본값부터 브랜드에 맞는 결과가 나와요.

## Layer 1: Tailwind CSS

유틸리티 클래스는 네이밍 결정을 없애줘요. `.card-header-title`로 할지 `.content-heading`으로 할지 고민할 필요가 없어요. 번들은 실제로 사용한 클래스만 포함해요.

```bash
npm install -D tailwindcss postcss autoprefixer
npx tailwindcss init -p
```

모든 레이어의 기반이 돼요.

## Layer 2: FramingUI

FramingUI는 Tailwind와 컴포넌트 사이에 위치해요. 디자인 결정을 모든 컴포넌트가 사용할 수 있는 CSS 변수로 변환해줘요.

pnpm으로 설치해요.

```bash
pnpm add @framingui/ui @framingui/core @framingui/tokens tailwindcss-animate
```

`init` 명령어는 전체 런타임 계약을 자동으로 연결해줘요 — 프로바이더 bootstrap, 전역 스타일시트 임포트, AI 도구를 위한 MCP 연결까지요.

```bash
npx -y @framingui/mcp-server@latest init
```

토큰은 `var(--foreground-accent)`이나 `var(--background-page)` 같은 CSS 변수가 돼요. 브랜드 색상을 바꿀 때 컴포넌트 파일을 뒤지는 게 아니라 토큰 정의 하나만 수정하면 돼요.

## Layer 3: shadcn/ui

```bash
npx shadcn@latest init
```

스타일 선호도와 기본 색상을 물어봐요. FramingUI 토큰 레이어와 최대 호환을 위해 CSS variables를 선택하세요.

필요한 컴포넌트를 하나씩 추가해요.

```bash
npx shadcn@latest add button input label card dialog form
```

각 컴포넌트는 코드베이스의 `components/ui/`에 추가돼요. 완전히 소유하는 코드예요. Radix UI 프리미티브 기반이라 키보드 탐색과 스크린 리더 지원이 기본으로 들어가 있어요.

## Layer 4: 제품 전용 컴포넌트

범용 컴포넌트가 80%를 커버해요. 나머지 20%는 제품을 구별되게 만드는 UI예요 — 가격 카드, 분석 대시보드, 도메인 특화 데이터 뷰.

이것들은 `components/features/`에 위치하고 아래 레이어를 조합해요.

```tsx
import { Button } from '@/components/ui/button';

export function PricingCard({ plan, price, features }: PricingCardProps) {
  return (
    <div className="bg-[var(--background-page)] border border-[var(--border-default)] rounded-lg p-[var(--spacing-6)]">
      <h3 className="text-2xl font-bold text-[var(--foreground-primary)]">{plan}</h3>
      <p className="text-4xl font-bold text-[var(--foreground-accent)] my-[var(--spacing-4)]">
        ${price}/mo
      </p>
      <ul className="space-y-[var(--spacing-2)] mb-[var(--spacing-6)]">
        {features.map((f) => (
          <li key={f} className="text-[var(--foreground-secondary)]">
            {f}
          </li>
        ))}
      </ul>
      <Button className="w-full">시작하기</Button>
    </div>
  );
}
```

모든 값이 토큰을 참조해요. 리브랜딩은 컴포넌트가 아니라 토큰을 바꾸는 일이 돼요.

## 이 스택을 망가뜨리는 세 가지

**shadcn 컴포넌트를 바로 커스터마이징하는 것.** 설치하고 한 달은 그대로 써봐요. 명확하고 반복되는 필요가 생길 때만 커스터마이징하세요. 너무 일찍 수정하면 업스트림 버그 수정을 잃어요.

**토큰 정의를 건너뛰는 것.** 첫 컴포넌트를 만들기 전에 브랜드 색상, neutral 스케일, 간격 리듬을 결정하세요. 초반 토큰 없는 일관성은 의지력으로 유지할 수 없어요.

**출시 전에 과잉 설계하는 것.** Storybook, 컴포넌트 테스트, 모노레포 구조는 규모가 커지면 유용해요. 출시가 급할 때는 몇 주를 잡아먹어요. 제품 먼저 만들고, 구체적인 고통을 느낄 때 도구를 추가하세요.

## 추가 작업 없는 접근성

shadcn 컴포넌트는 기본으로 접근 가능해요. Dialog가 포커스 트래핑과 Escape 키 처리를 담당해요. 폼 컴포넌트가 레이블 연결과 에러 알림을 자동으로 연결해줘요. Checkbox와 Switch는 키보드에 올바르게 반응해요.

직접 구현하지 않아도 완전한 키보드 탐색이 돼요. 주로 추가해야 할 것은 아이콘만 있는 버튼에 `aria-label` 정도예요.

전체 스택 비용은 $0이고, 몇 주가 아닌 며칠 안에 출시할 수 있어요. 그게 이 스택을 쓰는 이유예요.

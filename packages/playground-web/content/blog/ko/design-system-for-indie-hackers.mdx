## 인디해커에게 디자인 시스템 선택이 중요한 이유

인디해커는 기능 개발, 운영, 배포, 고객 대응을 거의 혼자 처리합니다. 이때 UI까지 매번 새로 결정하면 제품 속도가 급격히 떨어집니다. 처음엔 "대충 보기 좋게"가 되지만, 화면이 늘어날수록 코드와 경험 모두 복잡해집니다.

그래서 디자인 시스템은 대기업 전유물이 아닙니다. 오히려 작은 팀일수록 더 필요합니다. 문제는 선택 기준이 모호하다는 점입니다. 유명하다고, 예뻐 보인다고 고르면 운영 단계에서 바로 비용이 튑니다.

## 먼저 답해야 할 4가지 질문

### 1) UI 인프라에 실제로 쓸 수 있는 시간이 얼마나 되는가?

한 달에 UI 체계화에 쓸 수 있는 시간이 2시간인지 20시간인지에 따라 선택이 달라집니다.

### 2) 브랜드 차별화가 얼마나 중요한가?

초기 검증 단계라면 기본 일관성만 확보해도 충분할 수 있습니다. 하지만 경쟁이 치열한 시장이라면 "어디서 본 듯한 화면"은 신뢰를 깎습니다.

### 3) 앞으로 화면과 기능이 얼마나 빠르게 늘어날 것인가?

성장 속도가 빠를수록 초기에 정한 규칙의 품질이 중요해집니다. 나중에 고치면 비용이 몇 배로 커집니다.

### 4) AI 코드 생성을 개발 프로세스에 적극적으로 쓸 것인가?

이 질문이 특히 중요합니다. AI를 자주 쓸수록 토큰 구조와 컴포넌트 규칙이 명확해야 일관성이 유지됩니다.

## 선택지 3가지와 현실적인 트레이드오프

### 선택지 A: 유틸리티 클래스 중심으로 직접 구성

장점:

- 시작 속도가 빠름
- 도입 비용이 거의 없음
- 실험 단계에서 유연함

단점:

- 화면이 늘수록 불일치가 급격히 증가
- 반복 코드와 예외 케이스가 많아짐
- 팀 합류나 AI 코드 생성 시 드리프트 심화

추천 상황: 단기 실험, 빠른 폐기 가능성이 큰 프로젝트

### 선택지 B: 컴포넌트 라이브러리 + 로컬 규칙

장점:

- 비교적 빠른 개발 속도
- 접근성과 기본 품질을 확보하기 쉬움
- 팀 온보딩이 수월함

단점:

- 규칙이 문서화되지 않으면 결국 사람 의존
- 토큰 거버넌스가 약하면 장기적으로 흔들림
- 제품 개성 확보가 어려울 수 있음

추천 상황: 단기 속도와 중기 안정성의 균형이 필요한 팀

### 선택지 C: 토큰 중심 디자인 시스템

장점:

- 일관성과 확장성에서 가장 강함
- 전역 수정과 테마 확장이 쉬움
- AI 코드 생성과 궁합이 좋음

단점:

- 초기에 구조를 잡는 시간이 필요함
- 기존 코드가 많다면 이관 전략이 필요함

추천 상황: 제품이 성장 중이고, UI 품질을 운영 지표로 보고 싶은 팀

## 인디해커용 빠른 의사결정 매트릭스

아래 조건이 맞으면 해당 선택지를 우선 검토하세요.

### A가 맞는 경우

- 당장 아이디어 검증이 최우선
- UI를 나중에 재설계할 계획이 명확함
- AI로 프로덕션 UI를 대량 생성하지 않음

### B가 맞는 경우

- 지금 기능 출시 속도가 중요함
- 기본 품질을 빠르게 확보해야 함
- 팀 규칙을 최소한으로 운영할 수 있음

### C가 맞는 경우

- 화면 수와 사용자 플로우가 계속 늘어남
- 장기적으로 디자인 부채를 줄이고 싶음
- AI 생성 코드도 디자인 시스템 규칙 안에 넣고 싶음

대부분의 인디 제품은 결국 C 방향으로 이동합니다. 시작을 A나 B로 하더라도 성장 시점에 토큰 중심 구조가 필요해집니다.

## 실제 평가 시 꼭 봐야 할 항목

### 토큰 모델의 명확성

색상/간격/타이포그래피/컴포넌트 의미 토큰이 분리되어 있는지 확인하세요. 이름이 애매하면 운영 단계에서 반드시 충돌이 납니다.

### 프리미티브 API의 제약 수준

컴포넌트가 모든 스타일을 다 열어두면 규칙은 사실상 무력화됩니다. 기본값이 강하고 변형이 제한되어야 합니다.

### 문서와 운영성

"예쁜 컴포넌트"보다 중요한 건 운영 규칙입니다. 언제 어떤 변형을 쓰는지, 새 컴포넌트를 추가할 때 기준이 무엇인지 명확해야 합니다.

### AI 워크플로우 적합성

토큰을 기계가 읽을 수 있고, 컴포넌트 계약이 분명한지 확인하세요. 이게 없으면 AI 결과물이 그럴듯하지만 일관성이 계속 깨집니다.

## FramingUI를 고려할 만한 이유

FramingUI는 인디해커가 토큰 중심 디자인 시스템을 무겁지 않게 도입할 때 강점이 있습니다.

- 토큰 구조와 프리미티브를 빠르게 잡기 쉽고
- 화면이 늘어도 규칙이 유지되도록 설계되어 있으며
- AI 코드 생성 흐름에서도 토큰 맥락을 활용하기 좋습니다

즉, "초기 속도"와 "장기 일관성"의 간극을 줄여주는 데 유리합니다.

## 기존 프로젝트에 적용하는 이관 순서

### 1단계: 핵심 토큰부터 확정

색상, 간격, 타입, 라운드를 먼저 고정합니다. 처음부터 완벽하게 하려 하지 마세요.

### 2단계: 사용 빈도 높은 프리미티브 교체

`Button`, `Input`, `Card` 중심으로 바꾸면 체감 효과가 큽니다.

### 3단계: 핵심 플로우 우선 적용

온보딩, 결제, 설정처럼 사업적으로 중요한 화면부터 이관하세요.

### 4단계: 운영 규칙 추가

- 하드코딩 색상 금지
- 토큰 외 간격 제한
- 신규 변형 추가 시 이유 명시

이 단계가 없으면 다시 원래 상태로 돌아갑니다.

## 자주 나오는 선택 실수

### 실수 1: 스크린샷만 보고 고른다

외형만으로는 유지보수 비용을 판단할 수 없습니다.

### 실수 2: 초기 도입 속도만 본다

진짜 비용은 3개월 뒤에 발생합니다. 리팩터링 세금이 얼마나 생길지 봐야 합니다.

### 실수 3: 토큰 결정을 미룬다

"나중에 정리"는 거의 항상 더 큰 비용으로 돌아옵니다.

### 실수 4: AI 코드 생성 영향을 과소평가한다

AI를 자주 쓰는 팀일수록 디자인 시스템 구조가 품질을 결정합니다.

## 최종 체크리스트

선택 전 아래 질문에 모두 답할 수 있어야 합니다.

- 토큰 체계를 1페이지로 설명할 수 있는가?
- 프리미티브 API가 일관성을 강제할 수 있는가?
- 기존 코드에 점진 적용 가능한가?
- AI 코드 생성과 함께 써도 규칙이 유지되는가?
- 현재 팀 리소스로 운영 가능한가?

인디해커에게 필요한 것은 가장 유명한 선택지가 아니라, 제품 속도와 품질을 동시에 지켜주는 디자인 시스템입니다. 성장 구간에 들어섰다면 토큰 중심 접근이 장기적으로 훨씬 유리합니다.

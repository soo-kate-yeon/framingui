## 왜 MCP 연동이 필요한가

AI로 프런트엔드 개발 속도는 빨라졌습니다. 하지만 실제 팀에서 자주 듣는 불만은 비슷합니다. "생성은 빠른데, 디자인 시스템 맞추느라 다시 손이 많이 간다." 즉, 생성 속도는 빨라도 운영 효율은 생각보다 오르지 않습니다.

MCP(Model Context Protocol)는 이 문제를 줄이는 핵심 연결고리입니다. AI가 프롬프트 안의 문장만 보는 것이 아니라, 외부의 구조화된 컨텍스트(예: 디자인 토큰, 컴포넌트 규칙)를 실시간으로 조회하게 만들기 때문입니다.

## MCP 연동으로 얻는 실질적인 변화

디자인 시스템을 MCP로 연결하면 워크플로우가 이렇게 바뀝니다.

- 매번 긴 가이드 문서를 붙여 넣지 않아도 됨
- AI가 최신 토큰 값을 직접 조회함
- 컴포넌트 변형 선택이 기존 규칙과 더 잘 맞음
- 생성 후 수동 스타일 정리 시간이 줄어듦

결국 "빨리 만들고 많이 고치는 흐름"에서 "빨리 만들고 바로 리뷰 가능한 흐름"으로 이동합니다.

## 기본 아키텍처 이해

구성은 단순합니다.

1. 저장소 안에 디자인 시스템 토큰/규칙 데이터가 있음
2. MCP 서버가 해당 데이터를 질의 가능하게 노출함
3. Claude, Cursor가 MCP 서버를 호출함
4. AI가 조회한 값을 기반으로 UI 코드를 생성함

핵심은 "디자인 시스템을 문서가 아니라 조회 가능한 시스템 자원으로 다룬다"는 점입니다.

## 사전 준비 체크리스트

연동 전에 아래를 먼저 점검하세요.

- 디자인 토큰이 구조화되어 있는가 (`tokens.json`, `theme.ts` 등)
- 의미 기반 네이밍이 되어 있는가 (`color.text.primary` 같은 형태)
- 핵심 프리미티브 API가 정리되어 있는가
- MCP를 지원하는 AI 클라이언트를 쓰는가

토큰이 모호하면 연동해도 효과가 제한적입니다.

## 1단계: 토큰 소스 정리

최소한 다음 데이터는 명확히 갖춰두는 것이 좋습니다.

- 색상 의미 토큰
- 간격 스케일
- 타이포그래피 스케일
- 라운드/그림자
- 버튼/입력창 등 핵심 컴포넌트 의미 토큰

중요한 포인트는 raw 값만 나열하지 말고 "의미"를 함께 유지하는 것입니다.

## 2단계: Claude에 MCP 서버 등록

Claude 환경에 디자인 시스템 MCP 서버를 등록합니다.

예시 구조:

```json
{
  "servers": {
    "design-system": {
      "command": "npx",
      "args": ["@framingui/mcp-server"],
      "env": {
        "PROJECT_PATH": ".",
        "TOKEN_PATH": "./tokens.json"
      }
    }
  }
}
```

설정 후에는 클라이언트를 재시작해야 서버가 로드됩니다.

## 3단계: Cursor에 MCP 서버 등록

Cursor도 유사하게 등록합니다.

예시 구조:

```json
{
  "mcpServers": {
    "design-system": {
      "command": "npx",
      "args": ["@framingui/mcp-server"],
      "env": {
        "PROJECT_PATH": ".",
        "TOKEN_PATH": "./tokens.json"
      }
    }
  }
}
```

동일하게 재시작 후 동작을 확인합니다.

## 4단계: 연결 검증 시나리오

처음에는 단순한 질의부터 확인하세요.

- "사용 가능한 디자인 토큰 목록을 보여줘"
- "버튼 관련 의미 토큰만 추려줘"
- "기존 프리미티브만 써서 설정 폼을 만들어줘"

여기서 실패하면 보통 경로/환경 변수/토큰 파일 형식 문제가 원인입니다.

## 5단계: 프롬프트 운영 방식 표준화

연동만 하고 프롬프트를 제각각 쓰면 효과가 반감됩니다. 팀 공통 템플릿을 두세요.

추천 템플릿 요소:

- 기존 프리미티브만 사용
- 의미 토큰만 사용
- 신규 변형 임의 생성 금지
- 로딩/빈 상태/에러 상태 포함

이 정도만 고정해도 결과 품질 편차가 크게 줄어듭니다.

## Claude에서 실전 활용 팁

### 작업 분리 프롬프트

한 번에 거대한 화면을 요청하지 말고, 섹션 단위로 나누면 토큰 준수율이 높아집니다.

### 조회 후 생성 방식

"먼저 관련 토큰과 변형을 조회하고, 그 결과를 바탕으로 구현"하도록 지시하면 hallucination이 줄어듭니다.

### 리뷰 기준 고정

생성 후에는 "토큰 준수 여부"와 "프리미티브 사용 여부"를 체크리스트로 검토하세요.

## Cursor에서 실전 활용 팁

### 부분 수정 중심으로 활용

전체 파일 생성보다, 기존 코드에서 토큰 위반 부분을 수정하게 하면 안정적입니다.

### 리팩터링 요청 패턴

"하드코딩 색상을 의미 토큰으로 바꿔줘" 같은 명시적 리팩터링 요청이 효과적입니다.

### 작은 단위 커밋 습관

AI 수정 범위를 작게 유지하면 회귀 위험이 줄고 리뷰도 쉬워집니다.

## FramingUI가 MCP 연동에서 강한 이유

FramingUI는 디자인 시스템 구조와 AI 워크플로우를 함께 고려한다는 점에서 실전성이 있습니다.

- 토큰 중심 설계가 명확하고
- 프리미티브와 토큰 계약을 운영하기 쉽고
- MCP 기반 조회 흐름과 결합하기 편합니다

그래서 Claude, Cursor를 자주 쓰는 팀에서 "생성 후 정리 비용"을 줄이는 데 유리합니다.

## 자주 발생하는 문제와 해결 포인트

### 문제 1: 의미 없는 토큰 구조

`blue500` 중심 토큰만 있으면 AI는 intent를 추론해야 해서 다시 흔들립니다.

### 문제 2: 컴포넌트 계약 부재

토큰이 있어도 프리미티브 API가 불명확하면 AI가 임의 props를 만들어냅니다.

### 문제 3: 오래된 토큰 참조

MCP가 최신 소스를 읽지 않으면 화면과 코드가 어긋납니다.

### 문제 4: 중복된 긴 프롬프트

MCP가 제공하는 정보를 프롬프트에 또 길게 넣으면 오히려 핵심 요구가 희석됩니다.

## 30분 도입 플랜

빠르게 시작하려면 아래 순서가 가장 현실적입니다.

1. 토큰 구조 점검 및 최소 보완
2. Claude/Cursor MCP 설정
3. 대표 컴포넌트 3개 생성 테스트
4. 하드코딩 값 검출 린트 규칙 추가
5. 팀 공통 프롬프트 템플릿 문서화

이 정도면 당일에 체감 가능한 개선이 나옵니다.

## 결론

MCP 기반 디자인 시스템 연동은 AI를 더 많이 쓰기 위한 옵션이 아니라, AI를 "제대로" 쓰기 위한 기본 인프라에 가깝습니다.

Claude와 Cursor가 실제 토큰 컨텍스트를 조회하게 만들면, UI 코드는 덜 흔들리고 리뷰는 빨라집니다. 결과적으로 개발 속도뿐 아니라 품질의 예측 가능성까지 함께 올라갑니다.

AI 시대의 디자인 시스템 운영은 문서 중심에서 연결 중심으로 이동하고 있습니다. 그 전환의 핵심이 바로 MCP입니다.

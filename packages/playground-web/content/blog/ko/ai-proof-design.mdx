## 깨지는 UI 문제: AI 코드가 디자인을 망치는 이유

AI 코드 생성은 속도를 약속합니다. 기능을 설명하면 몇 초 만에 작동하는 컴포넌트 코드를 받습니다. 배포하고, 축하하고, 다음 작업으로 넘어갑니다.

그러다 불일치를 발견합니다:

- 패딩이 미묘하게 다른 버튼들
- "비슷해 보이지만" 팔레트와 맞지 않는 색상
- 8px 그리드에서 벗어난 간격
- 왜인지 모르겠지만 이상한 그림자와 테두리

각 생성 결과는 독립적으로는 작동하지만, 함께 모이면 시각적 혼돈을 만듭니다. 디자인 시스템은 제약이 아니라 제안이 됩니다. 제품은 정교하게 만든 것이 아니라 임시로 짜깁기한 것처럼 느껴집니다.

이것이 깨지는 UI 문제입니다. AI는 빠르게 코드를 생성할 수 있지만, 적절한 구조 없이는 그 코드가 매번 반복할 때마다 시각적 기반을 무너뜨립니다.

## AI 생성 UI 코드가 자주 깨지는 이유

근본 원인은 AI의 무능함이 아닙니다. 컨텍스트 불일치입니다.

### AI는 "그럴듯함"을 최적화한다, 당신의 시스템을 아니다

LLM은 수백만 개의 공개 코드베이스로 학습됩니다. 버튼을 요청하면 모델은 웹 전체에서 버튼이 일반적으로 어떻게 생겼는지를 생성합니다—당신의 버튼이 어떻게 생겨야 하는지가 아닙니다.

명시적인 제약 없이 AI는 일반적인 패턴으로 기본 설정됩니다:

- `padding: 12px 24px` (일반적이지만, 당신은 `spacing.3`과 `spacing.5`를 사용할 수도)
- `background: #3b82f6` (파란색 같지만, 당신의 `color.primary.solid`가 아님)
- `border-radius: 8px` (합리적이지만, 시스템은 `6px`인 `radius.md` 사용)

각 결정은 독립적으로는 괜찮아 보입니다. 수십 개 컴포넌트에 걸쳐 결합되면 일관성이 무너집니다.

### 암묵적 지식은 전달되지 않는다

당신 팀은 디자인 원칙을 알고 있습니다. 디자이너는 언제 `elevated` 대신 `default` 표면을 쓸지 압니다. 엔지니어는 폼 입력이 항상 특정 레이블 스타일과 쌍을 이룬다는 것을 압니다.

하지만 이 지식은 Figma 코멘트, Slack 스레드, 구전으로 존재합니다. 명시적이고 구조화된 형태로 만들지 않으면 AI는 접근할 수 없습니다.

### 자연어 프롬프트는 애매하다

"우리 브랜드 색상 써줘"는 맥락에 따라 다른 의미입니다:

- "브랜드"는 primary를 뜻하나, 아니면 전체 의미 팔레트를 뜻하나?
- 텍스트는 `color.text.primary`를 써야 하나, `color.primary.fg`를 써야 하나?
- hover 상태, focus ring, disabled 변형은 어떻게?

토큰으로 인코딩된 공유 어휘 없이는 모든 프롬프트가 전화 놀이가 됩니다.

### 컨텍스트 윈도우는 기능 요구사항과 경쟁한다

디자인 가이드라인을 매 프롬프트에 수동으로 붙여 넣어도, 실제 기능 명세를 위한 공간을 빼앗는 것입니다. 그리고 모델은 여전히 그 가이드라인을 특정 컴포넌트에 어떻게 적용할지 추론해야 합니다.

구조화된 디자인 토큰은 재사용 가능하고 조회 가능한 계약을 제공해 이 문제를 해결합니다.

## "AI-Proof" 디자인이란 무엇인가?

AI-proof 디자인은 AI가 작동하지 못하게 막는 것이 아닙니다. 기본적으로 AI가 올바른 출력을 향하도록 유도하는 시스템을 설계하는 것입니다.

AI-proof 디자인 시스템은 세 가지 속성을 가집니다:

### 1. 명시적 제약

값과 관계가 산문이 아닌 기계 판독 가능한 형식으로 정의됩니다.

- 나쁨: "일관된 간격 사용"
- 좋음: `spacing.3` → `16px`, `spacing.4` → `24px`, 컴포넌트는 항상 이 키 사용

### 2. 의미적 명확성

토큰 이름이 겉모습이 아니라 의도를 인코딩합니다.

- 나쁨: `color.blue.500`
- 좋음: `color.primary.solid`, `color.primary.fg`, `color.primary.border`

AI가 `color.primary.solid`를 읽으면 의미적 역할을 이해합니다. `#3b82f6`를 보면 추측합니다.

### 3. 조합 가능한 프리미티브

컴포넌트는 AI가 안정적으로 참조할 수 있는 안정적이고 재사용 가능한 단위로 구축됩니다.

- 나쁨: 인라인 CSS가 있는 커스텀 스타일 div
- 좋음: `<Button variant="primary" size="md">` 여기서 variants와 sizes는 열거되고 토큰 기반

컴포넌트가 프리미티브일 때 AI는 스타일을 재발명할 필요가 없습니다. 승인된 빌딩 블록에서 조합합니다.

## AI 가드레일로서의 디자인 토큰

디자인 토큰은 AI-proof 디자인의 기초입니다. 암묵적 디자인 지식을 명시적이고 조회 가능한 데이터로 변환합니다.

### 토큰이 AI를 위해 실제로 하는 일

토큰은 단순한 테마 변수가 아닙니다. AI 워크플로우에서 다음 역할을 합니다:

**계약:** 디자인 의도와 생성된 코드 사이의 계약.

**제약 레이어:** 무한한 추측에서 유효한 조합으로 솔루션 공간을 줄임.

**어휘:** 인간 프롬프트와 AI 출력을 정렬.

AI가 토큰 스키마를 읽으면 무엇이 허용되는지, 무엇이 의미론적인지, 부분들이 어떻게 맞춰지는지 봅니다.

### 최소 실행 가능 토큰 스키마

효과를 보기 위해 거대한 엔터프라이즈 디자인 시스템이 필요하지 않습니다. 필수 레이어로 시작하세요:

**색상 토큰:**

- `color.text.primary`, `color.text.secondary`, `color.text.inverse`
- `color.surface.default`, `color.surface.elevated`, `color.surface.sunken`
- `color.border.default`, `color.border.focus`, `color.border.hover`
- `color.primary.solid`, `color.primary.fg`, `color.primary.border`

**간격 토큰:**

- `spacing.1` → `4px`, `spacing.2` → `8px`, `spacing.3` → `16px`, 등
- 컴포넌트가 항상 참조하는 일관된 스케일

**타이포그래피 토큰:**

- `font.size.body`, `font.size.label`, `font.size.heading`
- `font.weight.regular`, `font.weight.medium`, `font.weight.semibold`
- `lineHeight.body`, `lineHeight.tight`, `lineHeight.loose`

**컴포넌트 의미 토큰:**

- `button.primary.bg`, `button.primary.fg`
- `button.secondary.bg`, `button.secondary.fg`
- `input.bg`, `input.border`, `input.focus`

이 스키마는 일반적인 UI 생성 시나리오의 80%를 커버합니다. 필요에 따라 확장하되, 여기서 시작하세요.

## FramingUI로 AI-Proof UI 만드는 실전 전략

FramingUI는 AI 워크플로우를 염두에 두고 설계되었습니다. 효과적으로 사용하는 방법은 다음과 같습니다.

### 전략 1: 항상 토큰 컨텍스트와 함께 생성

AI 메모리나 과거 예제에 의존하지 마세요. 모든 생성마다 명시적으로 토큰 컨텍스트를 제공하세요.

**나쁜 워크플로우:**

```
프롬프트: "이메일과 비밀번호 필드가 있는 로그인 폼 만들어줘"
```

결과: AI가 색상, 간격, focus 상태를 추측함.

**좋은 워크플로우:**

```
프롬프트: "다음을 사용해서 로그인 폼 만들어줘:
- Input: input.bg, input.border, input.focus
- Spacing: 간격은 spacing.3, 패딩은 spacing.5
- Button: button.primary variant
토큰은 [토큰 파일 또는 MCP 엔드포인트]에서 확인"
```

결과: AI가 토큰을 조회하고 규정을 준수하는 코드 생성.

MCP 통합을 사용하면 토큰 컨텍스트가 자동으로 제공됩니다. 그렇지 않으면 프롬프트 템플릿에 간단한 토큰 참조를 포함하세요.

### 전략 2: 스타일 div 대신 프리미티브 컴포넌트 사용

AI는 수백만 개의 예제를 봤기 때문에 커스텀 스타일 요소 생성을 좋아합니다. 이를 저항하세요.

**깨지기 쉬운 패턴:**

```tsx
<div style={{
  padding: '12px 24px',
  background: '#3b82f6',
  borderRadius: '8px',
  color: 'white'
}}>
  제출
</div>
```

모든 생성이 버튼을 재발명합니다. 값이 표류합니다. 유지보수는 지옥입니다.

**AI-proof 패턴:**

```tsx
<Button variant="primary" size="md">
  제출
</Button>
```

AI는 새 변형을 발명할 수 없습니다. 승인된 프리미티브에서 조합합니다. 일관성은 컴포넌트 API 수준에서 강제됩니다.

**이를 강제하는 방법:**

- 프롬프트에서: "`@framingui/ui`의 컴포넌트만 사용. 커스텀 스타일 요소 만들지 마."
- 린팅에서: 인라인 스타일 또는 승인되지 않은 유틸리티 조합 감지.
- 리뷰에서: 프리미티브를 우회하는 PR 거부.

### 전략 3: 의미 컴포넌트 토큰 정의

원시 디자인 토큰만으로는 충분하지 않습니다. 디자인 의도를 구현에 매핑하는 컴포넌트 수준 의미 토큰도 필요합니다.

버튼 예시:

```ts
const buttonTokens = {
  primary: {
    bg: 'color.primary.solid',
    fg: 'color.primary.fg',
    border: 'color.primary.border',
    hoverBg: 'color.primary.hover',
    activeBg: 'color.primary.active',
  },
  secondary: {
    bg: 'color.surface.elevated',
    fg: 'color.text.primary',
    border: 'color.border.default',
    hoverBg: 'color.surface.hover',
    activeBg: 'color.surface.active',
  }
}
```

AI가 버튼을 생성할 때 각 상태와 변형에 어떤 토큰을 사용할지 정확히 압니다. 추측 없음.

### 전략 4: 토큰 검증 자동화

토큰 컨텍스트가 있어도 AI는 실수할 수 있습니다. 자동 검사로 조기에 잡으세요.

**린트 규칙: 하드코딩된 값 감지**

토큰 참조 대신 원시 hex 색상, 픽셀 값 또는 매직 넘버를 사용하는 컴포넌트에 플래그.

**타입 체킹: 토큰 사용 강제**

TypeScript를 사용한다면 스타일 프리미티브가 임의 문자열이 아닌 토큰 키를 요구하도록 만드세요.

```ts
// 나쁨: 모든 문자열 허용
style={{ color: 'blue' }}

// 좋음: 유효한 토큰 키만 허용
style={{ color: tokens.color.text.primary }}
```

**시각 회귀 테스팅:**

AI가 새 컴포넌트를 생성하면 스냅샷 테스트가 예상치 못한 시각적 표류를 잡습니다.

### 전략 5: 토큰 계약이 포함된 프롬프트 라이브러리 유지

매번 프롬프트를 재발명하지 마세요. 기본적으로 토큰 컨텍스트를 포함하는 재사용 가능한 템플릿 라이브러리를 구축하세요.

**예시 템플릿: 검증이 있는 폼 입력**

```
다음으로 폼 입력 만들어줘:
- 토큰 컨텍스트: [토큰 파일 또는 MCP 참조]
- @framingui/ui의 Input 프리미티브 사용
- 상태: default, focus, error, disabled
- Error 상태는 테두리와 텍스트에 color.intent.danger 사용
- Focus 상태는 input.focus 토큰 사용
- 항상 Label 컴포넌트와 쌍으로
```

무엇이 작동하는지 배우면서 이 템플릿을 반복하세요. 시간이 지나면서 프롬프트 라이브러리는 AI 상호작용을 위한 디자인 시스템이 됩니다.

## 여전히 깨질 때: AI 생성 UI 디버깅

AI-proof 아키텍처가 있어도 예외 상황은 만나게 됩니다. 진단 방법은 다음과 같습니다.

### 증상: AI가 존재하지 않는 토큰 이름을 발명함

**원인:** 토큰 스키마가 명시적으로 제공되지 않았거나 불완전함.

**수정:** 모든 생성에 토큰 컨텍스트가 포함되었는지 확인. MCP 사용 시 연결 확인. 파일 사용 시 스키마가 최신인지 확인.

### 증상: AI가 올바른 토큰을 사용하지만 잘못된 의미 매핑

**원인:** 토큰 이름이 너무 애매함 (예: `color.primary.solid` 대신 `color.blue.500`).

**수정:** 토큰을 더 의미론적으로 리팩터링. 의도를 명확히 하도록 프롬프트 업데이트 ("제출 액션에는 primary button 토큰 사용").

### 증상: AI가 유효한 프리미티브를 생성하지만 잘못된 조합

**원인:** 컴포넌트 API는 올바르지만 AI가 어떤 변형을 언제 사용할지 이해하지 못함.

**수정:** 프롬프트에 조합 가이드라인 추가. 예: "주요 액션에는 primary 버튼, 취소/뒤로에는 secondary 사용."

### 증상: 생성된 코드는 작동하지만 시간이 지나면서 시각적으로 표류함

**원인:** 자동 검증 부족.

**수정:** 린트 규칙, 스냅샷 테스트, 주기적 디자인 QA 패스 추가.

## 장기 보상: 취약에서 탄력으로

AI-proof 디자인은 투자입니다. 초기 설정은 시간이 걸립니다: 토큰 정의, 프리미티브 구축, 프롬프트 템플릿 생성.

하지만 보상은 복리입니다:

**더 빠른 반복:** 디자이너가 토큰을 한 번 업데이트하면 모든 AI 생성 컴포넌트에 변경 사항이 전파됩니다.

**더 나은 일관성:** 생성된 모든 컴포넌트가 수동 리뷰가 아닌 기본적으로 시스템 규칙을 따릅니다.

**더 낮은 리뷰 오버헤드:** 엔지니어가 토큰 위반 수정 대신 로직과 UX에 더 많은 시간을 씁니다.

**확장 가능한 AI 사용:** 팀이 AI로 더 많은 코드를 생성할수록 품질이 저하되지 않습니다. 시스템이 정확성을 강제합니다.

이것은 AI를 위험한 프로토타입 도구에서 신뢰할 수 있는 프로덕션 워크플로우로 취급하는 전환입니다.

## FramingUI로 시작하기

FramingUI는 토큰 우선 아키텍처와 AI 친화적 도구를 결합하여 AI-proof 디자인을 실용적으로 만듭니다.

**1단계: 최소 실행 가능 토큰 스키마 정의.**

색상, 간격, 타이포그래피, 핵심 컴포넌트 토큰으로 시작. 과도하게 설계하지 마세요.

**2단계: 해당 토큰을 소비하는 프리미티브 구축.**

FramingUI 컴포넌트를 사용하거나 토큰 사용을 강제하는 자체 래퍼를 생성.

**3단계: AI 도구용 토큰 컨텍스트 설정.**

팀이 Claude, Cursor 또는 다른 AI 코딩 어시스턴트를 사용한다면 FramingUI의 MCP 서버를 통합해 토큰 컨텍스트를 자동으로 제공. 그렇지 않으면 재사용 가능한 토큰 참조 파일 생성.

**4단계: 토큰 계약이 포함된 프롬프트 템플릿 생성.**

팀이 AI에 프롬프트하는 방법을 표준화. 토큰 컨텍스트, 프리미티브 사용 규칙, 조합 가이드라인 포함.

**5단계: 가드레일 추가: 린팅, 타입 체킹, 스냅샷 테스트.**

자동화는 AI와 인간 모두가 만드는 실수를 잡습니다.

**6단계: 실제 사용을 기반으로 반복.**

AI가 특정 패턴에 대해 일관되게 잘못된 코드를 생성하면 격차를 해결하도록 토큰이나 프롬프트를 개선.

## 마무리 생각

AI 코드 생성은 본질적으로 취약하지 않습니다. AI를 명확한 제약이 필요한 도구가 아닌 마법 블랙 박스로 취급할 때 UI가 깨집니다.

디자인 토큰은 단순한 테마 변수가 아닙니다. 인간 디자인 의도와 기계 코드 생성 사이의 인터페이스입니다.

AI-proof 디자인 시스템을 구축할 때 AI를 제한하는 것이 아닙니다. 매번 일관되고 유지 보수 가능하며 프로덕션 준비가 된 코드를 생성하는 데 필요한 구조를 제공하는 것입니다.

AI가 UI 워크플로우의 신뢰할 수 있는 부분이 되기를 원한다면—기술 부채의 원천이 아니라—토큰으로 시작하고, 프리미티브로 강제하고, 의미 계약으로 AI를 안내하세요.

그것이 AI가 코드를 생성해도 깨지지 않는 UI를 만드는 방법입니다.

## MCP가 뭐고, 디자이너가 왜 알아야 할까?

디자이너라면 이런 경험 있을 겁니다: 몇 시간 동안 컬러 팔레트를 완성하고, 스페이싱 스케일을 정의하고, 타이포그래피 시스템을 문서화했는데—막상 개발자가 구현한 결과물을 보니 색이 미묘하게 다르고, 여백이 일관되지 않고, 폰트도 약간씩 어긋나 있더라고요.

**Model Context Protocol (MCP)**은 이 문제를 해결하는 게임체인저입니다. Anthropic이 만든 오픈 표준으로, Claude 같은 AI 어시스턴트가 여러분의 디자인 시스템을 직접 읽고 사용할 수 있게 해줍니다. AI에게 "디자인 시스템 API"를 제공하는 셈이죠. 덕분에 AI가 정확한 토큰을 사용해서 UI 컴포넌트를 생성합니다—더 이상 추측하지 않고, 브랜드 컬러가 `primary-600`인데 #3498db를 쓰는 일이 없어집니다.

디자이너에게 MCP의 의미:
- **AI가 디자인 언어를 이해합니다** — 색상, 여백, 타이포그래피 토큰을 개발자처럼 읽습니다
- **기본적으로 일관성 보장** — AI가 생성하는 컴포넌트는 실제 디자인 시스템 값을 사용합니다
- **핸드오프 오류 감소** — "목업과 파란색이 다른데요" 같은 일이 사라집니다

## AI 에이전트가 디자인 토큰을 읽는 방법

Claude Code 같은 AI 에이전트가 MCP를 통해 디자인 시스템과 상호작용하는 방식을 단계별로 살펴보겠습니다.

### 기존 워크플로우 (MCP 없이)

1. 디자이너가 Figma에서 `Primary/600` 색상으로 버튼을 만듭니다
2. 개발자가 묻습니다: "Primary/600의 hex 코드가 뭐예요?"
3. 디자이너가 문서를 확인합니다 → `#7C3AED`
4. 개발자가 하드코딩합니다 `background-color: #7C3AED`
5. 디자인 시스템에서 Primary/600을 `#8B5CF6`로 업데이트합니다
6. 컴포넌트가 이제 동기화되지 않습니다

### MCP 사용 시 (FramingUI 방식)

1. 디자이너가 FramingUI 디자인 시스템에 토큰을 정의합니다
2. AI 에이전트가 MCP 서버에 연결합니다
3. 개발자가 Claude에게 묻습니다: "우리 primary 색상으로 버튼 만들어줘"
4. Claude가 MCP 서버에 쿼리합니다: "`primary-600`이 뭐야?"
5. MCP 서버가 응답합니다: `#7C3AED` (현재 값이 무엇이든)
6. Claude가 생성합니다: `className="bg-primary-600"` (또는 CSS 변수)
7. 디자인 시스템 업데이트 → 모든 컴포넌트가 자동으로 동기화됩니다

### AI가 디자인 시스템에서 "볼 수 있는" 것들

AI 에이전트가 FramingUI의 MCP 서버에 연결하면 다음 정보에 접근할 수 있습니다:

- **컬러 토큰** — Primary, secondary, neutral, semantic 색상과 전체 shade 범위
- **스페이싱 스케일** — 일관된 여백 값 (4px, 8px, 16px 등)
- **타이포그래피 시스템** — 폰트 패밀리, 크기, 굵기, 행간
- **보더 반경** — 컴포넌트 전반의 일관된 모서리 둥글기
- **그림자 & 이펙트** — 엘리베이션 레벨과 시각 효과
- **시맨틱 토큰** — Success, warning, error 상태
- **접근성 데이터** — 대비율, WCAG 준수 여부

AI는 단순히 hex 코드만 받는 게 아닙니다—각 토큰의 **의미론적 의미**를 이해합니다. `primary-600`은 인터랙티브 요소용이고, `neutral-100`은 배경용이며, `spacing-4`는 편안한 패딩용이라는 걸 압니다.

## MCP로 향상되는 디자이너-개발자 협업

### MCP 이전: 핸드오프 문제

**디자이너 관점:**
- 몇 시간 동안 상세한 디자인 시스템을 만듭니다
- 모든 토큰, 변형, 사용 가이드라인을 문서화합니다
- 개발자에게 디자인을 넘깁니다
- 구현을 기다립니다
- 리뷰하고 불일치를 발견합니다
- 레드라인과 함께 왔다갔다 합니다

**개발자 관점:**
- 디자인 파일과 문서를 받습니다
- Figma, Notion, Storybook에서 토큰 값을 찾습니다
- 디자인 토큰을 수동으로 코드로 변환합니다
- 문서가 최신인지 확인합니다
- 불일치에 대한 피드백을 받습니다
- 수정하고 반복합니다

### MCP 이후: 실시간 협업

**디자이너 워크플로우:**
1. FramingUI에서 디자인 토큰을 정의합니다
2. AI 에이전트가 즉시 접근할 수 있습니다
3. 한 곳에서 토큰을 업데이트합니다
4. AI가 생성한 모든 컴포넌트가 자동으로 업데이트됩니다

**개발자 워크플로우:**
1. Claude에게 묻습니다: "우리 디자인 시스템으로 카드 컴포넌트 만들어줘"
2. Claude가 MCP 서버에서 토큰을 쿼리합니다
3. 정확한 값으로 컴포넌트를 생성합니다
4. 수동 검색 제로, 하드코딩 제로

**핵심:** 디자이너가 소스 오브 트루스(디자인 토큰)를 제어하고, AI가 모든 생성된 컴포넌트가 그 진실을 존중하도록 보장합니다.

## 디자이너 관점의 실용적 예시

### 예시 1: 컬러 일관성

**여러분이 정의하는 것:**
```json
{
  "primary": {
    "50": "#F5F3FF",
    "600": "#7C3AED",
    "900": "#4C1D95"
  }
}
```

**AI가 생성하는 것 (hex 코드를 묻지 않고):**
```jsx
<Button className="bg-primary-600 text-white hover:bg-primary-700">
  Click me
</Button>
```

AI는 컨텍스트를 기반으로 어떤 shade를 사용할지 **압니다**. "use #7C3AED"라고 명시할 필요 없이 "우리 primary 버튼 색상 사용해"라고만 하면 MCP가 알아서 처리합니다.

### 예시 2: 스페이싱 스케일

**여러분이 정의하는 것:**
```json
{
  "spacing": {
    "2": "8px",
    "4": "16px",
    "6": "24px"
  }
}
```

**AI가 이해하는 것:**
- "버튼에 편안한 패딩" → `spacing-4` (16px)
- "리스트 항목에 타이트한 간격" → `spacing-2` (8px)
- "섹션에 여유 있는 간격" → `spacing-6` (24px)

"16px 패딩 추가"라고 하는 대신 "표준 버튼 패딩 사용"이라고 하면 AI가 올바른 토큰을 선택합니다.

### 예시 3: 타이포그래피 위계

**여러분이 정의하는 것:**
```json
{
  "typography": {
    "heading-1": {
      "size": "48px",
      "weight": "700",
      "lineHeight": "1.2"
    },
    "body-md": {
      "size": "16px",
      "weight": "400",
      "lineHeight": "1.5"
    }
  }
}
```

**AI가 생성하는 것:**
```jsx
<h1 className="text-heading-1">제품에 오신 것을 환영합니다</h1>
<p className="text-body-md">이것은 설명 문단입니다.</p>
```

AI는 매번 폰트 크기를 설명할 필요 없이 여러분의 타이포그래피 스케일을 존중합니다.

## 디자이너를 위한 이점

### 1. **단일 진실 공급원**

FramingUI에서 토큰을 한 번 정의합니다. 개발자, AI 에이전트, 문서 모두 같은 소스에서 가져옵니다. "어느 버전이 맞는 거야?" 같은 일이 없어집니다.

### 2. **더 빠른 디자인 반복**

색상 토큰을 업데이트하면 해당 토큰을 사용하는 모든 컴포넌트가 자동으로 업데이트됩니다. 개발자를 쫓아다니며 hex 코드를 바꾸라고 할 필요가 없습니다.

### 3. **더 나은 협업**

개발자가 지속적인 왕복 없이 컴포넌트를 생성할 수 있습니다. 여러분은 결과물을 리뷰하고 피드백을 주면 AI가 조정합니다—디자인 시스템은 그대로 유지하면서요.

### 4. **디자인 부채 감소**

하드코딩된 값은 디자인 부채를 만듭니다. MCP를 사용하면 모든 컴포넌트가 토큰 기반이 되어 디자인 시스템을 유지하고 발전시키기 쉬워집니다.

### 5. **기본적으로 접근성 보장**

FramingUI의 MCP 서버는 대비율과 WCAG 준수 여부를 확인할 수 있습니다. Claude에게 "이 색상 조합이 AA 기준을 충족해?"라고 물으면 즉시 답변합니다.

## 디자이너로서 시작하기

MCP의 이점을 누리기 위해 개발자가 될 필요는 없습니다. 시작하는 방법:

### 1. **디자인 토큰 정의하기**

FramingUI의 비주얼 에디터를 사용해서 컬러 팔레트, 스페이싱 스케일, 타이포그래피 시스템을 만드세요. 코드 필요 없음—디자인 결정만 하면 됩니다.

### 2. **MCP 서버 활성화**

토큰을 정의하면 FramingUI가 자동으로 MCP를 통해 노출합니다. 추가 설정 필요 없습니다.

### 3. **개발자와 협업**

FramingUI 프로젝트를 팀과 공유하세요. 이제 개발자가 Claude Code 같은 AI 어시스턴트에게 여러분의 디자인 토큰을 사용해서 컴포넌트를 생성하라고 요청할 수 있습니다.

### 4. **리뷰 및 반복**

AI가 컴포넌트를 생성하면 개발자의 작업을 리뷰하듯이 검토하세요. 차이점? AI는 여러분이 명시적으로 토큰을 변경하지 않는 한 디자인 시스템에서 벗어나지 않습니다.

## 디자이너가 자주 묻는 질문

### "Figma 같은 디자인 툴을 대체하나요?"

아닙니다. Figma는 여전히 탐색과 고품질 목업을 위한 캔버스입니다. FramingUI + MCP는 **디자인과 코드 사이의 다리**입니다. Figma에서 디자인하고, FramingUI에서 토큰을 정의하면, AI가 그 토큰을 사용해서 프로덕션 준비된 컴포넌트를 생성합니다.

### "AI가 제 의도를 추측하나요?"

이게 핵심 차이입니다. MCP 없이는 AI가 `#3498db` 같은 색상이나 `padding: 20px` 같은 여백을 환각(hallucination)합니다. MCP를 사용하면 AI가 **여러분의 정확한 디자인 시스템을 읽고** 정의한 토큰만 사용합니다. 환각 제로.

### "코딩을 배워야 하나요?"

전혀 아닙니다. FramingUI에서 시각적으로 토큰을 정의합니다. AI가 코드 생성을 처리합니다. 여러분은 디자인 결정에 집중하고, AI가 구현을 처리합니다.

### "토큰을 업데이트하면 어떻게 되나요?"

토큰 기반 디자인의 장점이 바로 이겁니다. `primary-600`을 `#7C3AED`에서 `#8B5CF6`로 업데이트하면 `primary-600`을 사용하는 모든 컴포넌트가 자동으로 업데이트됩니다. 수동 찾기-바꾸기 필요 없습니다.

## 디자이너-AI 협업의 미래

MCP는 시작일 뿐입니다. 이런 미래를 상상해보세요:

- Figma에서 레이아웃을 스케치하면 AI가 몇 초 만에 완전히 토큰 기반 React 컴포넌트를 생성합니다
- Claude에게 "이 디자인을 더 접근성 높게 만들어"라고 하면 색상, 여백, 타이포그래피를 조정해서 WCAG AAA를 충족합니다
- 디자인 토큰으로 A/B 테스트를 실행하고, AI가 사용자 행동을 분석해서 승리한 변형을 제안합니다
- 디자인 시스템이 AI가 사용 패턴을 분석한 실시간 피드백과 함께 진화합니다

FramingUI의 MCP 서버로 이 미래는 이미 여기 있습니다. AI는 디자이너를 대체하는 게 아니라—일관되고 접근 가능하며 유지 관리 가능한 디자인 시스템을 만드는 여러분의 능력을 증폭시킵니다.

## 직접 사용해보기

AI 에이전트가 여러분의 디자인 시스템을 읽고 사용하는 것을 보고 싶으신가요?

1. **FramingUI 탐색** — [framingui.com](https://framingui.com)에서 미리 만들어진 디자인 시스템을 확인하세요
2. **무료 체험 시작** — 신용카드 불필요, 테마를 선택하고 실험을 시작하세요
3. **Claude Code와 연결** — [MCP 설정 가이드](https://framingui.com/blog/mcp-design-system-claude-code)를 따라 AI 워크플로우와 통합하세요

AI가 `#3498db`를 환각하던 시대는 끝났습니다. AI 기반 디자인 시스템의 시대에 오신 것을 환영합니다. 🎨✨

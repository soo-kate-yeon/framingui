# AI 코드 생성을 망치는 디자인 시스템 실수 5가지

디자인 시스템이 있어요. 문서화된 컴포넌트, 스타일 가이드, 재사용 가능한 프리미티브. 그런데 Claude나 Cursor에게 새 기능을 만들어달라고 하면, 색상이 틀리고 간격이 제각각이고 border radius는 다른 제품에서 온 것 같습니다.

이건 AI 능력의 문제가 아니에요. AI 코드 생성기는 그럴듯한 UI를 만드는 데 능숙해요. 문제는 "그럴듯함"과 "우리 시스템과 일치함"이 다른 개념이라는 거예요. 디자인 시스템이 기계가 읽을 수 있는 형태로 구조화되어 있지 않으면, AI는 공개 저장소에서 학습한 패턴을 기본으로 씁니다. 그것들은 당신 제품과 맞지 않아요.

이 문제를 일으키는 구조적 실수 다섯 가지, 그리고 각각의 해결책을 정리했습니다.

## 실수 1: 토큰이 Figma에만 존재한다

디자이너가 색상, 간격, 타이포그래피를 Figma에서 관리해요. 개발자들은 인스펙터를 확인하고 값을 하드코딩합니다.

```tsx
<button style={{ padding: '12px 24px', background: '#3b82f6', borderRadius: '8px' }}>
  제출
</button>
```

AI가 코드를 생성할 때 Figma에 접근할 수 없어요. 학습 데이터 기반으로 그럴듯한 값을 만들어냅니다.

```tsx
<button className="px-6 py-3 bg-blue-500 rounded-lg">제출</button>
```

비슷하지만 틀렸어요. `blue-500`은 우리 주요 색상이 아니고, `px-6`는 우리 간격 스케일이 아닙니다.

**해결책:** AI가 임포트할 수 있는 TypeScript 파일로 토큰을 내보내세요.

```typescript
// src/design-system/tokens.ts
export const tokens = {
  color: {
    primary: {
      solid: 'var(--foreground-accent)',
      hover: 'var(--color-primary-700)',
    },
    neutral: {
      text:  'var(--foreground-primary)',
      muted: 'var(--foreground-secondary)',
    },
  },
  spacing: { sm: '0.5rem', md: '1rem', lg: '1.5rem' },
  radius:  { md: '0.375rem', lg: '0.5rem' },
}
```

이제 AI가 추측 대신 실제 값을 참조합니다. `CLAUDE.md`나 AI 설정에 이 파일 경로를 명시해서 도구가 어디를 봐야 할지 알게 하세요.

## 실수 2: 컴포넌트 이름이 일관되지 않다

코드베이스에 `PrimaryButton`, `ButtonPrimary`, `MainButton`이 모두 존재해요. 사람은 이 셋이 같은 역할을 한다는 걸 추론할 수 있지만, AI는 그렇지 않아요.

"주요 액션 버튼"을 요청하면 AI가 컴포넌트 대신 일반 `<button>`을 생성할 수 있어요. 셋 중 어느 이름이 표준인지 모르니까요.

**해결책:** 하나의 이름을 정하고 강제하세요.

```typescript
// 하나의 컴포넌트, 하나의 이름, variant로 구분
export const Button = ({ variant }: { variant: 'primary' | 'secondary' | 'ghost' }) => {
  /* ... */
}
```

README에 이렇게 명시하세요. "`<Button variant="primary">`를 쓰세요. `PrimaryButton`이나 `MainButton`은 사용하지 않습니다." AI는 이 내용을 읽고, 이후 버튼이 필요하면 `<Button variant="primary">`를 일관되게 생성합니다.

## 실수 3: 기계가 읽을 수 없는 컴포넌트 문서

Storybook은 브라우저에서 보기 좋지만, AI는 Storybook을 브라우징하지 않아요. 코드를 읽습니다.

JSDoc이나 명시적 TypeScript 타입이 없는 컴포넌트는 AI가 API를 추측하도록 강요해요.

```tsx
// AI가 모든 prop 이름을 추측해야 함
export const TextField = (props) => { /* ... */ }
```

AI가 `<TextField text="이메일" onChange={setEmail} />`을 생성합니다. 실제 API는 `label`과 `onValueChange`예요. 사용자가 버그를 신고하고, prop 이름 디버깅에 시간을 씁니다.

**해결책:** 타입과 사용 예시를 추가하세요.

```tsx
/**
 * 레이블과 유효성 검사를 지원하는 텍스트 입력 필드.
 *
 * @example
 * <TextField label="이메일" value={email} onValueChange={setEmail} error={errors.email} />
 */
export const TextField = ({
  label,
  value,
  onValueChange,
  error,
}: {
  label: string
  value: string
  onValueChange: (value: string) => void
  error?: string
}) => { /* ... */ }
```

AI가 JSDoc 예시를 읽고 올바른 prop 이름을 생성합니다. 타입 시그니처가 어긋남을 잡아내죠.

## 실수 4: 기존 컴포넌트의 매직 넘버

간격 시스템은 잘 정의되어 있어요. 그런데 기존 컴포넌트들이 값을 하드코딩합니다.

```tsx
<div style={{ marginTop: '24px', paddingLeft: '16px' }}>
```

AI는 컨텍스트에서 학습해요. 코드베이스에서 하드코딩된 값을 보면, 새 컴포넌트도 같은 패턴으로 생성합니다. 결국 `12px`, `18px`, `20px`, `24px`가 곳곳에 흩어지게 돼요. 모두 8px 그리드에 근접하지만 정확히 맞지는 않아요.

**해결책:** 하드코딩을 시맨틱 토큰 이름으로 교체하세요.

```typescript
export const spacing = {
  xs: '0.25rem', // 4px — 아이콘 간격, 좁은 레이블
  sm: '0.5rem',  // 8px — 컴팩트 리스트 항목
  md: '1rem',    // 16px — 기본 컴포넌트 패딩
  lg: '1.5rem',  // 24px — 섹션 간격
  xl: '2rem',    // 32px — 페이지 레벨 여백
}
```

의도를 설명하는 주석과 함께면, AI는 값뿐만 아니라 언제 쓸지도 이해합니다. 새 UI 생성이 그리드 안에 머물러요.

## 실수 5: 색조로만 이름 붙인 색상 토큰

`--blue-500`이나 `gray-700` 같은 색상 정의는 AI에게 사용 컨텍스트를 전달하지 않아요. `blue-500`이 주요 액션용인지, 링크용인지, 정보 상태용인지, 차트 채움용인지 알 수 없거든요.

결과: AI는 파란색이 필요한 곳이면 어디든 `blue-500`을 씁니다. 주요 액션 색상이 장식 요소에 등장하고, 링크 색상이 상태 배지에 올라갑니다.

**해결책:** 시맨틱 역할로 토큰 이름을 정하세요.

```typescript
export const color = {
  primary: {
    solid: 'var(--foreground-accent)',       // 주요 액션, CTA
    hover: 'var(--color-primary-700)',
    text:  'var(--foreground-inverse)',  // 밝은 배경의 주요 색상
  },
  neutral: {
    solid:  'var(--foreground-primary)',        // 기본 텍스트, 테두리
    muted:  'var(--foreground-secondary)',      // 보조 텍스트
    subtle: 'var(--foreground-muted)',       // 비활성, 플레이스홀더
  },
  feedback: {
    success: 'var(--color-success-500)',
    error:   'var(--border-error)',
  },
}
```

AI가 주요 버튼 색상이 필요하면 `color.primary.solid`를 씁니다. 보조 텍스트가 필요하면 `color.neutral.muted`를. 의도가 명확하니, 추가 지침 없이도 올바른 선택을 해요.

## 모든 것을 잇는 구조적 해결책

AI는 디자인 규칙을 찾으려고 코드베이스 전체를 탐색하지 않아요. 명확한 진입점을 찾습니다. 토큰이 `styles/colors.css`, `theme/spacing.ts`, `constants/typography.js`에 흩어져 있으면 AI가 연결하지 못해요.

단일 임포트 지점을 만드세요.

```
src/
  design-system/
    tokens.ts       ← 모든 것이 한 곳에
    components/
      Button.tsx    ← tokens.ts에서 임포트
      TextField.tsx
```

`CLAUDE.md`나 프로젝트 README에 이렇게 명시하세요.

```markdown
## 디자인 시스템

모든 토큰은 `src/design-system/tokens.ts`에 있습니다.
모든 컴포넌트는 이 파일에서 임포트합니다.
UI 생성 시 이 경로에서 토큰을 임포트하세요.
```

파일 하나, 지침 하나. AI가 일관되게 따를 수 있어요.

## 수정 후 달라지는 것

수정 전:

```tsx
// AI가 공개 패턴에서 추측
<button className="px-6 py-3 bg-blue-500 rounded-lg font-semibold">제출</button>
```

수정 후:

```tsx
// AI가 실제 시스템을 참조
import { Button } from '@/design-system/components'

<Button variant="primary">제출</Button>
```

AI가 즉흥을 멈추고 시스템을 따르기 시작해요. 빠른 속도로 UI를 생성하더라도 일관성이 유지됩니다. 제약이 지침이 아닌 구조에 있으니까요.

위의 다섯 가지 실수는 AI 코드 생성의 볼륨이 어느 수준에 달했을 때 비로소 표면에 드러나기 때문에 놓치기 쉬워요. 그 전에 고쳐두면, AI는 디자인 시스템을 위협하는 요소가 아닌 신뢰할 수 있는 확장 수단이 됩니다.

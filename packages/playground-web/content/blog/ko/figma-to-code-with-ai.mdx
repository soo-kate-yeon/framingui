프론트엔드 개발자가 기능 하나를 구현할 때, Figma 스펙을 코드로 옮기는 데만 몇 시간이 걸려요. 그리고 번역이 잘못된 부분을 고치는 데 또 시간이 들고요. 능력의 문제가 아니에요. Figma는 원시 값으로 말하고, 코드는 토큰으로 말하는데, 그 사이를 자동으로 이어주는 것이 없었던 거죠.

디자인 토큰이 바로 그 연결고리예요. 디자이너가 Figma Variables로 작업하고 그 변수들이 구조화된 JSON으로 내보내지면, AI가 전체 디자인 시스템을 읽고 양쪽 어느 팀도 픽셀 단위로 확인하지 않아도 디자인과 맞는 컴포넌트를 생성할 수 있어요.

## 기존 워크플로우가 망가지는 이유

Figma의 Inspect 패널을 열면 원시 CSS 값이 보여요. `#3B82F6`, `24px`, `Inter SemiBold 16px` 같은 것들이요. 이 값들에는 맥락이 없어요. 그 헥스 코드가 브랜드 메인 색상인지, 정보 배지 색조인지, 3개월 전 일회성으로 선택된 건지 개발자는 알 수 없거든요.

개발자는 추측해야 해요. 가장 가까운 Tailwind 클래스를 고르고, 간격을 어림잡고, 결과물이 디자인 검토를 통과하길 바라죠. 첫 번째 시도에서 통과하는 경우는 거의 없어요. 디자이너가 "그 파란색이 다르고 간격도 맞지 않아요"라고 답하고, 사이클이 반복돼요.

더 깊은 문제는 구조적이에요. 디자인은 Figma에 살고, 코드는 리포지토리에 살아요. 공유 데이터 형식 없이는 어느 쪽이든 변경이 생기는 순간 두 곳이 어긋나기 시작해요.

## 토큰이 다리가 되는 방식

Figma Variables는 디자인 토큰 구조에 직접 매핑돼요. `color/primary/500`이라는 이름의 색상 변수는 값, 타입, 그리고 선택적으로 설명을 가진 JSON 항목이 돼요. `spacing/md`라는 스페이싱 변수는 dimension 토큰이 되고요. 그 변수들을 내보내고 JSON을 커밋하면, 양쪽 플랫폼이 모두 참조할 수 있는 단일 진실 공급원이 생기는 거예요.

워크플로우는 이렇게 돼요.

1. 디자이너가 모든 색상, 간격, 타이포그래피 결정에 Figma Variables를 일관되게 사용
2. Figma REST API 또는 Tokens Studio로 토큰을 `tokens.json`으로 내보내기
3. 리포지토리에 `tokens.json` 커밋
4. AI가 FramingUI의 MCP 서버를 통해 토큰 읽기 — 프롬프트에 복붙할 필요 없음

Claude Code에게 카드 컴포넌트를 만들어달라고 하면, MCP 서버에 쿼리해서 실제 토큰 이름을 확인하고 그 이름을 사용해서 코드를 생성해요.

```tsx
<div className="bg-surface-primary p-6 rounded-md shadow-sm">
  <h3 className="text-lg font-semibold text-gray-900">Title</h3>
  <p className="text-sm text-gray-600 mt-4">Content</p>
</div>
```

모든 클래스가 여러분이 정의한 토큰으로 연결돼요. 코드베이스에 `#3B82F6`이 하드코딩되어 흘러들어오는 일이 없어요.

## MCP 연결 설정하기

FramingUI의 MCP 서버는 AI 도구와 디자인 시스템을 연결하는 런타임 레이어예요. 프로젝트당 한 번만 설정하면 돼요.

```bash
npx -y @framingui/mcp-server@latest init
```

이 명령어는 프레임워크를 감지하고, 의존성을 설치하고, 프로젝트 루트에 `.mcp.json`을 작성해요.

```json
{
  "mcpServers": {
    "framingui": {
      "type": "stdio",
      "command": "npx",
      "args": ["-y", "@framingui/mcp-server@latest"]
    }
  }
}
```

Claude Code를 재시작하면 AI가 필요할 때마다 토큰을 조회할 수 있어요. 디자이너가 Figma에서 색상을 업데이트하고 새 토큰을 내보내면, 다음 AI 생성 컴포넌트는 자동으로 업데이트된 값을 사용해요. 재설정이 필요 없어요.

## 토큰이 커버하지 못하는 부분

디자인 토큰은 *값*을 담아요. 색상, 간격, 보더 반지름, 폰트 패밀리 같은 것들이요. *행동*은 담지 못해요. Figma의 Auto Layout 규칙, 컴포넌트 variant 로직, 프로토타입 인터랙션, 반응형 제약사항은 토큰에 없어요.

레이아웃 의도는 AI에게 직접 설명해야 해요. "왼쪽에 고정 너비 이미지가 있고 오른쪽에 flex-grow 콘텐츠가 있는 카드"처럼 구조적 맥락을 주어야 해요. 토큰이 어휘라면, 여러분의 프롬프트가 문법이에요.

Figma의 텍스트 스타일, 이펙트 스타일, 그리드 스타일도 구조화된 토큰으로 내보내려면 추가 도구가 필요해요. Figma Variables에서 자동으로 매핑되지 않거든요. Tokens Studio가 대부분 처리해주지만, 의존하기 전에 실제로 내보내진 내용이 무엇인지 확인해보는 게 좋아요.

## 실질적인 변화

디자이너 입장에서 변화는 크지 않아요. 일회성 스타일 오버라이드 대신 Figma Variables를 일관되게 사용하는 것, 디자인 결정이 확정되면 토큰을 내보내고 파일을 커밋하는 것뿐이에요.

개발자 입장에서 변화는 더 커요. 컴포넌트 작업이 프롬프트 중심으로 바뀌어요. 필요한 것을 설명하면 AI가 토큰을 사용해서 코드를 생성하고, 개발자는 색상 조회나 간격 계산 대신 아키텍처와 접근성에 집중해서 검토해요.

검토 단계는 여전히 중요해요. AI는 구조적으로 올바른 코드를 생성하지만 레이아웃 엣지 케이스, 복잡한 상태 관리, ARIA 레이블 배치를 넘어선 접근성 세부 사항에는 사람의 판단이 필요해요. 병목이 번역에서 검토로 옮겨가는 건데, 사실 개발자 시간이 진짜 가치를 발휘하는 곳이 거기거든요.

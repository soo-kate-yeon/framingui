구조 없이 AI에게 버튼 컴포넌트를 만들어달라고 하면 뭔가를 발명해요. `<Button variant="primary" size="lg">`처럼 정의된 API로 요청하면 정확히 그걸 만들어내고요. 차이는 컴포넌트 variants — AI 출력의 일관성에 가장 큰 영향을 미치는 단일 패턴이에요.

## Boolean Props의 문제

디자인 시스템 컴포넌트는 단순하게 시작해서 점점 복잡해져요. 프로젝트 6개월 차가 되면 버튼 컴포넌트가 이렇게 변해 있기 일쑤예요:

```tsx
<Button
  primary
  large
  rounded
  loading={false}
  danger={false}
  outline={false}
  fullWidth={false}
  disabled={false}
>
  클릭하세요
</Button>
```

Boolean prop이 아홉 개예요. 무효한 조합도 무한히 가능하죠. AI가 이 컴포넌트를 생성하면 존재하지 않는 조합을 만들어내요 — `primary secondary large small`처럼 말이 안 되는 조합이요. TypeScript도 잡지 못해요. 각 prop이 개별적으로는 유효하니까요.

컴포넌트 variant 패턴은 boolean 플래그를 상호 배타적인 명명된 옵션으로 교체해서 이 문제를 해결해요.

## 컴포넌트 Variants가 뭐냐고요

Variants는 컴포넌트 설정을 **축(axis)** 으로 정리해요 — 고정된 세트에서 하나를 고르는 독립적인 차원들이에요.

버튼은 보통 세 가지 축이 필요해요:

```tsx
type ButtonProps = {
  variant?: 'primary' | 'secondary' | 'outline' | 'ghost';  // 시각적 스타일
  size?: 'sm' | 'md' | 'lg';                                // 크기
  children: React.ReactNode;
};
```

각 축은 상호 배타적이에요. `primary`이면서 동시에 `secondary`일 수 없죠. TypeScript가 컴파일 시점에 강제해요. AI도 정확히 어떤 값이 유효한지 알아요. 타입에 열거되어 있으니까요.

## CVA로 Variants 구현하기

CVA(class-variance-authority)는 Tailwind 프로젝트에서 variants를 관리하는 표준 라이브러리예요. 0.5kb짜리고 variant 로직을 선언적으로 만들어줘요:

```tsx
import { cva, type VariantProps } from 'class-variance-authority';

const buttonVariants = cva(
  'rounded-md font-medium transition', // 기본 클래스, 항상 적용
  {
    variants: {
      variant: {
        primary: 'bg-primary-500 hover:bg-primary-600 text-white',
        secondary: 'bg-secondary-500 hover:bg-secondary-600 text-white',
        outline: 'bg-transparent border border-primary-500 text-primary-500 hover:bg-gray-50',
        ghost: 'bg-transparent hover:bg-gray-100 text-text-primary',
      },
      size: {
        sm: 'px-3 py-1.5 text-sm',
        md: 'px-4 py-2 text-base',
        lg: 'px-6 py-3 text-lg',
      },
    },
    defaultVariants: {
      variant: 'primary',
      size: 'md',
    },
  }
);

type ButtonProps = VariantProps<typeof buttonVariants> & {
  children: React.ReactNode;
};

export function Button({ variant, size, children }: ButtonProps) {
  return (
    <button className={buttonVariants({ variant, size })}>
      {children}
    </button>
  );
}
```

TypeScript가 `buttonVariants`에서 유효한 옵션을 추론해요. `variant="blue"`를 넘기면 컴파일 에러가 나요. `variant="primary"`를 넘기면 올바른 Tailwind 클래스로 변환돼요.

## AI가 Variants로 더 나은 결과를 내는 이유

구조화된 API 없이 AI가 "primary 버튼"을 생성하면 이것들 중 뭔가가 나올 수 있어요:

```tsx
<button className="bg-blue-500 text-white px-4 py-2 rounded">
<button type="primary" className="...">
<Button primary={true} color="primary" type="button">
```

같은 개념의 세 가지 다른 표현이에요. 실제 컴포넌트 API와는 하나도 맞지 않아요.

디자인 토큰이나 MCP를 통해 variants가 정의되면:

```json
{
  "component": "button",
  "variants": {
    "variant": ["primary", "secondary", "outline", "ghost"],
    "size": ["sm", "md", "lg"]
  }
}
```

AI가 만드는 건 이거예요:

```tsx
<Button variant="primary" size="md">제출</Button>
<Button variant="ghost" size="sm">취소</Button>
```

이 패턴은 AI에게 디자인 결정을 내릴 만큼의 시맨틱 정보도 줘요. `ghost` variant에 `description: "Low-emphasis actions, tertiary buttons"`가 있으면 취소 버튼에는 `ghost`를, 주요 액션에는 `primary`를 써야 한다는 걸 알아요.

## Compound Variants

특정 variant 조합이 개별 variant와는 다른 스타일이 필요할 때가 있어요.

작은 outline 버튼(2px 테두리)은 `sm` 크기에서 너무 두꺼워 보여요. 거기선 1px이 맞아요:

```tsx
const buttonVariants = cva('rounded-md font-medium', {
  variants: {
    variant: {
      primary: 'bg-primary-500 text-white',
      outline: 'border-2 border-primary-500 text-primary-500',
    },
    size: {
      sm: 'px-3 py-1.5 text-sm',
      md: 'px-4 py-2 text-base',
    },
  },
  compoundVariants: [
    {
      variant: 'outline',
      size: 'sm',
      className: 'border', // border-2 대신 border(1px)로 오버라이드
    },
  ],
});
```

`<Button variant="outline" size="sm">`은 compound 클래스 오버라이드를 받아요. 나머지 조합은 개별 variant 스타일을 써요.

## 시맨틱하게 Variants 이름 짓기

Variant 이름은 외관이 아니라 목적을 설명해야 해요.

**피해야 할 것:** `variant: 'blue' | 'red' | 'gray'`

버튼이 어떻게 생겼는지를 설명해요. 브랜드 색상이 바뀌면 이름이 오해를 일으켜요. AI가 `variant="blue"`를 보면 이게 정보성 액션인지 브랜드 액션인지 알 방법이 없어요.

**써야 할 것:** `variant: 'primary' | 'destructive' | 'ghost'`

버튼이 무엇을 하는지를 설명해요. `destructive`는 "이건 위험한 동작"이라는 뜻이에요. 디자인 시스템이 danger 상태에 어떤 색을 쓰든 상관없이요. AI가 이름만 보고도 각 variant를 언제 써야 하는지 추론할 수 있어요.

Variants는 작게 유지해요. 축당 4~5개 옵션이면 보통 충분해요. 더 필요하다 싶으면 기존 축에 값을 더 추가하는 대신 새 축(예: `intent`)을 추가할지 고민해봐요.

## Boolean Props에서 마이그레이션하기

이미 boolean props가 있는 컴포넌트라면 마이그레이션 경로는 간단해요.

각 boolean을 축에 매핑해요:
- `primary`, `secondary`, `danger` → `variant` 축
- `large`, `small` → `size` 축
- `loading` → boolean 유지 (시각적 variant가 아닌 상태니까요)

기존 props에 deprecation 경고를 추가해요:

```tsx
type ButtonProps = {
  variant?: 'primary' | 'secondary' | 'danger';
  size?: 'sm' | 'md' | 'lg';

  /** @deprecated variant="primary" 를 사용하세요 */
  primary?: boolean;

  /** @deprecated size="lg" 를 사용하세요 */
  large?: boolean;
};
```

사용처를 점진적으로 마이그레이션해요. deprecated props는 여전히 동작하니까 아무것도 즉시 깨지지 않아요.

## 모든 조합 테스트하기

Variants는 완전한 테스트를 간단하게 만들어줘요:

```tsx
const variants = ['primary', 'secondary', 'outline', 'ghost'] as const;
const sizes = ['sm', 'md', 'lg'] as const;

describe('Button variants', () => {
  variants.forEach(variant => {
    sizes.forEach(size => {
      it(`${variant} ${size} 렌더링`, () => {
        const { container } = render(
          <Button variant={variant} size={size}>테스트</Button>
        );
        expect(container.firstChild).toMatchSnapshot();
      });
    });
  });
});
```

12개 조합이 자동으로 커버돼요. 새 variant를 추가하면 루프가 자동으로 잡아줘요. 테스트 코드는 건드릴 필요가 없어요.

Variant 패턴은 빠르게 값어치를 해요 — 깔끔한 API, 나은 TypeScript 커버리지, 더 예측 가능한 AI 출력, 그리고 컴포넌트 라이브러리가 커져도 자동으로 따라오는 테스트 스위트.

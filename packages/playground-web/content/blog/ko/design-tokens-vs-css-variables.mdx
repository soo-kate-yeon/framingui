"디자인 토큰이랑 CSS 변수는 같은 거 아닌가요?" 자주 듣는 질문이에요. 둘 다 재사용 가능한 디자인 값을 저장한다는 점에서 혼동하는 게 이해는 돼요. 그런데 이 둘은 서로 다른 추상화 수준에서 작동하고, 같은 것으로 취급하면 확장하기 어렵고 플랫폼 간에 공유할 수 없는 디자인 시스템이 돼요.

짧게 정리하면 이렇게요. 디자인 토큰은 소스 데이터예요. CSS 변수는 그 소스의 출력 형식 중 하나고요.

## CSS 변수: 브라우저 기능

CSS 커스텀 프로퍼티—흔히 CSS 변수라고 부르는—는 2016년에 브라우저에 도입됐어요. 값을 한 번 정의하고 스타일시트 전체에서 참조할 수 있게 해주죠.

```css
:root {
  --color-primary: #3b82f6;
  --spacing-base: 8px;
}

.button {
  background: var(--color-primary);
  padding: calc(var(--spacing-base) * 2);
}
```

브라우저가 렌더링 시에 이 값들을 계산해요. JavaScript로 재정의할 수도 있고, 특정 선택자에 범위를 한정할 수도 있고, `:root` 선언 몇 줄만 바꿔서 전체 UI를 바꿀 수도 있어요. 빌드 과정이 필요 없고요.

이 단순함이 장점이자 한계예요. CSS 변수는 브라우저 네이티브 문자열이에요. 여러분이 수동으로 만드는 구조 외에는 타입도, 설명도, 구조도 없어요. `--blue-500`이라는 이름의 변수는 그냥 색상스러운 이름을 가진 문자열이에요. 프라이머리 버튼 배경으로 의도됐어도 보더, 그림자, 호버 상태에 쓰는 걸 막을 방법이 없어요. 바로 옆에 `--blue-500-new`가 생겨나는 것도 막을 수 없고요.

브라우저를 벗어나지도 못해요. CSS 파일에서 iOS Swift 상수나 Android 색상 리소스를 생성할 수는 없어요.

## 디자인 토큰: 구조화된 소스 데이터

디자인 토큰은 플랫폼에 종속되지 않는 형식—일반적으로 JSON—으로 저장된 이름 붙은 디자인 결정이에요. 디자인 시스템의 공식 소스이자, 어떤 플랫폼 형식으로도 변환될 수 있을 만큼 충분한 구조를 갖고 있어요.

```json
{
  "color": {
    "primary": {
      "value": "#3b82f6",
      "type": "color",
      "description": "프라이머리 브랜드 색상. CTA, 링크, 포커스 상태에 사용."
    }
  },
  "spacing": {
    "base": { "value": "8px", "type": "dimension" },
    "md": { "value": "16px", "type": "dimension" }
  }
}
```

`type` 필드는 빌드 도구에게 토큰을 어떻게 변환할지 알려줘요. `color` 토큰은 CSS에서 헥스값이 되고, Swift에서는 `UIColor`가 되고, Android XML에서는 `<color>` 리소스가 돼요. `description` 필드는 사람과 AI 도구에게 이 토큰이 *무엇인지*가 아니라 *무엇을 위한 것인지*를 알려주고요.

이 구조가 디자인 토큰을 단순한 재사용 그 이상으로 가치 있게 만들어요. 토큰은 쿼리 가능하고, 검증 가능하고, 원시 CSS 파일로는 불가능한 방식으로 변환 가능해요.

## 관계: 소스와 출력

이런 모델로 이해하면 돼요.

```
디자인 토큰 (JSON)
        │
        ├──► CSS 변수         → 웹
        ├──► Swift 상수       → iOS
        └──► XML 리소스       → Android
```

디자인 토큰이 단일 진실 공급원이에요. CSS 변수는 파생 출력 형식 중 하나고요. JSON에서 토큰 값을 바꾸고 빌드를 실행하면 모든 플랫폼 형식이 동기화되어 업데이트돼요.

실용적으로 이게 중요한 이유가 있어요. 디자인 시스템이 CSS 변수로만 존재한다면 모바일 팀과 공유할 수 없어요. AI 도구에 디자인 결정에 대한 구조화되고 타입이 있고 설명이 있는 맥락을 줄 수 없어요. 토큰이 유효한 색상값인지 배포 전에 검증할 수도 없고요.

## 각각 어디에 맞는가

CSS 변수는 *런타임* 문제에서 강해요. 사용자가 다크 모드로 전환하면? `document.documentElement.setAttribute('data-theme', 'dark')`만 하면 그 선택자 아래의 CSS 변수가 즉시 적용돼요. 리빌드도 없고 지연도 없어요. 뷰포트 크기에 따른 동적 간격? CSS `calc()`와 커스텀 프로퍼티가 자연스럽게 처리해요. 빌드 도구 없는 빠른 프로토타입? CSS 변수는 브라우저에서 바로 작동해요.

디자인 토큰은 *작성*과 *배포* 문제에서 강해요. 웹과 모바일이 디자인 결정을 공유해야 하는 멀티 플랫폼 제품? 토큰만이 작업을 중복하지 않고 갈 수 있는 길이에요. AI 코드 생성이 임의의 Tailwind 기본값 대신 실제 색상 팔레트를 사용해서 컴포넌트를 만들기를 원해요? 토큰이 AI에게 필요한 구조화되고 시맨틱한 맥락을 줘요. `--color-primary`가 항상 유효한 헥스 또는 OKLCH 값인지 보장하고 싶어요? 토큰 스키마가 빌드 시에 검증해요.

대부분의 프로덕트 팀에게 실용적인 답은 둘 다예요. 토큰을 소스로, CSS 변수를 웹 출력으로 쓰는 거죠.

## FramingUI가 둘을 생성하는 방식

FramingUI는 디자인 토큰을 입력으로 받아서 하나의 소스에서 스택에 필요한 모든 것을 생성해요.

```json
{
  "color": {
    "primary": {
      "500": { "value": "#3b82f6", "type": "color" }
    }
  }
}
```

여기서 빌드 단계가 라이트 모드와 다크 모드 변형이 있는 CSS 변수, 그 변수를 참조하는 Tailwind config, 토큰 이름에 대한 TypeScript 타입 정의, 그리고 MCP 서버가 AI 쿼리에 답하는 데 사용하는 메타데이터를 만들어내요.

AI 통합은 특히 CSS 출력이 아닌 토큰 구조에 의존해요. Claude Code가 MCP 서버에 "어떤 색상이 사용 가능해요?"라고 물으면, JSON 토큰 데이터—이름, 값, 타입, 설명—를 돌려받아요. 그게 `bg-blue-500` 대신 `bg-primary-500`을 선택하게 하고, `primary-500`이 배경이 아니라 CTA에 적합하다는 걸 알게 해주는 거예요.

## CSS 변수에서 토큰으로 마이그레이션

이미 CSS 변수 시스템이 있고 토큰 기반 워크플로우로 옮기고 싶다면, 마이그레이션은 기계적으로 할 수 있어요.

기존 변수를 JSON으로 추출하는 것부터 시작해요.

```css
/* 이전 */
:root {
  --color-blue-500: #3b82f6;
  --spacing-md: 16px;
}
```

```json
{
  "color": { "blue": { "500": { "value": "#3b82f6", "type": "color" } } },
  "spacing": { "md": { "value": "16px", "type": "dimension" } }
}
```

그리고 원시 값을 참조하는 시맨틱 별칭을 추가해요.

```json
{
  "color": {
    "primary": {
      "value": "{color.blue.500}",
      "type": "color",
      "description": "프라이머리 브랜드 색상"
    }
  }
}
```

빌드 단계를 실행하면 토큰에서 CSS 변수가 생성돼요. 출력물은 이전과 동일하게 보이지만, 이제는 변환 가능하고 검증 가능하며 공유 가능하고 쿼리 가능한 구조화된 소스에서 나오는 거예요.

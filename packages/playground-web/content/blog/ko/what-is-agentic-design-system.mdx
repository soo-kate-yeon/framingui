## 이런 경험 있으시죠

AI에게 대시보드를 만들어달라고 합니다. 깔끔한 인터페이스가 나옵니다. 하지만 다른 AI 생성 대시보드와 똑같이 생겼습니다. 같은 버튼 스타일. 같은 카드 레이아웃. 같은 밋밋한 색상 팔레트. 작동은 하지만, *당신 것*이 아닙니다.

AI 잘못이 아닙니다. AI는 아는 것으로 작업합니다. 훈련 데이터의 일반적인 컴포넌트들입니다. AI는 당신의 브랜드, 디자인 언어, 시각적 아이덴티티를 이해하지 못합니다.

AI가 진짜 당신 제품처럼 느껴지는 인터페이스를 생성할 수 있다면 어떨까요?

## TL;DR

- **에이전틱 디자인 시스템**은 기계 가독형 디자인 토큰과 컴포넌트 지식으로 AI가 디자인 결정을 이해하게 합니다
- **문제**: 기존 디자인 시스템은 AI에게 컴포넌트 이름만 주지, 그 뒤의 *의미*는 주지 않습니다
- **해결책**: AI가 추론할 수 있는 의미론적 메타데이터로 디자인 토큰을 계층화합니다
- **framingui**는 3계층 아키텍처로 구현합니다: 토큰 생성 → 컴포넌트 지식 → AI 기반 생성
- 결과: 디자이너가 모든 변형을 수동으로 템플릿화하지 않아도 AI가 자동으로 브랜드에 맞는 컴포넌트를 생성합니다

## 에이전틱 디자인 시스템이란?

**에이전틱 디자인 시스템**은 AI 에이전트가 이해하고 자율적으로 사용할 수 있는 디자인 시스템입니다.

기존 디자인 시스템은 사람을 위해 만들어졌습니다. 제공하는 것:
- 시각적 가이드라인 (색상 팔레트, 타이포그래피 스케일)
- 컴포넌트 라이브러리 (버튼, 카드, 입력)
- 문서 (언제 무엇을 사용할지)

하지만 AI는 사람처럼 디자인을 "보지" 않습니다. Claude나 ChatGPT에게 "전문적인 대시보드 만들어줘"라고 하면, *당신의* 디자인 시스템 컨텍스트가 없어서 일반적인 UI를 생성합니다.

에이전틱 디자인 시스템은 디자인 결정을 기계 가독형 포맷으로 인코딩해 이 문제를 해결합니다:

1. **디자인 토큰** (색상, 간격, 타이포그래피)을 구조화된 데이터로
2. **컴포넌트 지식** (목적, 제약, 시각적 임팩트)을 의미론적 메타데이터로
3. **생성 규칙** (무엇을 어디에, 왜)을 계산 논리로

이렇게 하면 디자인 시스템이 참조 문서에서 AI가 픽셀 퍼펙트하고 브랜드에 맞는 인터페이스를 생성하는 **실행 가능한 명세**로 변환됩니다.

## 문제: AI 생성 UI가 왜 획일화되는가

### 디자인 컨텍스트 부재

"사용자 프로필 카드 만들어줘"라고 프롬프트하면, AI는 훈련 데이터 패턴을 기반으로 생성합니다:

```tsx
// 디자인 컨텍스트 없이 AI가 생성하는 것
export function ProfileCard() {
  return (
    <div className="p-4 bg-white rounded-lg shadow-md">
      <img src="/avatar.jpg" className="w-16 h-16 rounded-full" />
      <h3 className="text-lg font-semibold">홍길동</h3>
      <p className="text-gray-600">소프트웨어 엔지니어</p>
      <button className="mt-4 px-4 py-2 bg-blue-500 text-white rounded">
        프로필 보기
      </button>
    </div>
  );
}
```

작동은 합니다만, 주목하세요:
- 하드코딩된 색상 (`bg-white`, `text-gray-600`, `bg-blue-500`)
- 임의의 간격 (`p-4`, `mt-4`, `px-4 py-2`)
- 브랜드 시각 언어와 연결 없음
- 접근성 고려 없음 (대비율, 포커스 상태)

디자인 시스템이 있어도, 이 컴포넌트는 완전히 무시합니다.

### 토큰 적용 격차

AI에게 "우리 디자인 토큰 써"라고 해도 근본적인 격차가 있습니다:

**당신의 디자인 토큰:**
```css
:root {
  --color-primary-500: #3B82F6;
  --color-neutral-100: #F5F5F5;
  --spacing-4: 1rem;
  --radius-md: 0.5rem;
}
```

**AI의 도전:**
- 카드 배경에 어떤 토큰을 써야 하지?
- `--color-primary-500`과 `--color-primary-600` 언제 쓰지?
- `--spacing-4`가 시각적 계층과 어떤 관계지?
- 컴포넌트 변형의 의미론적 차이가 뭐지?

AI가 토큰을 기계적으로 적용할 수는 있지만, 그 뒤의 *디자인 의도*를 이해하지 못합니다.

### 일관되지 않은 AI 출력

디자인 시스템 컨텍스트 없이, 모든 AI 생성은 주사위를 굴리는 것과 같습니다:

- **세션 1**: AI가 둥근 모서리 버튼 생성
- **세션 2**: AI가 같은 버튼을 각진 모서리로 생성
- **세션 3**: AI가 완전히 다른 버튼 스타일 사용

이것은 AI가 디자인 결정의 메모리가 없기 때문입니다. 매번 처음부터 생성하고, 이전 선택에 대한 참조가 없습니다.

## 해결책: 디자인 토큰 + AI 이해

에이전틱 디자인 시스템은 세 계층을 결합해 이 격차를 해소합니다:

### 계층 1: 결정론적 토큰 생성

지각적으로 균일한 색상 공간(OKLCH)과 내장 접근성 검증으로 디자인 토큰을 생성합니다.

색상을 수동으로 골라 WCAG 대비 요구사항 충족을 바라는 대신, 토큰이 **알고리즘적으로 생성**됩니다:

```typescript
import { generateTokensFromArchetype } from '@framingui/token-generator';

const archetype = {
  name: "professional-dark",
  primaryColor: { l: 0.5, c: 0.15, h: 220 }, // OKLCH 블루
  neutralTone: "tinted",
  contrast: "high"
};

const tokens = await generateTokensFromArchetype(archetype, {
  wcagLevel: 'AA',
  generateDarkMode: true
});

// 결과: WCAG 준수가 보장된 100개 이상 토큰
// - Primary 스케일: 50, 100, 200...900, 950
// - 의미론적 토큰: bg.surface.base, fg.primary, border.default
// - 컴포넌트 토큰: button.bg.primary, card.bg.elevated
```

핵심 이점:
- **결정론적**: 같은 입력은 항상 같은 출력
- **기본으로 접근성 확보**: 자동 WCAG AA/AAA 검증
- **수학적으로 건전**: OKLCH가 지각적 균일성 보장

### 계층 2: 컴포넌트 지식 시스템

토큰을 AI가 추론할 수 있는 **의미론적 메타데이터**로 변환합니다.

컴포넌트를 단순히 나열하는 대신, 목적, 제약, 시각적 임팩트를 인코딩합니다:

```typescript
// packages/component-knowledge/src/catalog/button.ts
export const ButtonKnowledge: ComponentKnowledge = {
  name: "Button",
  category: "action",
  purpose: "클릭 시 액션이나 내비게이션 트리거",
  visualImpact: "high", // 사용자 주의 끌기
  complexity: "low",

  // 슬롯 친화도: 이 컴포넌트를 어디에 쓸까?
  slotAffinity: {
    card_actions: 0.9,  // 카드 액션 슬롯에 완벽
    modal_footer: 0.85, // 모달 액션에 좋음
    header: 0.3,        // 헤더에는 드뭄
    sidebar: 0.1        // 사이드바에는 거의 없음
  },

  variants: [
    {
      name: "primary",
      purpose: "주어진 컨텍스트의 메인 콜투액션",
      tokens: {
        bg: "button.bg.primary",
        fg: "button.fg.primary",
        border: "button.border.primary"
      },
      constraints: ["max-one-per-viewport"]
    },
    {
      name: "ghost",
      purpose: "primary 액션과 경쟁하지 않는 저강조 액션",
      tokens: {
        bg: "button.bg.ghost",
        fg: "button.fg.ghost"
      }
    }
  ]
};
```

이 메타데이터가 AI에게 알려주는 것:
- **언제 컴포넌트를 사용할지** (슬롯 친화도 점수)
- **어떤 변형을 선택할지** (시각적 계층 기반)
- **어떤 토큰을 적용할지** (의미론적 토큰 매핑)
- **디자인 제약** (화면당 primary 버튼 최대 1개)

### 계층 3: AI 기반 컴포넌트 생성

의미론적 점수로 컴포넌트를 자동 선택하고 올바른 토큰 적용과 함께 배치합니다.

AI가 "사용자 대시보드 만들어줘" 같은 프롬프트를 받으면, 생성 엔진이:

1. **의도 분석**: "대시보드 = 데이터 중심, 읽기 전용, 전문적 톤"
2. **컴포넌트 적합성 점수화**: 슬롯 친화도로 모든 컴포넌트 순위
3. **안전 프로토콜 적용**: 제약 검증 및 환각 방지
4. **React 코드 생성**: 올바른 토큰과 컴포넌트 참조 주입

```typescript
// framingui가 전체 컨텍스트로 생성하는 것
import { Card, Badge, DataTable, Button } from '@framingui';
import { tokens } from '@framingui/design-tokens';

export function UserDashboard() {
  return (
    <Card
      variant="elevated"
      style={{
        background: tokens.bg.surface.elevated,
        padding: tokens.spacing['6'],
        borderRadius: tokens.radius.lg
      }}
    >
      <div style={{ display: 'flex', gap: tokens.spacing['3'] }}>
        <Badge
          variant="success"
          style={{
            background: tokens.badge.bg.success,
            color: tokens.badge.fg.success
          }}
        >
          활성
        </Badge>
      </div>

      <DataTable
        columns={columns}
        data={users}
        style={{
          marginTop: tokens.spacing['6']
        }}
      />

      <div style={{
        marginTop: tokens.spacing['4'],
        display: 'flex',
        gap: tokens.spacing['2']
      }}>
        <Button variant="primary">데이터 내보내기</Button>
        <Button variant="ghost">새로고침</Button>
      </div>
    </Card>
  );
}
```

주목할 점:
- 모든 색상이 디자인 토큰 참조 (하드코딩 값 없음)
- 간격이 의미론적 스케일 사용 (일관된 시각적 리듬)
- 컴포넌트 변형이 디자인 의도와 매칭 (primary vs ghost 버튼)
- 레이아웃이 접근성 존중 (적절한 대비, 포커스 관리)

## framingui 작동 방식

framingui는 이 3계층 아키텍처를 엔드투엔드로 구현합니다.

### 1단계: 디자인 아키타입 정의

브랜드의 디자인 성격을 설명하는 것부터 시작합니다:

```typescript
// 아키타입 JSON 생성
{
  "name": "fintech-professional",
  "primaryColor": { "l": 0.45, "c": 0.13, "h": 240 },
  "neutralTone": "tinted",
  "contrast": "high",
  "borderRadius": "minimal",
  "fontScale": "compact"
}
```

### 2단계: 디자인 토큰 생성

framingui가 100개 이상 토큰을 자동 생성합니다:

```bash
npx @framingui/cli init --archetype fintech-professional

# 출력:
# - tokens.css (CSS 커스텀 속성)
# - tokens.json (디자인 도구용)
# - tailwind.config.js (Tailwind 통합)
```

```css
/* 생성된 tokens.css */
:root {
  /* 의미론적 배경 토큰 */
  --bg-surface-base: oklch(0.98 0.01 240);
  --bg-surface-elevated: oklch(1.0 0.0 0);

  /* 전경 토큰 */
  --fg-primary: oklch(0.25 0.03 240);
  --fg-secondary: oklch(0.45 0.02 240);

  /* 컴포넌트별 토큰 */
  --button-bg-primary: oklch(0.45 0.13 240);
  --button-fg-primary: oklch(0.98 0.01 240);

  /* 간격 스케일 */
  --spacing-1: 0.25rem;
  --spacing-2: 0.5rem;
  --spacing-4: 1rem;
  --spacing-6: 1.5rem;
}
```

### 3단계: 컴포넌트 지식 로드

framingui는 의미론적 메타데이터가 미리 설정된 20개 이상 컴포넌트를 제공합니다:

```typescript
import { getAllComponents } from '@framingui/component-knowledge';

const components = getAllComponents();

// 반환: ComponentKnowledge[] 포함:
// - Button, Card, Input, Badge, DataTable, Modal...
// - 지능적 배치를 위한 슬롯 친화도 점수
// - 각 변형의 토큰 매핑
// - 접근성 제약
```

### 4단계: AI가 화면 생성

framingui의 MCP (Model Context Protocol) 통합 사용:

```typescript
import { renderScreen } from '@framingui/studio-mcp';

const blueprint = {
  blueprintId: "dash-001",
  recipeName: "user-dashboard",
  analysis: {
    intent: "사용자 데이터가 있는 대시보드 화면",
    tone: "professional"
  },
  structure: {
    componentName: "Card",
    props: { variant: "elevated" },
    slots: {
      header: {
        componentName: "Badge",
        props: { variant: "success", text: "활성" }
      },
      content: {
        componentName: "DataTable",
        props: { columns: ["이름", "이메일", "역할"] }
      },
      footer: {
        componentName: "Button",
        props: { variant: "primary", label: "내보내기" }
      }
    }
  },
  themeId: "fintech-professional"
};

const result = await renderScreen(blueprint);

// 작성: src/screens/user-dashboard/page.tsx
```

생성된 컴포넌트는 토큰을 사용하고, 컴포넌트 계약을 따르며, 디자인 시스템과 정확히 매칭됩니다.

### 5단계: 토큰 강제 스타일링 (SPEC-STYLED-001)

framingui는 **3계층 강제**로 디자인 드리프트를 방지합니다:

#### 계층 1: 컴파일 타임 (TypeScript)

```typescript
import { styled, tokens } from '@framingui/styled';

// ✅ 유효: 토큰만 사용하는 스타일링
const Card = styled.div`
  background: ${tokens.bg.surface.elevated};
  padding: ${tokens.spacing[6]};
  border-radius: ${tokens.radius.lg};
`;

// ❌ 무효: 하드코딩 값에 TypeScript 오류
const BadCard = styled.div`
  background: #ffffff; // 오류: 'string' 타입 할당 불가
  padding: 16px;       // 오류: 'string' 타입 할당 불가
`;
```

#### 계층 2: 런타임 검증

```typescript
// 런타임 체크가 하드코딩 값 감지
const DynamicCard = styled.div`
  background: ${props => props.customBg || tokens.bg.surface.base};
`;

// customBg가 하드코딩 색상이면 오류
<DynamicCard customBg="#fff" /> // 런타임 오류!
```

#### 계층 3: 빌드 타임 (esbuild 플러그인)

```typescript
// esbuild.config.js
import { tektonPlugin } from '@framingui/esbuild-plugin';

export default {
  plugins: [
    tektonPlugin({
      strict: true,        // 위반 시 빌드 실패
      threshold: 100       // 100% 토큰 준수 요구
    })
  ]
};
```

하드코딩된 색상, 간격, 디자인 값을 사용하면 빌드가 실패합니다. AI 에이전트(또는 인간 개발자)가 디자인 시스템을 우회하는 것이 **불가능**해집니다.

## 비교: 기존 vs 에이전틱 디자인 시스템

| 측면 | 기존 디자인 시스템 | 에이전틱 디자인 시스템 |
|--------|---------------------------|------------------------|
| **토큰 포맷** | CSS 변수, 사람 읽는 문서 | 의미론적 메타데이터가 있는 JSON |
| **컴포넌트 문서** | Storybook, Figma 파일 | 슬롯 친화도 점수가 있는 ComponentKnowledge 카탈로그 |
| **AI 통합** | 문서에서 복사-붙여넣기 | 의미론적 추론이 있는 프로그래밍 API |
| **일관성** | 수동 강제, 코드 리뷰 | 빌드 타임 검증으로 자동 강제 |
| **토큰 적용** | 개발자가 수동 선택 | AI가 의미론적 점수로 선택 |
| **접근성** | 수동 WCAG 체크 | 토큰 생성 시 자동 WCAG 검증 |
| **디자인 드리프트** | 팀 규모 커지면 시간 지나며 발생 | 3계층 강제로 방지 (컴파일, 런타임, 빌드) |
| **온보딩** | 문서 읽기, 관례 학습 | AI가 첫날부터 올바른 코드 생성 |
| **업데이트** | 수동 찾기-바꾸기 | 업데이트된 아키타입에서 재생성 |

### 실제 영향

**기존 접근법:**
1. 디자이너가 Figma에서 목업 생성
2. 개발자가 간격과 색상 눈대중
3. 하드코딩 값이 코드베이스에 침투
4. 시간 지나며 디자인 시스템 드리프트
5. 분기별 "디자인 시스템 정리" 스프린트

**에이전틱 접근법:**
1. 디자이너가 아키타입 정의 (5분)
2. 토큰 + 컴포넌트 지식 생성 (자동)
3. AI가 올바른 토큰으로 컴포넌트 생성 (초)
4. 하드코딩 값 있으면 빌드 실패 (강제)
5. 디자인 시스템 영구적으로 일관성 유지

- CSS, Tailwind, JSON으로 토큰 내보내기

### 옵션 2: 로컬 설치

```bash
# framingui 설치
npm install @framingui/core @framingui @framingui/styled

# 디자인 시스템 초기화
npx @framingui/cli init

# 대화형 프롬프트 따라:
# - Primary 색상
# - 대비 수준 (AA 또는 AAA)
# - 테두리 반경 (minimal, moderate, rounded)
# - 타이포그래피 스케일
```

### 옵션 3: AI 도구와 통합

framingui는 Claude Code, Cursor, MCP 지원 도구와 연동됩니다:

```json
// .claude/mcp.json
{
  "servers": {
    "tekton": {
      "command": "npx",
      "args": ["@framingui/mcp-server"],
      "env": {
        "TEKTON_PROJECT_PATH": "."
      }
    }
  }
}
```

이제 Claude가:
- 컴포넌트 카탈로그 쿼리
- 디자인 토큰으로 화면 생성
- 계약에 대해 컴포넌트 검증

## FAQ

### 에이전틱 디자인 시스템이 일반 디자인 시스템과 뭐가 다른가요?

일반 디자인 시스템은 사람을 위한 디자인 결정 문서입니다. 에이전틱 디자인 시스템은 AI가 자율적으로 이해하고 적용할 수 있도록 기계 가독형 포맷으로 그 결정을 인코딩합니다. 여기에는 의미론적 메타데이터(슬롯 친화도, 시각적 임팩트), 계산 규칙(언제 무엇을 사용), 강제 메커니즘(디자인 드리프트 방지)이 포함됩니다.

### 디자이너를 대체하나요?

아니요. 디자이너가 여전히 모든 창작 결정을 합니다—색상 팔레트, 타이포그래피, 컴포넌트 변형. framingui는 단순히 그 결정을 프로덕션 코드로 번역하는 것을 자동화합니다. 디자인 시스템의 컴파일러로 생각하세요: 디자이너가 스펙을 쓰고, AI가 구현을 생성합니다.

### 이미 디자인 시스템이 있으면요?

framingui는 기존 디자인 시스템과 통합할 수 있습니다:
1. 기존 토큰 가져오기 (CSS 변수 → FramingUI JSON)
2. 컴포넌트를 ComponentKnowledge 포맷으로 매핑
3. 컴포넌트 위에 FramingUI 생성 레이어 사용

토큰 강제 레이어(`@framingui/styled`)는 모든 React 컴포넌트 라이브러리와 작동합니다.

### AI가 브랜드에 안 맞는 UI 생성을 어떻게 방지하나요?

세 가지 메커니즘:
1. **의미론적 점수**가 AI가 디자인 의도에 맞는 컴포넌트 선택 보장 (예: 메인 CTA에 primary 버튼)
2. **토큰 강제**가 하드코딩 색상/간격 사용 불가능하게 만듦 (컴파일, 런타임, 빌드 체크)
3. **컴포넌트 계약**이 생성 코드를 디자인 규칙에 대해 검증 (primary 버튼 최대 1개, 필수 접근성 속성)

### Tailwind CSS와 함께 쓸 수 있나요?

네! framingui는 Tailwind 호환 설정을 내보냅니다:

```javascript
// tailwind.config.js (생성됨)
module.exports = {
  theme: {
    extend: {
      colors: {
        primary: {
          50: 'oklch(0.95 0.03 240)',
          500: 'oklch(0.5 0.15 240)',
          900: 'oklch(0.25 0.08 240)'
        }
      },
      spacing: {
        '4': '1rem',
        '6': '1.5rem'
      }
    }
  }
};
```

그다음 컴포넌트에서 토큰 사용:

```tsx
<div className="bg-primary-500 p-6 rounded-lg">
  {/* 여전히 디자인 토큰 참조 */}
</div>
```

### React만 지원하나요?

현재는 그렇습니다. framingui는 TypeScript와 함께 React 컴포넌트를 생성합니다. Vue, Svelte, 기타 프레임워크 지원은 로드맵에 있습니다. 하지만 토큰 생성 레이어(계층 1)는 프레임워크 무관하며 표준 CSS 변수를 출력합니다.

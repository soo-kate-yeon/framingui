AI에게 대시보드 카드를 만들어 달라고 하면 그럴듯해 보이지만 제품이랑 안 맞는 걸 받게 돼요. 파란색이 약간 틀려요. 여백이 비슷하지만 어긋나 있어요. 테두리 반경은 그 자체로는 괜찮은데 화면 나머지와 일관성이 없어요.

이걸 "hallucination"이라고 부르는데, 맞는 표현이지만 불완전해요. 더 깊은 원인은 컨텍스트 부족이에요. 모델이 여러분 디자인 시스템에 접근할 수 없으니 학습 데이터에서 가장 통계적으로 그럴듯한 값으로 빈칸을 채우는 거예요. 결과는 정확한 코드가 아닌 일반적인 코드입니다.

## UI 생성에서 hallucination이란 무엇인가

사실 영역에서 hallucination은 모델이 거짓 정보를 만들어낸다는 뜻이에요. UI 생성에서는 일반적으로는 그럴듯하지만 우리 시스템에는 틀린 디자인 결정을 자신 있게 내리는 걸 말해요.

**의미적 hallucination:** 유효해 보이지만 존재하지 않는 토큰 이름을 생성해요. 우리 시스템이 `color.primary.solid`를 쓰는데 `color.brand.700`을 만들어냅니다. 우리 스케일이 숫자 키를 쓰는데 `spacing.large`를 써요.

**시각적 hallucination:** 확립된 패턴과 충돌하는 임의 색상, 여백, 라디우스 값을 선택해요. 코드는 돌아가는데 디자인이 깨집니다.

**구조적 hallucination:** 인기 라이브러리를 닮았지만 우리 프리미티브와 맞지 않는 컴포넌트 API와 variant를 발명해요. 우리 API가 `<Button variant="primary">`인데 `<Button type="submit" color="blue">`를 만들어냅니다.

이 세 가지 형태는 모델 컨텍스트에 디자인 토큰이 있으면 훨씬 덜 발생해요.

## 토큰이 없으면 왜 hallucination이 생기나

### 모델은 옳은 것이 아닌 가장 그럴듯한 것을 생성한다

LLM은 현재 컨텍스트에서 가장 그럴듯한 다음 토큰을 예측해서 출력을 생성해요. 프롬프트에 토큰 데이터 없이 "대시보드 카드 만들어줘"라고 하면 수백만 개의 학습 예제에서 대시보드 카드가 어떻게 생겼는지가 가장 그럴듯한 출력이 됩니다. 합리적인 파란 hex 코드가 나와요. `padding: 16px`가 나오고요. 통계적으로 흔한 것, 우리 시스템이 명시한 것이 아닌.

### 자연어는 너무 모호하다

"우리 브랜드 색상 써줘"는 디자인 시스템을 한 번 본 사람에게도 애매해요. 시스템 접근권이 없는 모델에게는 거의 의미가 없어요. 브랜드가 primary를 뜻할 수도 있고 전체 시맨틱 팔레트를 뜻할 수도 있어요. 텍스트가 `color.text.primary`여야 할지 `color.primary.fg`여야 할지는 명시적 지정 없이 불투명하고요.

구조화된 토큰 데이터가 이 모호함을 해소해요. 기계 판독 가능한 이름과 값은 자연어가 전달할 수 없는 정밀도를 가집니다.

### 산문 가이드라인은 제약을 강제하지 못한다

위키에 "8px 간격을 사용하세요"라고 써 있어도 생성된 JSX를 안정적으로 관리하지 못해요. 모델이 그 가이드라인을 파싱하고, 이 특정 컴포넌트에 무엇을 의미하는지 결정하고, 적용해야 해요. 각 단계에서 오류가 생길 수 있어요. `spacing.2 = 8px`가 있는 토큰 스키마는 추론 없이 같은 제약을 제공합니다.

### 컨텍스트 윈도우 경쟁

매 프롬프트에 긴 디자인 가이드라인을 붙여 넣는 건 두 가지 면에서 역효과예요. 실제 기능 명세와 컨텍스트 윈도우 공간을 경쟁하고, 적용하려면 여전히 추론이 필요해서 동일한 가이드라인으로도 생성마다 결과가 달라져요.

파일 참조든 MCP 같은 프로토콜이든 구조화된 토큰 조회는 세션 전반에 걸쳐 확장되는 밀도 높은 조회 가능한 제약을 제공합니다.

## 제약 인터페이스로서의 디자인 토큰

디자인 토큰은 테마나 디자인-개발 핸드오프만을 위한 게 아니에요. AI 워크플로우에서 디자인 시스템과 코드 생성 사이의 제약 인터페이스 역할을 합니다.

좋은 토큰 스키마는 AI 생성에 세 가지를 해줘요. 외형이 아닌 역할을 담은 **안정적인 시맨틱 이름** 제공, 추론 필요성을 없애는 **명시적 값** 정의, 디자인 의도를 구현에 매핑하는 **컴포넌트 수준 의미론** 확립.

AI가 이 계약을 조회할 수 있으면 생성이 확률적 추측에서 제약된 조합으로 전환돼요. 모델 능력이 떨어지는 게 아니라 방향이 더 잘 잡히는 거예요.

## 추측-수정에서 생성-배포로

토큰 컨텍스트 없이 일반적인 워크플로우는:

1. AI에게 컴포넌트 생성 요청
2. 그럴듯하지만 어긋난 코드 수신
3. 값을 수동으로 찾아 교체
4. 모든 출력에 반복

이러면 모든 AI 생성 컴포넌트에 리뷰 오버헤드가 생겨요. 엔지니어가 동작과 로직 대신 스타일 교정에 시간을 써요. AI 사용이 늘수록 오버헤드가 쌓입니다.

토큰 컨텍스트 있으면:

1. AI가 토큰 정의와 컴포넌트 계약을 읽음
2. 유효한 시맨틱 참조로 생성
3. 리뷰가 스타일 위반이 아닌 동작, 카피, 조합에 집중

전환이 비대칭이에요. 생성 시점의 더 나은 컨텍스트가 다운스트림 수정을 불균형하게 줄여줍니다.

## 안정적인 생성을 위한 최소 토큰 스키마

hallucination을 의미 있게 줄이는 최소 실행 가능 스키마는 다섯 영역을 커버해요.

**색상:** 텍스트, 표면, 테두리, 시맨틱 액션 토큰을 분리하세요. `color.text.primary`, `color.text.secondary`. `color.surface.default`, `color.surface.elevated`. `color.border.default`, `color.border.focus`. `color.intent.success`, `color.intent.warning`, `color.intent.danger`.

**여백과 사이즈:** 일관된 숫자 스케일. `spacing.1` ~ `spacing.10`. `radius.sm`, `radius.md`, `radius.lg`. `size.control.sm`, `size.control.md`, `size.control.lg`.

**타이포그래피:** `font.size.body`, `font.size.label`, `font.size.title`. `font.weight.regular`, `font.weight.medium`, `font.weight.semibold`. `lineHeight.body`, `lineHeight.heading`.

**컴포넌트 의미 토큰:** `button.primary.bg`, `button.primary.fg`, `input.border.default`, `input.border.focus`.

일반적인 UI 생성 시나리오의 대부분을 커버해요. 시스템이 성장하면 레이어를 추가하되, 여기서 시작하세요.

## AI 도구에 토큰을 전달하는 방법

세 가지 접근법이 있고, 각각 다른 트레이드오프가 있어요.

**정적 프롬프트 주입** — 토큰 JSON을 매 프롬프트에 직접 붙여 넣어요. 테스트하기 쉽고 확장이 불가능해요. 시스템이 발전하면서 토큰 데이터가 구식이 되고, 팀 전반에서 규율이 무너집니다.

**저장소 내 토큰 파일** — 토큰을 버전 관리 파일에 저장하고 워크플로우 지침에서 참조해요. 수동 붙여넣기보다 안정적이지만, 여전히 프롬프트 규율에 의존하고 파일이 관리되지 않으면 구식이 돼요.

**프로토콜 기반 조회 (MCP)** — AI 클라이언트가 생성 시점에 구조화된 인터페이스로 토큰 데이터를 조회해요. 컨텍스트가 항상 최신이고, 팀원들이 토큰 포함에 대한 프롬프트 규율이 필요 없고, 수동 오버헤드 없이 확장돼요. 지속적인 AI 지원 개발에는 이 방식이 유리합니다.

## 토큰을 넘어서는 가드레일

토큰이 hallucination을 크게 줄이지만 모든 오류를 없애지는 못해요. 완전한 설정은 다음을 추가합니다.

컴파일 시점에 잘못된 생성을 실패하게 만드는 **제약된 컴포넌트 API**. `<Button>`이 정의된 variant와 size만 받는다면 `variant="super-primary"`를 발명한 AI는 잘못된 스타일이 있는 실행 앱이 아닌 타입 오류를 냅니다.

하드코딩 hex 색상, 임의 픽셀 값, 승인되지 않은 유틸리티 조합을 감지하는 **린트 규칙**. 토큰 컨텍스트가 놓친 부분을 잡아요.

빠른 생성 사이클에서 의도치 않은 시각적 표류를 잡는 **스냅샷 또는 시각 회귀 테스트**. 많은 컴포넌트가 빠르게 생성되는 스프린트 기간에 유용합니다.

지시 세트를 표준화하는 **명시적 제약이 있는 프롬프트 템플릿**: "기존 프리미티브만 사용. 변형 발명 금지. 하드코딩 값 대신 토큰 참조 사용." 개별 프롬프트 작성자에게 의존하지 말고 팀 공유 템플릿으로 인코딩하세요.

## FramingUI가 여기서 하는 역할

FramingUI는 토큰 우선 디자인 시스템 아키텍처와 AI 워크플로우를 위한 도구를 결합해서 이 맥락에서 유용해요. 시맨틱 토큰을 정의하고 발전시키고, 컴포넌트 프리미티브를 토큰에 맞게 유지하고, 생성 시점에 AI 도구에 토큰 컨텍스트를 연결하는 것을 쉽게 만들어줘요.

핵심 문제를 해결하는 조합이에요. 웹 전반의 통계적 평균이 아닌 우리 시스템의 실제 제약에서 시작하는 AI 생성이요.

---

AI가 일반적인 UI를 생성하면 모델을 탓하고 싶어져요. 하지만 더 생산적인 관점은 어떤 컨텍스트가 빠졌는지 묻는 거예요. 디자인 토큰이 그 질문에 대한 답입니다. AI 한계에 대한 우회책이 아니라, AI 지원 UI 워크플로우가 필요로 하는 기준이에요.

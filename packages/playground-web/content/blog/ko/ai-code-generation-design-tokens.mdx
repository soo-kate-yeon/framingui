## AI가 UI를 "그럴듯하게" 망치는 이유

AI로 컴포넌트를 만들면 처음엔 감탄하게 됩니다. 속도는 빠르고 코드도 얼핏 깔끔해 보입니다. 그런데 실제 서비스에 붙이면 이상한 지점이 반복됩니다.

- 색상이 브랜드와 미묘하게 다름
- 간격이 기존 화면과 맞지 않음
- 버튼/입력창 상태 표현이 제각각임
- 존재하지 않는 변형 이름을 만들어냄

많은 팀이 이걸 hallucination이라고 부릅니다. 맞는 표현이지만, 원인을 더 정확히 보면 "디자인 시스템 맥락 부족"입니다.

## UI 생성에서 hallucination은 무엇을 뜻하나

일반적으로 hallucination은 사실 오류를 의미하지만, UI 생성에서는 조금 다릅니다. "일반적으로는 맞아 보이지만 우리 시스템에는 틀린 결정"도 hallucination입니다.

### 의미 hallucination

실제로 없는 토큰 이름을 생성합니다. 예: `color.brand.700` 같은 그럴듯한 가짜 키.

### 시각 hallucination

색상/간격/라운드 값을 임의로 선택합니다. 결과는 작동하지만 화면 통일성이 깨집니다.

### 구조 hallucination

컴포넌트 API를 추측으로 발명합니다. 예: 존재하지 않는 `Button` props를 사용.

이 세 가지는 대부분 토큰 컨텍스트가 부족할 때 발생합니다.

## 토큰이 없으면 왜 이런 일이 생길까

### 모델은 "확률"로 답한다

LLM은 현재 컨텍스트에서 가장 그럴듯한 다음 토큰을 생성합니다. 즉, 여러분 팀의 내부 규칙을 자동으로 아는 게 아닙니다.

### 자연어 지시만으로는 모호하다

"우리 브랜드 색상 써줘"는 사람에게도 애매합니다. 모델에게는 더 애매합니다. 구체적인 키와 값, 의미 매핑이 필요합니다.

### 문장형 가이드는 강제력이 약하다

위키에 "8px 스케일 사용"이라고 써 있어도 코드 생성 단계에서 항상 지켜지지 않습니다. 구조화된 데이터가 필요합니다.

### 컨텍스트 윈도우는 한정되어 있다

토큰 정보를 매번 프롬프트에 길게 붙여 넣으면 기능 요구사항과 충돌합니다. 결국 누락되고, 누락된 부분은 모델이 추측합니다.

## 디자인 토큰은 AI를 위한 제약 인터페이스다

디자인 토큰은 사람이 보기 위한 문서가 아니라, 사람이 만든 디자인 시스템 결정을 기계가 재사용할 수 있게 만든 계약입니다.

좋은 토큰 구조는 다음을 포함합니다.

- 안정적인 의미 기반 이름
- 명시적인 값
- 라이트/다크 같은 테마 매핑
- 컴포넌트 단위 의미 토큰

이 계약이 있으면 AI는 "창작"보다 "조합"에 집중하게 되고, hallucination이 크게 줄어듭니다.

## 토큰 없는 워크플로우 vs 토큰 기반 워크플로우

### 토큰 없는 경우

1. AI에게 컴포넌트 생성 요청
2. 그럴듯하지만 어긋난 코드 수신
3. 사람이 토큰 규칙에 맞게 수동 교정
4. 다음 요청에서 다시 반복

결과:

- 리뷰 시간이 길어짐
- 코드 일관성 저하
- AI 활용 효율 하락

### 토큰 기반인 경우

1. AI가 디자인 토큰과 컴포넌트 계약을 조회
2. 의미 토큰 중심으로 코드 생성
3. 사람은 스타일 교정 대신 UX/로직 검토에 집중

결과:

- 일관성 향상
- 리뷰 비용 감소
- 생산성 증가

## AI 품질을 높이는 최소 토큰 스키마

처음부터 거대하게 설계할 필요는 없습니다. 아래 정도면 체감 효과가 큽니다.

### 색상 토큰

- `color.text.primary`, `color.text.secondary`
- `color.surface.default`, `color.surface.elevated`
- `color.border.default`, `color.border.focus`
- `color.intent.success`, `color.intent.warning`, `color.intent.danger`

### 간격/사이즈 토큰

- `spacing.1` ~ `spacing.10`
- `radius.sm`, `radius.md`, `radius.lg`
- `size.control.sm`, `size.control.md`, `size.control.lg`

### 타이포그래피 토큰

- `font.size.body`, `font.size.label`, `font.size.title`
- `font.weight.regular`, `font.weight.medium`, `font.weight.semibold`
- `lineHeight.body`, `lineHeight.heading`

### 컴포넌트 의미 토큰

- `button.primary.bg`
- `button.primary.fg`
- `input.border.default`
- `input.border.focus`

이 정도만 갖춰도 임의 값 생성이 눈에 띄게 줄어듭니다.

## AI에 토큰을 전달하는 방법

### 방법 1: 프롬프트에 매번 직접 붙여넣기

장점은 즉시 시작 가능하다는 점입니다. 단점은 누락, 중복, 최신성 문제입니다.

### 방법 2: 저장소 내 토큰 파일 참조

직접 붙여넣기보다 낫지만, 여전히 프롬프트 습관에 의존합니다.

### 방법 3: MCP 같은 프로토콜 기반 조회

AI가 생성 시점에 최신 토큰을 가져옵니다. 팀 단위 운영과 장기 유지보수에 가장 유리합니다.

## 토큰만으로는 부족한 이유: 보조 가드레일 필요

토큰을 도입해도 100% 자동으로 해결되지는 않습니다. 아래 장치를 함께 두는 것이 안전합니다.

### 가드레일 1: 프리미티브 API 제약

지원하지 않는 변형/사이즈는 타입 레벨에서 막아야 합니다.

### 가드레일 2: 린트 규칙

하드코딩 색상, 임의 간격 값을 잡아내는 자동 검사를 추가하세요.

### 가드레일 3: 시각 회귀 테스트

빠른 반복에서 생기는 미세한 변화는 사람이 놓치기 쉽습니다.

### 가드레일 4: 프롬프트 템플릿 표준화

"기존 컴포넌트만 사용, 토큰만 사용, 변형 발명 금지" 같은 문구를 팀 템플릿으로 고정하세요.

## FramingUI가 주는 실전 이점

FramingUI는 토큰 중심 디자인 시스템과 AI 코드 생성 워크플로우를 연결하는 데 유리합니다.

- 의미 토큰 구조를 정리하기 쉽고
- 프리미티브와 토큰 계약을 맞추기 편하며
- AI 도구에서 토큰 맥락을 활용하는 과정이 자연스럽습니다

결과적으로 "AI가 빨리 만들고 사람이 다 고치는" 패턴에서 벗어나는 데 도움이 됩니다.

## 바로 적용할 수 있는 5단계

### 1단계: 현재 생성 코드 위반률 측정

토큰 미사용, 존재하지 않는 변형 사용 비율을 먼저 확인하세요.

### 2단계: 최소 토큰 스키마 확정

색상/간격/타입/핵심 컴포넌트 의미 토큰부터 시작합니다.

### 3단계: AI가 읽을 수 있는 형태로 노출

정적 파일 참조든 MCP든, 최신 값을 안정적으로 조회할 수 있게 만드세요.

### 4단계: 자동 검증 추가

토큰 위반 코드는 CI 단계에서 빠르게 차단합니다.

### 5단계: 반복되는 오류를 토큰 설계로 흡수

예외를 코드에서 매번 처리하지 말고, 토큰 의미를 확장해 구조적으로 해결하세요.

## 결론

AI UI 코드의 hallucination은 모델 성능 문제만이 아닙니다. 대부분은 디자인 시스템 컨텍스트가 빠져서 생기는 구조적 문제입니다.

디자인 토큰은 단순 스타일 변수 모음이 아니라, AI가 여러분 제품 규칙 안에서 코드를 생성하도록 만드는 제약 장치입니다.

AI 코드 생성을 실제 제품 개발에 안정적으로 쓰고 싶다면, 토큰 기반 디자인 시스템은 선택이 아니라 기본 조건입니다.

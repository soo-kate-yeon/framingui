Figma에서 복사-붙여넣기는 빠르게 느껴져요. 실제로 빠르기도 해요. 브랜드 메인 색상을 바꿔야 하는 시점까지는요. 그 색이 47개 파일에 흩어져 있다는 걸 발견하기 전까지는요.

Inspect 패널에서 CSS 값을 직접 복사하는 문제는, 숫자를 가져오지 의미를 가져오지 않는다는 거예요. `#3B82F6`은 이 색이 왜 쓰이는지 아무것도 말해주지 않아요. `24px`는 그게 카드 패딩인지, 섹션 간격인지, 아이콘 크기인지 설명하지 않고요. 하드코딩된 값은 조용히 쌓이고, 코드베이스는 서서히 디자인에서 멀어져요.

## 복사-붙여넣기가 실제로 치르는 비용

컴포넌트 하나에 값을 복사해서 CSS에 붙여넣는 데 20분이면 될 수 있어요. 효율적으로 느껴지죠. 숨은 비용은 나중에 나타나요.

**리브랜딩 작업.** 브랜드 색상을 바꾸려면 그 값을 하드코딩한 모든 파일에서 찾아서 바꿔야 해요. 하나라도 놓치면 시각적으로 깨져요.

**다크 모드.** 하드코딩된 색상 하나하나마다 다크 테마용 대응 값을 수동으로 만들어야 해요. 컴포넌트가 늘어날수록 목록이 길어지고요.

**디자인 드리프트.** 개발자 A는 `12px` 패딩을 쓰고, 개발자 B는 "비슷해 보이니까" `16px`를 써요. 6개월 후에는 일관된 게 하나도 없어요.

**유지보수.** 1년 뒤에 `#3B82F6`이 PR에 나타나면, 그게 primary 색인지 accent인지 레거시 잔재인지 아무도 기억하지 못해요.

## 시맨틱 네이밍으로의 전환

디자인 토큰은 값과 의미의 관계를 뒤집어요. "이 값이 무엇인가?"가 아니라 "이 값이 무슨 역할을 하는가?"를 묻게 되는 거예요.

`--color-primary`라는 이름의 토큰은 헥스값이 바뀌어도 그것을 참조하는 모든 곳을 깨뜨리지 않아요. 이름이 의도를 담고 있거든요. 브랜드 색이 파란색에서 보라색으로 바뀔 때는 이렇게 되는 거죠.

```css
/* 이전 */
--color-primary: #3B82F6;

/* 이후 */
--color-primary: #9333EA;
```

모든 버튼, 링크, 포커스 상태가 즉시 업데이트돼요. 이름은 외관이 아니라 역할을 설명하기 때문에 여전히 의미가 있어요.

`--color-blue-500`을 `--color-purple-500`으로 이름 바꾸는 것과 비교해보면, 어차피 모든 참조를 찾아서 업데이트해야 하니 토큰을 쓰는 의미가 없어요.

## FramingUI가 토큰을 사용하는 방식

FramingUI는 디자인 토큰을 컴포넌트 API에 직접 내장해요. `variant="primary"`로 `Button` 컴포넌트를 사용하면 내부적으로 올바른 토큰이 적용돼요.

```css
.button-primary {
  background: var(--bg-primary-default);
  color: var(--fg-on-primary);
  border-radius: var(--radius-md);
  padding: var(--spacing-3) var(--spacing-6);
}
```

어떤 요소에 어떤 토큰이 쓰이는지 찾아볼 필요가 없어요. 컴포넌트가 이미 알고 있거든요. 토큰 값을 한 곳에서 바꾸면 애플리케이션의 모든 버튼이 업데이트를 반영해요.

테마 전환도 같은 방식으로 작동해요. 토큰은 다른 data 속성 아래에서 다른 값으로 해석돼요.

```css
:root {
  --bg-primary-default: #3B82F6;
  --bg-canvas: #FFFFFF;
}

[data-theme="dark"] {
  --bg-primary-default: #60A5FA;
  --bg-canvas: #1F2937;
}
```

컴포넌트 코드는 바뀌지 않아요. CSS 변수가 활성 테마에 따라 다르게 해석되는 거예요.

## 기존 코드베이스 마이그레이션

이미 하드코딩된 값으로 가득한 코드베이스가 있다면 전면 재작성은 필요 없어요. 점진적으로 이전할 수 있거든요.

일관성 영향이 가장 큰 색상부터 시작해요.

```bash
# 하드코딩된 헥스 색상 전부 찾기
rg '#[0-9A-Fa-f]{6}' --type css --type tsx
```

값들을 의도별로 묶어요. `#3B82F6`이 12군데에 나타나고 항상 "주요 액션"을 의미한다면, 그게 `--color-primary` 토큰이에요. 토큰을 정의하고, 컴포넌트 하나씩 하드코딩된 값을 교체해나가면 돼요.

가장 많이 쓰이는 컴포넌트(버튼, 인풋, 카드)부터 시작해요. 하나씩 마이그레이션할수록 미래의 유지보수 부담이 줄어들어요. 타이포그래피와 간격도 같은 패턴이에요. 반복되는 값을 찾고, 시맨틱하게 이름 짓고, 리터럴을 교체해요.

## 복사-붙여넣기가 여전히 맞는 경우

디자인 토큰이 모든 상황에서 정답은 아니에요. 독특한 아트 디렉션의 일회성 마케팅 페이지, 제어할 수 없는 외부 CSS, 빠르게 버려질 프로토타입은 토큰 시스템의 오버헤드가 맞지 않아요.

기준은 수명과 재사용이에요. 코드가 유지보수될 것이고, 여러 개발자가 손댈 것이고, 테마나 플랫폼을 넘어서 작동해야 한다면—토큰은 금세 본전을 뽑아요. 하루 실험으로 버려질 코드라면 값을 그냥 복사해도 괜찮아요.

프로덕트 UI, 컴포넌트 라이브러리, 몇 달이나 몇 년에 걸쳐 진화해야 하는 코드베이스가 바로 토큰 접근법이 복리로 가치를 쌓는 곳이에요. 초기 비용은 이름을 정의하는 일이에요. 계속되는 혜택은 의도가 코드 자체에 담기고 변경이 자동으로 전파된다는 거고요.

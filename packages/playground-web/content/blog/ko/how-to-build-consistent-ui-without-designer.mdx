## 혼자 만들수록 UI 일관성은 더 쉽게 무너진다

솔로 개발자나 인디해커가 UI를 망치는 이유는 감각이 없어서가 아닙니다. 대부분은 속도 때문입니다. 기능을 붙이느라 바쁘고, 오늘 급한 화면부터 완성하다 보면 버튼 하나도 페이지마다 다르게 생깁니다.

처음에는 별문제 없어 보입니다. 그런데 기능이 늘어날수록 어색함이 커집니다. 같은 동작인데 모양이 다르고, 같은 경고인데 색이 다르고, 같은 입력창인데 간격이 다릅니다. 사용자 입장에서는 "뭔가 불안한 서비스"처럼 느껴집니다.

좋은 소식은 분명합니다. 전담 디자이너가 없어도 해결할 수 있습니다. 핵심은 감으로 만드는 UI에서 규칙 기반의 디자인 시스템으로 넘어가는 것입니다.

## 일관된 UI의 본질: 예쁘게가 아니라 예측 가능하게

일관성은 화면을 똑같이 복제하는 것이 아닙니다. 어떤 화면에서도 사용자가 예상 가능한 방식으로 동작하게 만드는 것입니다.

### 시각적 일관성

- 색상 의미가 고정되어 있다: 기본 액션, 성공, 경고, 위험
- 간격이 정해진 스케일 안에서만 쓰인다
- 타이포그래피 위계가 반복된다

### 상호작용 일관성

- 버튼 상태(기본, hover, disabled, loading)가 동일한 규칙으로 동작한다
- 폼 에러 표시 방식이 모든 페이지에서 같다
- 로딩/빈 상태/에러 상태가 같은 패턴으로 제공된다

### 구조적 일관성

- 페이지 헤더, 카드, 폼 섹션 같은 반복 블록이 재사용된다
- 같은 문제를 같은 UI 구조로 푼다

이 세 가지가 맞물리면 화면이 화려하지 않아도 "정돈된 제품"으로 보입니다.

## 디자이너 없이도 되는 4단계 구조

실무에서 가장 잘 먹히는 순서는 아래 네 단계입니다.

### 1단계: 기반값 정의 (디자인 토큰)

디자인 토큰은 디자인 시스템의 최소 단위입니다. 색상, 간격, 폰트 크기, 라운드, 그림자 같은 값이 여기에 해당합니다.

토큰이 없으면 컴포넌트마다 임의의 값이 들어갑니다. 결국 비슷한 색상이 5개, 비슷한 간격이 7개가 됩니다.

이름은 의미 중심으로 정하세요.

- `color.text.primary`
- `color.surface.elevated`
- `spacing.4`
- `radius.md`

`blue500` 같은 팔레트 이름을 컴포넌트에서 직접 쓰기 시작하면 확장이 어렵습니다.

### 2단계: 프리미티브 컴포넌트 정리

프리미티브는 비즈니스 로직이 없는 기본 UI 조각입니다.

- `Button`
- `Input`
- `Select`
- `Card`
- `Badge`

여기서 중요한 건 "옵션을 줄이는 것"입니다. 예를 들어 버튼은 `primary`, `secondary`, `ghost`, `danger` 정도면 충분합니다. 커스텀 옵션을 무제한으로 열어두면 일관성은 바로 깨집니다.

### 3단계: 반복 패턴 추출

대부분의 서비스는 비슷한 화면을 계속 만듭니다.

- 페이지 헤더
- 필터 바
- 빈 상태
- 폼 섹션
- 테이블 툴바

두 번 이상 나온 구조는 패턴으로 추출하세요. 패턴이 늘수록 새 화면 만들 때 디자인 결정이 줄어듭니다.

### 4단계: 사용 규칙 문서화

문서는 길 필요 없습니다. 한 페이지면 충분합니다.

- 언제 primary 버튼을 쓰는지
- 폼 간격 기본값은 얼마인지
- 에러 메시지 위치와 색상은 무엇인지
- 페이지 제목 위계는 어떻게 가져가는지

이 규칙이 있어야 "사람마다 다르게"가 아니라 "팀 규칙대로"가 됩니다.

## 7일 안에 기존 UI를 정리하는 실행 플랜

기존 프로젝트에 적용할 때는 전면 개편보다 점진 이관이 낫습니다.

### Day 1: UI 실사

핵심 화면 캡처를 모아서 다음을 셉니다.

- 버튼 스타일 개수
- 입력창 스타일 개수
- 본문 글자 크기 개수
- 카드 패딩 값 개수

숫자를 먼저 보면 어디가 가장 심각한지 명확해집니다.

### Day 2: 토큰 최소 세트 확정

처음부터 거창하게 하지 마세요.

- 간격 8~10단계
- 텍스트/배경/보더 의미 색상
- 라운드 3단계
- 본문 중심 타이포그래피 3~4단계

이 정도면 대부분 화면을 커버합니다.

### Day 3: 버튼 통일

가장 많이 쓰는 `Button`부터 정리합니다. 기존 화면을 전부 바꾸지 말고, 신규 코드부터 토큰 기반 버튼만 쓰게 만드세요.

### Day 4: 폼 컴포넌트 정리

`Input`, `Select`, `Textarea`, `Label`, `FieldError`를 한 세트로 묶어 통일합니다. 폼은 사용자 체감이 커서 개선 효과가 바로 드러납니다.

### Day 5: 페이지 패턴 적용

`PageHeader`, `SectionCard`, `EmptyState` 같은 공통 패턴을 만들고 실제 화면 두세 군데에 먼저 적용해 봅니다.

### Day 6: 중요 화면 우선 이관

트래픽이나 매출에 직접 영향 있는 화면부터 바꾸세요. 온보딩, 결제, 설정, 핵심 대시보드 순서가 보통 효율적입니다.

### Day 7: 규칙 잠금

코드리뷰 기준과 린트 규칙을 추가합니다.

- 하드코딩 색상 금지
- 임의 간격 금지
- 신규 UI는 디자인 시스템 컴포넌트 우선 사용

여기까지 오면 일관성이 유지되는 구조가 생깁니다.

## 빠르게 개발하면서도 무너지지 않게 만드는 운영법

### 기본값 중심 API를 만든다

좋지 않은 API:

```tsx
<Button bgColor="..." textColor="..." px={...} py={...} radius={...}>
```

좋은 API:

```tsx
<Button variant="primary" size="md">프로젝트 만들기</Button>
```

기본값이 강할수록 개발 속도가 빨라지고 결과도 안정적입니다.

### 예외는 예외로 관리한다

완전 금지는 현실적이지 않습니다. 대신 `unstyled` 같은 탈출구를 두되, 왜 필요한지 PR에서 설명하게 하세요.

### 드리프트를 수치로 본다

매달 한 번만 체크해도 충분합니다.

- 하드코딩 hex 색상 개수
- 토큰 밖 간격 값 개수
- 컴포넌트 변형 종류 증가 추이

숫자가 오르면 시스템이 새고 있다는 신호입니다.

## FramingUI가 특히 유용한 지점

솔로 개발자는 방향은 알아도 구현과 운영에서 막히는 경우가 많습니다. FramingUI는 그 구간을 줄여줍니다.

- 디자인 시스템의 토큰/컴포넌트 구조를 빠르게 잡을 수 있고
- AI 코드 생성 워크플로우와 연결해 토큰 기반 결과를 얻기 쉬우며
- 반복적인 스타일 정리 시간을 크게 줄일 수 있습니다

결국 "예쁘게 만들기"보다 "지속 가능한 일관성"에 집중하게 해줍니다.

## 자주 실패하는 패턴

### 1. 토큰 없이 컴포넌트부터 만든다

처음에는 빨라 보이지만, 결국 값이 제각각이 됩니다.

### 2. 변형을 너무 많이 만든다

버튼 종류가 너무 많으면 누구도 규칙을 기억하지 못합니다.

### 3. 이관 계획 없이 새 시스템만 만든다

기존 화면이 그대로면 사용자는 개선을 느끼지 못합니다.

### 4. 일관성을 미적 문제로만 본다

일관성은 개발 생산성과도 직결됩니다. 재작업, 리뷰 시간, QA 비용을 줄입니다.

## 마무리 체크리스트

아래 다섯 가지가 되면 실전에서 충분히 굴러갑니다.

- 색상/간격/타입/라운드 토큰이 있다
- 핵심 프리미티브가 토큰만 사용한다
- 반복 패턴이 컴포넌트로 추출되어 있다
- 사용 규칙이 짧게라도 문서화되어 있다
- 하드코딩을 잡는 리뷰/린트가 있다

디자이너가 없어서 못 만드는 게 아닙니다. 규칙이 없어서 반복적으로 무너지는 것입니다. 작은 디자인 시스템부터 시작하면, 혼자 개발해도 UI는 충분히 안정적으로 성장합니다.

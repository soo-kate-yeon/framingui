UI 일관성이 무너지는 이유는 감각이 없어서가 아니에요. 결정보다 빠르게 만들어서예요. 1주차에 대시보드, 2주차에 온보딩, 3주차에 설정 페이지를 만들다 보면 어느 날 같은 버튼이 다섯 가지 스타일로 존재하게 돼요.

디자이너가 없어도 해결할 수 있어요. 필요한 건 강제되는 기본값이 있는 작은 디자인 시스템이에요.

## 일관성이 실제로 요구하는 것

일관된 UI는 모든 화면을 똑같이 만드는 게 아니에요. 사용자가 같은 문제를 같은 패턴으로 만나게 하는 거예요.

이건 세 레이어로 나뉘어요. 시각적 일관성은 색상이 고정된 의미에 매핑되고(기본, 성공, 경고, 위험), 간격이 하나의 스케일을 따르고, 타이포그래피 위계가 반복되는 것이에요. 상호작용 일관성은 버튼이 모든 화면에서 같은 방식으로 반응하고, 입력창이 검증과 포커스를 같은 방식으로 보여주고, 로딩과 에러 상태가 공유된 규칙을 따르는 거예요. 구조적 일관성은 페이지 헤더, 카드, 폼, 섹션 블록이 매번 새로 만들어지지 않고 재사용되는 거예요.

이 세 레이어가 유지되면 커스텀 비주얼 디자인 없이도 제품이 의도적으로 보여요.

## 4단계 구조

순서대로 쌓아야 해요. 레이어를 건너뛰면 나중 레이어를 유지하기 어려워져요.

### 1단계: 디자인 토큰

토큰은 시스템의 가장 작은 재사용 가능한 값이에요. 색상, 간격, 테두리 반경, 타이포그래피 크기, 그림자 값이 여기에 속해요.

토큰 없이는 새 컴포넌트마다 새로운 임의의 결정이 시작돼요. 거의 비슷한 회색이 세 가지, 거의 겹치는 간격 값이 일곱 개가 생기죠.

처음부터 의미 기반 이름을 쓰세요.

- `gray-900` 대신 `color.text.primary`
- `white` 대신 `color.background.surface`
- `16px` 대신 `spacing.4`
- `6px` 대신 `radius.md`

컴포넌트가 팔레트 값 대신 의미 이름을 사용하면, 색상을 바꿀 때 컴포넌트 파일 40개를 뒤지는 게 아니라 토큰 정의 하나를 수정하는 일이 돼요.

### 2단계: 프리미티브

프리미티브는 제품 로직이 없는 기본 UI 조각이에요. `Button`, `Input`, `Select`, `Card`, `Badge`. 프리미티브 설계의 핵심 원칙은 옵션 제한이에요.

`primary`, `secondary`, `ghost`, `danger` 변형을 가진 Button이 거의 모든 실제 케이스를 커버해요. 색상과 패딩을 독립적인 props로 받는 Button은 아무것도 커버하지 못해요 — 그냥 추가 단계가 붙은 styled div거든요.

강한 기본값은 모든 개발자가 규칙을 기억하지 않아도 일관된 결과를 내게 해줘요.

### 3단계: 반복 패턴

패턴은 프리미티브를 자주 쓰는 제품 구조로 결합해요. 필터 바, 폼 섹션, 빈 상태, 데이터 테이블 툴바, 인증 페이지 레이아웃. 실제 제품에서 시각적 불일관성 대부분이 이 레벨에서 생겨요. 추출된 패턴을 재사용하는 대신 공통 섹션을 매번 새로 만들기 때문이에요.

규칙은 단순해요. UI 구조가 두 번 나타나면 추출하세요.

### 4단계: 사용 규칙

규칙은 결정 피로를 없애는 문서화된 결정이에요. 40페이지 핸드북이 필요 없어요. 한 페이지면 충분해요.

- 언제 primary 대 secondary 버튼을 쓰는지
- 폼 요소 사이의 표준 간격
- 페이지 제목 위계
- 에러 메시지 위치와 색상

이 페이지 하나가 각 개발자가 같은 질문을 독립적으로 다시 결정하는 것을 막아요. 거기서 시각적 드리프트가 시작되거든요.

## 7일 안에 기존 UI를 정리하는 플랜

출시를 멈추지 않고 이 시스템을 기존 제품에 이식할 수 있어요.

**Day 1 — 실사.** 주요 페이지를 캡처하고 세요. 버튼 스타일 종류, 입력창 스타일 종류, 본문 폰트 크기 종류, 카드 패딩 값 종류. 숫자가 불일관성이 가장 심한 곳을 알려줘요.

**Day 2 — 토큰 최소 세트 확정.** 간격 8-10단계, 의미 텍스트 색상 5-7개, 의미 배경/보더 색상 5-7개, 라운드 3단계, 앱 UI용 폰트 크기 3-4개. 작게 유지하세요. 나중에 확장할 수 있어요.

**Day 3 — 버튼 리팩터링.** 가장 많이 쓰는 프리미티브부터 시작해요. 새 코드에서 임의의 유틸리티 조합 대신 토큰 기반 변형을 써요. 아직 전체 이관을 목표로 하지 않아도 돼요.

**Day 4 — 폼 스택.** `Input`, `Textarea`, `Select`, `Label`, `FieldError`를 한 세트로 리팩터링해요. 폼은 불일관성을 크게 드러내고, 개선 효과가 사용자에게 바로 보여요.

**Day 5 — 레이아웃 패턴.** `PageHeader`, `SectionCard`, `EmptyState` 컴포넌트를 만들고 실제 화면 두 곳에 적용해봐요.

**Day 6 — 트래픽 많은 화면 우선.** 사용자 노출이 가장 많은 화면을 먼저 건드려요. 전체 이관이 아니라 가시적 임팩트를 우선시해요.

**Day 7 — 규칙 잠금.** 짧은 사용 규칙 문서와 두 가지 강제 메커니즘을 추가해요. 하드코딩된 hex 색상을 잡는 린트 규칙, 그리고 토큰 사용 여부를 확인하는 PR 체크리스트. 강제 없는 규칙은 몇 주 안에 무너져요.

## 출시 압박 아래서 일관성 유지하기

기본값이 충분히 강하면 속도와 일관성은 반대가 아니에요.

핵심 API 결정은 props 대신 변형이에요. `variant="primary"`를 받는 `Button`은 압박 아래서도 일관된 결과를 만들어요. `bgColor`, `textColor`, `px`, `py`, `radius`를 독립적인 props로 받는 `Button`은 누군가 급할 때마다 드리프트를 초대해요.

탈출구는 남겨두세요 — `unstyled` prop이나 `className` 오버라이드 — 하지만 PR에서 설명을 요구하게 해요. 이게 우발적인 엔트로피 없이 유연성을 유지하는 방법이에요.

한 달에 한 번 간단한 체크 세 가지를 해요. JSX의 하드코딩된 hex 색상 수, 토큰 스케일 밖 간격 값 수, 실제로 쓰이는 컴포넌트 변형 수. 이 숫자가 꾸준히 올라가면 시스템이 새고 있다는 신호예요.

## FramingUI가 유용한 지점

원칙은 알아도 구현과 운영에서 막히는 경우가 많아요. FramingUI는 토큰과 컴포넌트 구조를 빠르게 잡을 수 있게 해줘요. AI 코딩 도구와 연결하면 생성된 UI가 임의의 기본값 대신 토큰 규칙을 따르게 돼서 수동 정리 시간이 줄어들고요.

유용한 조합은 무엇을 만들어야 할지 아는 것(4단계 구조)과 첫 레이어를 빠르게 세울 수 있는 도구를 갖는 거예요. 강제 메커니즘이 있으면 그 이후엔 시스템이 스스로 유지돼요.

처음 방문부터 파워 유저 워크플로우까지 일관된 제품은 디자인 팀이 필요한 게 아니에요. 일관되게 적용된 작고 강제된 결정이 필요한 거예요.

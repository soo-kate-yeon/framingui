h1 하나에 미디어 쿼리 다섯 개가 붙어 있다면, 그건 코드가 아니라 기술 부채예요. 그 h1이 전체 디자인 시스템의 모든 텍스트 요소에 곱해지면, 건드리기 두려운 수백 개의 `@media` 블록이 남습니다.

`clamp()`와 디자인 토큰의 조합이 이 문제를 풉니다. 폰트 사이즈가 최솟값과 최댓값 사이에서 연속으로 스케일되고, 토큰 하나가 미디어 쿼리 스택 전체를 대체해요.

## 브레이크포인트 방식이 무너지는 이유

전형적인 반응형 h1은 이렇게 생겼어요:

```css
h1 { font-size: 32px; }

@media (min-width: 640px)  { h1 { font-size: 36px; } }
@media (min-width: 768px)  { h1 { font-size: 40px; } }
@media (min-width: 1024px) { h1 { font-size: 48px; } }
@media (min-width: 1536px) { h1 { font-size: 56px; } }
```

h1 하나에 브레이크포인트 다섯 개. 그리고 각 브레이크포인트를 넘을 때 텍스트는 점프해요 — 자연스럽게 흐르는 게 아니라 딱딱 끊기는 거죠. 900px 뷰포트 사용자는 768px와 1024px 정의 사이 어딘가에 어정쩡하게 낀 상태가 됩니다.

이걸 모든 헤딩 레벨, 본문, 캡션에 적용하면... 결국 누구도 손대고 싶지 않은 코드베이스가 남아요.

## clamp()가 하는 일

`clamp()`는 세 값을 받아요: 최솟값, 선호값(뷰포트 상대), 최댓값.

```css
font-size: clamp(최솟값, 선호값, 최댓값);
```

선호값은 `vw` 단위를 써서 뷰포트 너비에 따라 스케일돼요. 최솟값과 최댓값이 양 끝 가드 역할을 하고, 그 사이에서 폰트 사이즈가 선형으로 커집니다.

h1이 640px에서 2rem, 1536px에서 3rem이어야 한다면 계산은 이렇게:

```
기울기 = (3 - 2) / (1536 - 640) = 1 / 896 ≈ 1.786vw

Y절편 = 2 - (1.786 × 640 / 100) ≈ 1.286rem
```

결과:

```css
font-size: clamp(2rem, 1.786vw + 1.286rem, 3rem);
```

640px~1536px 사이에서 부드럽게 스케일되고, 그 아래에선 2rem, 그 위에선 3rem으로 고정돼요. 미디어 쿼리 필요 없습니다.

한 가지 실용적인 팁: 최솟값과 최댓값에 `px` 말고 `rem`을 쓰세요. 브라우저 기본 폰트 사이즈를 변경한 사용자의 설정을 `px` 기반 clamp는 무시해버리거든요. WCAG에서 요구하는 사항이기도 해요.

## 유체 값을 디자인 토큰으로 저장하기

진짜 레버리지는 clamp 공식을 토큰으로 만드는 데서 나와요. 한 번 정의하고, CSS 변수를 생성하고, 어디서든 변수를 참조하는 거죠.

주요 헤딩 레벨을 커버하는 토큰 파일:

```json
{
  "typography": {
    "fontSize": {
      "h1": { "value": "clamp(2rem, 1.786vw + 1.286rem, 3rem)" },
      "h2": { "value": "clamp(1.75rem, 1.339vw + 1.161rem, 2.5rem)" },
      "h3": { "value": "clamp(1.5rem, 0.893vw + 1.107rem, 2rem)" },
      "h4": { "value": "clamp(1.25rem, 0.446vw + 1.054rem, 1.5rem)" },
      "body-lg": { "value": "clamp(1rem, 0.223vw + 0.929rem, 1.125rem)" },
      "body": { "value": "1rem" }
    }
  }
}
```

본문과 UI 레이블은 고정으로 유지해요. 유체 스케일링은 헤딩과 큰 디스플레이 텍스트에 가장 효과적이거든요. 읽기 크기의 본문 텍스트는 스케일해도 얻는 게 별로 없어요.

## 생성 결과물

토큰 빌드 단계에서 위 정의들이 CSS 변수와 Tailwind 유틸리티로 변환돼요:

```css
:root {
  --font-size-h1: clamp(2rem, 1.786vw + 1.286rem, 3rem);
  --font-size-h2: clamp(1.75rem, 1.339vw + 1.161rem, 2.5rem);
  --font-size-h3: clamp(1.5rem, 0.893vw + 1.107rem, 2rem);
  --font-size-body-lg: clamp(1rem, 0.223vw + 0.929rem, 1.125rem);
}
```

```js
// tailwind.config.js (생성됨)
module.exports = {
  theme: {
    fontSize: {
      'h1': 'var(--font-size-h1)',
      'h2': 'var(--font-size-h2)',
      'h3': 'var(--font-size-h3)',
      'body-lg': 'var(--font-size-body-lg)',
    }
  }
}
```

이 시스템을 쓰는 컴포넌트에는 반응형 클래스가 전혀 필요 없어요:

```tsx
export function Hero({ title, subtitle }) {
  return (
    <section>
      <h1 className="text-h1 font-bold leading-tight">{title}</h1>
      <p className="text-body-lg text-text-secondary mt-4">{subtitle}</p>
    </section>
  );
}
```

브레이크포인트 버전과 비교해보세요: `text-3xl md:text-4xl lg:text-5xl xl:text-6xl`. 토큰 버전이 더 짧고, 읽기 쉽고, 브레이크포인트 사이에서도 더 정확해요.

## 줄 간격과 자간도 함께 스케일하기

폰트 사이즈만 스케일할 필요는 없어요. 줄 간격도 작은 크기에서는 약간 촘촘하게, 큰 크기에서는 여유 있게 가져가는 게 좋고, 헤딩의 자간은 텍스트가 커질수록 더 좁아지는 게 자연스럽거든요.

```json
{
  "typography": {
    "h1": {
      "fontSize": { "value": "clamp(2rem, 1.786vw + 1.286rem, 3rem)" },
      "letterSpacing": {
        "value": "clamp(-0.02em, -0.005vw + -0.017em, -0.01em)",
        "description": "큰 사이즈에서 더 촘촘한 자간"
      }
    },
    "body": {
      "fontSize": { "value": "1rem" },
      "lineHeight": {
        "value": "clamp(1.4, 0.05vw + 1.37, 1.6)",
        "description": "데스크탑에서 약간 더 여유 있는 줄 간격"
      }
    }
  }
}
```

비용 대비 타이포그래피 품질을 크게 높이는 작업이에요.

## 흔한 실수들

**최솟값과 최댓값에 px 사용하기.** 사용자가 브라우저 기본 폰트를 20px로 설정했다면, `clamp(32px, ...)` 헤딩은 그 설정을 완전히 무시해요. `clamp(2rem, ...)`을 써야 사용자 설정이 반영돼요.

**기울기를 너무 가파르게 설정하기.** `clamp(1rem, 10vw, 4rem)`은 500px 뷰포트에서 `10vw = 50px`이니까 이미 최댓값을 초과해요. 넓은 구간에서 4rem에 박혀 있다가 좁은 구간에서 갑자기 뚝 떨어지게 됩니다. 본문 크기 텍스트는 1vw~3vw 사이를 권장해요.

**모든 텍스트 스타일에 유체 스케일링 적용하기.** 버튼 레이블, 뱃지 텍스트, 캡션 같은 UI 텍스트는 고정 크기가 더 명확해요. 1rem 미만 요소에 clamp를 써봤자 복잡도만 늘어나고 얻는 게 없어요. 헤딩과 디스플레이 텍스트에 집중하세요.

## 계산 도구 활용하기

기울기-절편 계산은 손으로 하면 실수하기 쉬워요. [Utopia](https://utopia.fyi/type/calculator/)는 최솟값, 최댓값, 최소 뷰포트, 최대 뷰포트를 입력하면 clamp 공식을 뽑아줘요. [Fluid Type Scale Calculator](https://www.fluid-type-scale.com/)는 슬라이더로 조정하면서 실시간 프리뷰를 볼 수 있고요. 둘 다 바로 쓸 수 있는 공식을 출력해줘요.

## AI 코드 생성과의 관계

AI 코딩 어시스턴트가 MCP 서버나 직접 파일 접근으로 디자인 토큰을 읽으면, 토큰 안의 `clamp()` 값을 문맥에서 파악해요. "가격 섹션을 헤딩과 설명으로 만들어줘"라는 프롬프트에서 `text-5xl`이나 `text-xl` 대신 `text-h1`, `text-body-lg`를 쓰는 컴포넌트가 나오죠.

토큰 없이는 AI가 자기가 학습 때 본 패턴 기반으로 Tailwind 고정 스케일(`text-5xl`은 딱딱한 3rem이에요)을 쓰고 브레이크포인트 수식어를 덧붙여요. 생성 때마다 여러분의 시스템에서 조금씩 멀어져요.

유체 타이포그래피 토큰은 그 기본값을 바꿔요. AI가 여러분의 스케일을 쓰고, 그 스케일이 반응형이고, 그래서 브레이크포인트가 필요 없는 거예요.

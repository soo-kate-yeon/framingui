버튼 두 개, 둘 다 "명도 50%"로 설정했어요. 노란 버튼은 눈부시게 밝고, 파란 버튼은 칙칙해 보여요. 눈으로 맞추다 보면 숫자가 엉망이 되고, 다음 사람은 왜 이렇게 됐는지 알 수 없죠.

이게 HSL의 문제예요. 명도 채널이 지각된 밝기와 일치하지 않아요. OKLCH는 이걸 근본부터 고치는데, 그래서 Tailwind v4, Radix Colors, shadcn/ui가 모두 여기로 넘어가고 있어요.

## OKLCH이 실제로 측정하는 것

OKLCH는 세 채널을 써요: **L** (명도, 0~1), **C** (채도, 선명도), **H** (색상, 0~360°).

```css
color: oklch(0.65 0.2 250);
/* 0.65 = 65% 밝기, 0.2 = 적당한 채도, 250 = 파랑 계열 */
```

다른 점은 L이 *지각적으로 균일*하다는 거예요. `L = 0.65`인 두 색은 색상(H)에 상관없이 눈에 같은 밝기로 보여요:

```css
oklch(0.65 0.2 60)   /* 노랑 */
oklch(0.65 0.2 250)  /* 파랑 */
/* 둘 다 지각적으로 같은 밝기 */
```

HSL에서 똘 다 "명도 50%"인 이 두 색과 비교해보세요:

```css
hsl(60, 100%, 50%)   /* 노랑 — 거의 하얀색처럼 보임 */
hsl(240, 100%, 50%)  /* 파랑 — 눈에 띄게 어둡게 보임 */
```

이 차이는 사람 눈이 노란 파장에 더 민감하기 때문이에요. HSL 명도 채널은 이걸 반영하지 않아요. OKLCH는 반영해요.

## OKLCH이 해결하는 세 가지 문제

**예측 가능한 색상 스케일.** 디자인 시스템에는 가장 밝은 것부터 가장 어두운 것까지 9~11단계 색상 램프가 필요해요. HSL에서 균등하게 명도를 낮추면 시각적으로 불균일한 점프가 생겨요 — 90%에서 70%로 내려가는 단계가 50%에서 30%로 내려가는 단계와 다르게 느껴지는 거죠. OKLCH에서는 같은 숫자 간격이 같은 지각적 간격을 만들어요:

```css
oklch(0.9 0.1 250)  /* 가장 밝음 */
oklch(0.7 0.1 250)
oklch(0.5 0.1 250)  /* 중간 */
oklch(0.3 0.1 250)
oklch(0.1 0.1 250)  /* 가장 어두움 */
```

각 단계가 이전보다 딱 한 단계 어둡게 느껴져요. 스케일 전체를 프로그래밍으로 생성하고 균형 잡혀 보일 것을 믿을 수 있어요.

**색상 전반에 걸친 일관된 채도.** HSL 채도는 색상에 따라 같은 의미가 아니에요. OKLCH에서 채도를 `0.2`로 설정하면 노랑, 파랑, 보라, 빨강 모두 시각적으로 비슷한 선명함을 가져요:

```css
oklch(0.65 0.2 60)   /* 노랑   — 같은 채도로 느껴짐 */
oklch(0.65 0.2 150)  /* 초록   — 같은 채도로 느껴짐 */
oklch(0.65 0.2 250)  /* 파랑   — 같은 채도로 느껴짐 */
oklch(0.65 0.2 330)  /* 마젠타 — 같은 채도로 느껴짐 */
```

채도 값 하나로 여러 액센트 컬러를 정의하고, 수동 조정 없이도 시각적으로 균형 잡힌 느낌을 줄 수 있어요.

**더 나은 그라디언트.** RGB와 HSL 그라디언트는 색상이 크게 회전할 때 탁한 중간 톤을 지나가는 경우가 많아요. OKLCH 그라디언트는 선명함을 유지해요:

```css
/* HSL에서 탁함 */
background: linear-gradient(90deg, hsl(0, 100%, 50%), hsl(240, 100%, 50%));

/* OKLCH에서 선명함 */
background: linear-gradient(90deg in oklch,
  oklch(0.6 0.25 30),
  oklch(0.6 0.25 270)
);
```

## OKLCH으로 마이그레이션하기

**기존 색상 변환.** [oklch.com](https://oklch.com/)에서 hex 값을 붙여넣으면 OKLCH로 변환돼요. [colorjs.io](https://colorjs.io/apps/convert/)는 일괄 변환을 지원해요.

```css
/* 이전 */
--primary: hsl(220, 90%, 56%);

/* 이후 */
--primary: oklch(0.6 0.22 250);
```

**프로그래밍으로 스케일 생성.** 기본 색상을 OKLCH로 잡으면 전체 램프 만들기가 산수예요:

```javascript
function generateScale(baseC, baseH) {
  const lightnesses = [0.95, 0.85, 0.75, 0.65, 0.55, 0.45, 0.35, 0.25, 0.15];
  return lightnesses.map((l, i) => ({
    step: i + 1,
    value: `oklch(${l} ${baseC} ${baseH})`,
  }));
}

const blue = generateScale(0.2, 250);
// blue[0].value = "oklch(0.95 0.2 250)"  ← 가장 밝음
// blue[4].value = "oklch(0.55 0.2 250)"  ← 중간
// blue[8].value = "oklch(0.15 0.2 250)"  ← 가장 어두움
```

**인터랙티브 상태에 color-mix 사용.** `color-mix()` 함수는 색공간을 존중해요. OKLCH 안에서 믹스하면 밝기를 조정하는 동안 색조가 안정적으로 유지돼요:

```css
:root { --primary: oklch(0.6 0.22 250); }

.button:hover  { background: color-mix(in oklch, var(--primary), black 10%); }
.button:active { background: color-mix(in oklch, var(--primary), black 20%); }
```

RGB 공간에서 믹스하면 색조가 예상치 못하게 변하는 경우가 많아요. OKLCH 공간에서 믹스하면 색조를 유지해요.

**브라우저 지원과 폴백.** Chrome 111+, Safari 15.4+, Firefox 113+이 OKLCH를 기본 지원해요 — 2026년 기준 사용자의 90% 이상을 커버해요. 더 오래된 브라우저를 위해서는 `@csstools/postcss-oklab-function` 플러그인이 RGB 폴백을 자동으로 추가해줘요:

```css
/* 입력 */
color: oklch(0.6 0.2 250);

/* PostCSS 출력 */
color: rgb(66, 135, 245);    /* 폴백 */
color: oklch(0.6 0.2 250);   /* 모던 */
```

## 스위치하지 말아야 할 때

브랜드 가이드라인에 특정 hex 값이 고정돼 있을 때는 OKLCH가 불편해요. hex를 OKLCH로 변환해서 똑같이 렌더링되는지 확인할 수 있지만, 프로그래밍으로 스케일을 생성하면 새 값이 나와서 승인이 필요해요. 브랜드 팀이 팔레트를 hex로 관리한다면 마이그레이션 전에 먼저 조율하세요.

인쇄 워크플로우는 CMYK를 써요. OKLCH는 디스플레이 색공간이에요.

## 디자인 토큰 워크플로우에서의 실질적 변화

OKLCH로 바꾸면 수동으로 정의하는 원시 색상이 줄어요. 기본 색상과 채도를 하나 정하고, 스케일을 생성하고, OKLCH 값을 토큰으로 저장해요. 각각 수동으로 조정된 9개 색조를 하드코딩하는 대신 생성 함수의 결과물을 저장하니까 토큰 파일이 더 작아져요.

감사하기도 쉬워요. 토큰을 보면 밝기(L), 선명도(C), 색상(H)이 바로 보여요. 컨버터 없이도 알 수 있어요. 스케일 변경도 숫자 하나 바꾸는 거지, 9개를 수동으로 조정하는 게 아니에요.

AI 도구로 UI를 생성하는 팀에게는: OKLCH의 색상 토큰이 모델에게 더 해석 가능해요. 채널이 시맨틱 의미를 가지거든요. `oklch(0.6 0.22 250)`은 명확하게 중간 밝기의 적당히 채도 있는 파랑이에요. 모델이 외부 조회 없이도 대비와 페어링을 추론할 수 있어요. 덕분에 생성된 색상 사용이 더 일관되게 나와요.

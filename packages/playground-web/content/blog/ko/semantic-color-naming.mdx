코드베이스에 파란색 계열이 47가지 있어요. 대부분은 그때그때 "좋아 보이는 것"을 고른 결과죠. 이제 브랜드 색상을 바꿔야 하는데, 빠진 것 없이 전부 찾을 수 있을지 걱정하면서 하루를 search-and-replace에 써야 할 것 같아요.

이게 리터럴 컬러 네이밍이 치르는 비용이에요. 해결책은 더 많은 기율이 아니라 다른 네이밍 시스템이에요.

## 리터럴 vs. 시맨틱 네이밍

리터럴 이름은 색상이 어떻게 생겼는지를 묘사해요. `blue-500`, `gray-700`, `#3B82F6`. 베이스 팔레트에서는 괜찮지만, 컴포넌트에서 직접 사용하는 순간 부채가 돼요.

시맨틱 이름은 색상이 무엇을 하는지를 묘사해요. `primary`, `text-secondary`, `surface-elevated`, `border-subtle`. 의도를 전달하죠. 컴포넌트에서 `bg-surface-elevated`를 읽으면 이게 상승된 서피스, 즉 모달, 팝오버, 플로팅 요소용이라는 걸 알아요. `bg-gray-50`을 읽으면 추측해야 하죠.

실용적인 차이: 리터럴 이름에서는 브랜드 색상을 바꾸면 이전 색상의 모든 인스턴스를 찾아서 교체해야 해요. 시맨틱 이름에서는 매핑 하나를 업데이트하면 모든 컴포넌트가 자동으로 바뀌어요.

## 2계층 시스템

가장 유지보수하기 좋은 구조는 원시 색상 값과 시맨틱 의미를 분리하는 거예요.

**1계층: 베이스 팔레트**

리터럴 이름을 가진 원시 색상 값이에요. 이게 구성 요소예요. 컴포넌트에서 직접 쓰지 않아요.

```css
:root {
  --palette-blue-400: #60A5FA;
  --palette-blue-500: #3B82F6;
  --palette-blue-600: #2563EB;
  --palette-red-500:  #EF4444;
  --palette-red-600:  #DC2626;
  --palette-gray-50:  #F9FAFB;
  --palette-gray-100: #F3F4F6;
  --palette-gray-900: #111827;
}
```

**2계층: 시맨틱 별칭**

1계층 값을 참조하는 역할 기반 이름이에요. 컴포넌트에서 사용하는 게 이거예요.

```css
:root {
  /* 브랜드 */
  --color-primary:       var(--palette-blue-500);
  --color-primary-hover: var(--palette-blue-600);

  /* 텍스트 */
  --color-text-primary:   var(--palette-gray-900);
  --color-text-secondary: var(--palette-gray-600);
  --color-text-disabled:  var(--palette-gray-400);

  /* 서피스 */
  --color-surface-base:     var(--palette-gray-50);
  --color-surface-card:     #FFFFFF;
  --color-surface-elevated: #FFFFFF;

  /* 보더 */
  --color-border-default: var(--palette-gray-200);
  --color-border-subtle:  var(--palette-gray-100);
  --color-border-strong:  var(--palette-gray-400);

  /* 피드백 */
  --color-danger:         var(--palette-red-500);
  --color-danger-hover:   var(--palette-red-600);
  --color-danger-surface: #FEF2F2;
}
```

파란색에서 보라색으로 리브랜딩할 때는 1계층을 업데이트해요.

```css
--palette-blue-500: #8B5CF6;  /* 이제 보라색 */
```

`--color-primary`를 참조하는 모든 것이 자동으로 바뀌어요. 컴포넌트 파일에서 바꿀 게 없어요.

## 수동 오버라이드 없이 다크 모드 구현

시맨틱 시스템은 다크 모드를 수동으로 힘들게 처리하는 대신 구조적으로 단순하게 만들어줘요.

잘못된 방식:

```tsx
<div className="bg-white dark:bg-gray-900 text-gray-900 dark:text-white border-gray-200 dark:border-gray-700">
```

작동하긴 해요. 하지만 모든 컴포넌트에 개별 다크 모드 변형이 필요해요. 일부는 빠뜨리게 돼요. 다른 건 잘못 처리하고요.

올바른 방식:

```css
:root {
  --color-surface-card:   #FFFFFF;
  --color-text-primary:   #111827;
  --color-border-default: #E5E7EB;
}

[data-theme="dark"] {
  --color-surface-card:   #1F2937;
  --color-text-primary:   #F9FAFB;
  --color-border-default: #374151;
}
```

```tsx
<div className="bg-[var(--color-surface-card)] text-[var(--color-text-primary)] border-[var(--color-border-default)]">
```

루트 엘리먼트의 속성 하나를 바꾸면 앱 전체가 리테마돼요. 컴포넌트 코드를 건드리지 않고 하이 컨트라스트, 색각 이상 친화적, 브랜드 변형 등 원하는 수만큼 테마를 만들 수 있어요.

## 알아두면 좋은 시맨틱 카테고리

**브랜드 색상**은 의도적인 액션을 이끌어요. primary CTA, 활성 내비게이션 상태, 포커스 링, 링크 등이에요. 호버, 활성, 비활성 상태 변형을 포함해요.

**피드백 색상**은 시스템 상태를 전달해요. 성공 확인, 에러 메시지, 경고, 정보 알림이죠. 각각 텍스트, 배경, 보더 변형이 필요해요. 에러 텍스트와 에러 배경 둘 다에 `--color-error` 하나를 쓰면 대비 문제가 생겨요.

**텍스트 색상**은 콘텐츠 계층을 만들어요. 3단계 시스템이면 대부분의 경우를 커버해요. primary (제목, 본문 텍스트), secondary (메타데이터, 설명), disabled (비활성 상태). 어두운 배경 위의 텍스트를 위한 inverse 색상도 있으면 완성돼요.

**서피스 색상**은 배경 레이어를 정의해요. 서피스마다 시각적 무게감이 달라요. 페이지 배경, 카드, 모달과 툴팁 같은 상승된 요소, 오버레이 배경 각각에 고유한 토큰이 필요해요.

**보더 색상**은 구분선, 외곽선, 포커스 링을 다루어요. Subtle, default, strong 변형으로 과도하게 세부화하지 않으면서 필요한 범위를 커버해요.

## 흔한 실수들

**시맨틱 레이어에 너무 많은 shade.** `primary-50`부터 `primary-900`을 시맨틱 토큰으로 만들면, 개발자들은 버튼 배경에 어떤 것을 쓸지, 호버 상태에 어떤 것을 쓸지 모르게 돼요. 시맨틱 레이어는 얇고 구체적으로 유지하세요. `--color-primary`, `--color-primary-hover`, `--color-primary-surface`. 전체 범위는 베이스 팔레트가 가지게 하면 돼요.

**네이밍 시스템 혼합.** `--blue-500`과 `--color-primary`가 같은 값으로 둘 다 존재하면 혼란이 생겨요. 개발자들은 그 순간 더 적합해 보이는 것을 집어요. 하나의 시스템을 선택하고 유지하세요.

**컴포넌트별 토큰 폭발.** `--button-primary-background`, `--card-background`, `--modal-background`, `--input-background`를 별도 토큰으로 만들면 이득 없이 표면적만 늘어나요. 카드 배경과 모달 배경은 보통 같은 서피스 색상을 공유해요. 시맨틱 토큰을 참조하고, 진짜 필요할 때만 컴포넌트별 오버라이드를 만드세요.

## AI 코드 생성에 이게 중요한 이유

디자인 시스템 컨텍스트 없이 AI 도구가 UI 컴포넌트를 생성하면, 통계적으로 그럴듯한 색상 선택을 해요. 카드에는 `bg-white`, 보조 텍스트에는 `text-gray-600`, 에러에는 `bg-red-500`. 흔하니까요.

이 선택들이 추상적으로 틀린 건 아니에요. 내 특정 시스템에 맞지 않는 거죠.

MCP 연결로 시맨틱 토큰이 표면화되면, AI 도구가 코드를 생성하기 전에 디자인 시스템을 조회할 수 있어요. 보조 텍스트가 아마도 `text-gray-600`일 거라고 추론하는 대신, 토큰 정의를 읽어요. `--color-text-secondary`가 특정 값에 매핑되어 있고, 그걸 쓰는 거죠. 생성된 컴포넌트가 처음부터 내 시스템 안에 착지해요.

토큰 이름이 의도를 담아요. `--color-danger`가 파괴적인 액션을 의미한다고 아는 AI는, 빨간색이 적절해 보이는 곳 어디서나가 아니라 올바른 자리에 `bg-[var(--color-danger)]`를 쓸 거예요.

시맨틱 컬러 네이밍은 프로젝트 초반에 할 수 있는 레버리지 높은 구조적 선택 중 하나예요. 네이밍 규약 자체는 비용이 없어요. 색상을 바꾸거나, 테마를 추가하거나, AI에 컴포넌트 생성을 요청할 때마다 돌아와요.

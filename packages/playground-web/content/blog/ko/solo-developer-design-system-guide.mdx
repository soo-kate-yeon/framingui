보더 레디어스 값이 네 가지. 각각 "그때는 맞았던" 파란색이 세 가지. 시간이 갈수록 뒤죽박죽이 된 색상 팔레트. 시스템 없이 UI를 만들면 자연스럽게 나타나는 결과예요.

디자이너가 없어도 고칠 수 있어요. 필요한 건 명확한 결정들을 기록하는 것, 그리고 6주 뒤에 자기 자신이 그 결정을 뒤집지 못하게 하는 형식이에요. 그게 디자인 토큰이 해주는 일이거든요. 오후 반나절이면 기반을 다질 수 있어요.

## 왜 일관성이 무너지는가

1인 개발에는 특유의 실패 패턴이 있어요. 스타일 결정을 내리는 사람이 나 혼자이니, 비일관성을 잡아낼 사람도 나 혼자예요. 새벽 3시에 기능 구현에 집중하다 보면 색상 값을 감사할 여유가 없죠. "좋아 보이는 것"을 쓰고 넘어가요.

1일차: `<button className="bg-blue-500 px-4 py-2 rounded">Submit</button>`

15일차: `<button className="bg-blue-600 px-6 py-3 rounded-lg">Continue</button>`

30일차: `<button style={{ background: '#3B82F6', padding: '12px 24px' }}>Next</button>`

그 시점마다 각자 "맞는" 선택이었어요. 모아놓고 보면 세 명이 따로따로 만든 것처럼 보이는 제품이 되죠. 과거의 나, 현재의 나, 새벽 3시의 나가 디자인 위원회인 셈이에요.

디자인 토큰은 사용 시점에서의 결정을 없애서 이 문제를 해결해요. `var(--color-brand-primary)`를 쓸 때 색상을 고르는 게 아니에요. 이미 한 번 고른 거예요. 토큰이 곧 결정이고, 한 번 만들면 어디서든 적용돼요.

## 토큰 기반 만들기

디자인 토큰 기반은 두 레이어로 구성돼요. 원시 색상 값의 베이스 팔레트와, 그 값에 의미를 부여하는 시맨틱 레이어예요.

**시맨틱 레이어부터 시작하세요.** 진짜 가치가 거기서 나오거든요.

```css
/* globals.css */
:root {
  /* 브랜드 */
  --color-brand-primary:       oklch(0.50 0.15 240);
  --color-brand-primary-hover: oklch(0.44 0.14 240);

  /* 서피스 */
  --color-surface-page:     #FAFAFA;
  --color-surface-card:     #FFFFFF;
  --color-surface-elevated: #FFFFFF;

  /* 텍스트 */
  --color-text-primary:   #111827;
  --color-text-secondary: #6B7280;

  /* 보더 */
  --color-border-default: #E5E7EB;

  /* 간격 스케일 */
  --spacing-1: 0.25rem;
  --spacing-2: 0.5rem;
  --spacing-4: 1rem;
  --spacing-6: 1.5rem;
  --spacing-8: 2rem;

  /* 레디어스 */
  --radius-sm: 0.25rem;
  --radius-md: 0.5rem;
  --radius-lg: 0.75rem;
}

[data-theme="dark"] {
  --color-surface-page:     #111827;
  --color-surface-card:     #1F2937;
  --color-surface-elevated: #374151;
  --color-text-primary:     #F9FAFB;
  --color-text-secondary:   #9CA3AF;
  --color-border-default:   #374151;
}
```

이 CSS는 파일 하나에 담겨요. 다크 모드는 루트 엘리먼트의 속성 하나를 바꾸는 것으로 처리돼요. 이 변수를 쓰는 모든 컴포넌트가 자동으로 적응하죠.

## 컴포넌트에서 토큰 사용하기

토큰을 정의했으면 일관성 있게 사용하세요. 목표는 컴포넌트 파일에 하드코딩된 색상 값을 없애는 거예요.

```tsx
// components/Card.tsx
export function Card({ children }: { children: React.ReactNode }) {
  return (
    <div
      style={{
        background: 'var(--color-surface-card)',
        border: '1px solid var(--color-border-default)',
        borderRadius: 'var(--radius-md)',
        padding: 'var(--spacing-6)',
      }}
    >
      {children}
    </div>
  );
}
```

Tailwind를 쓴다면 config에 토큰을 매핑하세요.

```javascript
// tailwind.config.js
export default {
  theme: {
    extend: {
      colors: {
        brand: {
          primary:      'var(--color-brand-primary)',
          primaryHover: 'var(--color-brand-primary-hover)',
        },
        surface: {
          page:     'var(--color-surface-page)',
          card:     'var(--color-surface-card)',
          elevated: 'var(--color-surface-elevated)',
        },
        text: {
          primary:   'var(--color-text-primary)',
          secondary: 'var(--color-text-secondary)',
        },
      },
      spacing: {
        1: 'var(--spacing-1)',
        2: 'var(--spacing-2)',
        4: 'var(--spacing-4)',
        6: 'var(--spacing-6)',
        8: 'var(--spacing-8)',
      },
      borderRadius: {
        sm: 'var(--radius-sm)',
        md: 'var(--radius-md)',
        lg: 'var(--radius-lg)',
      },
    },
  },
}
```

이제 `bg-surface-card`와 `text-text-primary`가 토큰을 기반으로 하는 유효한 Tailwind 클래스가 돼요.

## AI 도구를 디자인 시스템에 연결하기

토큰이 준비되면 AI 도구를 연결할 수 있어요. 이 부분에서 투자 대비 효과가 극적으로 나타나거든요. AI가 생성하는 모든 컴포넌트가 기본적으로 브랜드에 맞게 나올 수 있어요.

FramingUI는 디자인 시스템 컨텍스트를 Claude Code 같은 AI 도구에 직접 제공하는 MCP 서버를 제공해요. 권장하는 설정 방법은 이거예요.

```bash
npx -y @framingui/mcp-server@latest init
```

이 명령어가 프로젝트를 감지하고, 필요한 패키지를 설치하고, MCP config를 작성해요. Claude Code를 재시작하면 토큰 정의와 컴포넌트 카탈로그에 접근할 수 있게 돼요.

생성되는 `.mcp.json`은 이렇게 생겼어요.

```json
{
  "mcpServers": {
    "framingui": {
      "type": "stdio",
      "command": "npx",
      "args": ["-y", "@framingui/mcp-server@latest"]
    }
  }
}
```

이게 있으면 Claude Code가 컴포넌트를 생성하기 전에 사용 가능한 토큰을 조회할 수 있어요. 학습 데이터에서 `bg-blue-600`을 고르는 대신, 내 시스템의 `var(--color-brand-primary)`를 선택하죠.

## 디자이너 없이 색상 고르기

디자인 시스템을 처음 접하는 개발자에게 가장 어려운 부분이 임의적으로 보이지 않는 색상 팔레트를 고르는 거예요. OKLCH 색상 공간이 이걸 훨씬 다루기 쉽게 만들어줘요.

OKLCH는 색상을 밝기(lightness), 채도(chroma), 색조(hue)로 지각적으로 균일하게 표현해요. HSL이나 hex 값에서는 그렇지 않지만, OKLCH에서 밝기의 동일한 간격은 사람 눈에도 동일한 간격으로 보여요.

실용적인 primary 스케일 예시:

```css
:root {
  --color-primary-50:  oklch(0.95 0.03 240);
  --color-primary-100: oklch(0.90 0.06 240);
  --color-primary-500: oklch(0.50 0.15 240);  /* 메인 브랜드 */
  --color-primary-700: oklch(0.38 0.12 240);
  --color-primary-900: oklch(0.25 0.08 240);
}
```

hue 값(여기서는 240)이 색상 계열을 결정해요. 초록색은 140, 주황색은 30, 보라색은 290으로 바꾸면 돼요. 밝기와 채도의 관계는 비율이 유지돼요.

WCAG 대비 요건도 OKLCH로 훨씬 맞추기 쉬운 게, 밝기 값이 지각된 대비에 예측 가능하게 매핑되거든요. 밝기 0.95의 텍스트가 밝기 0.20의 배경 위에 있으면 읽히는 거예요.

## 나중에 색상을 바꾸고 싶다면

이게 이 전체 접근법의 정당성을 증명하는 부분이에요. 브랜드 색상을 바꾸고 싶으면 CSS 변수 한 곳만 업데이트해요.

```css
/* 전 */
--color-brand-primary: oklch(0.50 0.15 240);  /* 파란색 */

/* 후 */
--color-brand-primary: oklch(0.50 0.15 140);  /* 초록색 */
```

앱의 모든 버튼, 링크, 뱃지, 포커스 링이 업데이트돼요. 나중에 디자이너가 팀에 합류해서 팔레트를 다듬고 싶다면, grep으로 코드베이스를 뒤질 게 아니라 구조화된 파일을 받아서 작업할 수 있죠.

디자인 시스템은 대형 팀만의 전유물이 아니에요. 6개월 뒤에도 자기 코드가 여전히 이해되기를 원하는 사람이라면 누구에게나 필요해요. 기반을 한 번 잡아두면, 규칙이 알아서 스스로를 강제해요.

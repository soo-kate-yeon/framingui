## まず前提

個人開発では「理想的な設計」より「継続できる運用」が重要です。デザインシステム選定は次の 4 軸で判断すると失敗しにくくなります。

- 立ち上げ速度
- カスタマイズ容易性
- 保守負荷
- AI 連携しやすさ

## 3 つの選択肢

### 1. 最小構成で自作

自由度は高いですが、命名や一貫性維持に時間がかかります。

### 2. OSS コンポーネントを組み合わせ

初速は速いですが、プロジェクト後半で見た目差分の吸収コストが出ます。

### 3. トークン中心の既成システムを採用

初期学習は必要ですが、長期運用が最も安定します。

## 判断基準の目安

| 状況 | 推奨 |
| --- | --- |
| MVP を最速で出したい | 既成システム + 最小トークン |
| 機能追加が継続する | トークン中心運用 |
| 複数プロダクトに展開予定 | コンポーネント規約を先に固定 |

## AI 時代の追加基準

生成 AI を使うなら、次を必須条件にするとよいです。

- トークンが構造化されている
- コンポーネント API が公開されている
- MCP などで外部参照できる

## まとめ

インディーハッカーにとっての正解は、短期の速さと長期の保守を両立できる構成です。トークン中心で始めると、AI 生成の品質も安定しやすくなります。

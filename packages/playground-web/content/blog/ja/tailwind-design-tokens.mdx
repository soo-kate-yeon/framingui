## Tailwindの罠

Tailwindが大好きです。速く、直感的で、HTMLから離れずにUIを構築できます。

```tsx
<button className="bg-blue-600 hover:bg-blue-700 text-white px-4 py-2 rounded-lg">
  Click me
</button>
```

デプロイ。完了。美しい。

しかし6ヶ月後には:
- 23種類の異なる青のshade
- `px-4 py-2`、`px-5 py-2.5`、`px-6 py-3`、`px-4 py-1.5`のボタン
- `gap-4`を使うフォーム、`gap-6`を使うカード、`gap-3`を使うグリッド

**Tailwindがデザインシステムだと思っていました。違います。**

Tailwindは**ユーティリティツールキット**です。デザインシステムは**決定**の集合です。この記事はTailwindの速度を失わずにTailwindの上に決定をレイヤー化する方法を示します。

## 要点まとめ

- Tailwindはユーティリティを提供し、デザイン制約ではありません
- トークンなしではチームが一貫性のなさに流れます(Tailwindを使っても)
- Design tokensはTailwindユーティリティクラスの上にブランド決定をレイヤー化
- FramingUIはトークンをTailwind設定に接続し、構造と速度の両方を提供
- `bg-primary`、`px-md`を書き続けますが、今はTailwindデフォルトではなくあなたの決定です

## Tailwindだけではデザインシステムでない理由

### 問題1: 選択肢が多すぎる

Tailwindは各カラーの22種類のshadeを提供:

```javascript
colors: {
  blue: {
    50: '#eff6ff',
    100: '#dbeafe',
    // ...
    900: '#1e3a8a',
    950: '#172554'
  }
}
```

**質問: `blue-600` vs. `blue-700`をいつ使いますか?**

答えが「良く見えるもの」なら、デザインシステムがありません。きれいなグラデーションのある混沌です。

### 問題2: コピー&ペーストの抽選

開発者1がprimaryボタンを作成:
```tsx
<button className="bg-blue-600 hover:bg-blue-700 ...">
```

開発者2が別ファイルでprimaryボタンが必要:
```tsx
<button className="bg-blue-500 hover:bg-blue-600 ...">
```

両方とも独立しては「問題なし」に見えます。でも並べると? 同じ目的の異なるボタンです。

### 問題3: ブランドカラー変更は悪夢

デザイナーが言います: 「青からindigoに変えましょう。」

**生のTailwindクラス**では、検索置換を実行:
- `blue-600` → `indigo-600`
- `blue-700` → `indigo-700`
- `blue-500` → `indigo-500`(待って、これは500? 600?)

**所要時間:** 2〜4時間  
**インスタンスを見逃すリスク:** 高  
**何かを壊す可能性:** 中程度

### 問題4: Dark Modeが災害に

```tsx
// Light modeはblue-600使用
<div className="bg-blue-600 text-white">

// Dark mode... 何色にすべき?
<div className="bg-blue-600 dark:bg-blue-400 text-white dark:text-black">
// 待って、テキストは黒? gray-900? 白?
```

**セマンティックトークン**なしでは、毎回コントラスト比と視覚階層を推測します。

## Design Tokensが実際に解決すること

Design tokensは値にマッピングされる**名前付き決定**:

```typescript
// これの代わり:
bg-blue-600

// これを書きます:
bg-brand-primary

// これがマッピング:
colors.brand.primary = '#2563EB'  // (またはblue-600、indigo-700、何でも)
```

**魔法:** `colors.brand.primary`を変更すると、すべてのコンポーネントが自動更新されます。

### セマンティックネーミング = デザイン意図

```typescript
// ❌ 実装中心(Tailwindデフォルト)
colors: {
  blue: { 600: '#2563EB' }
}

// ✅ 意図中心(Design Tokens)
colors: {
  brand: {
    primary: '#2563EB',      // メインCTA、リンク
    primaryHover: '#1D4ED8', // Hover状態
  },
  semantic: {
    success: '#10B981',
    error: '#EF4444',
    warning: '#F59E0B',
  },
  surface: {
    base: '#FFFFFF',         // カード背景
    elevated: '#F9FAFB',     // モーダル、ポップオーバー
  }
}
```

今`bg-surface-elevated`を見れば、その色が**なぜ**存在するかわかり、単に**何**であるかだけでなく理解できます。

## 欠けている接続: Tailwind + Tokens

Tailwindの速度とデザインシステムの一貫性の間で選択する必要はありません。両方得られます。

### アーキテクチャ

```
Design Tokens(source of truth)
         ↓
   Tailwind Config(ユーティリティ)
         ↓
   Your Components(クラス)
```

### ステップ1: トークン定義(YAML、JSON、またはTypeScript)

```typescript
// tokens/colors.ts
export const colorTokens = {
  brand: {
    primary: '#4338CA',
    primaryHover: '#3730A3',
    secondary: '#7C3AED',
  },
  neutral: {
    50: '#F9FAFB',
    100: '#F3F4F6',
    200: '#E5E7EB',
    // ... 全スケール
    900: '#111827',
  },
  semantic: {
    success: '#10B981',
    successBg: '#D1FAE5',
    error: '#EF4444',
    errorBg: '#FEE2E2',
  }
}
```

### ステップ2: Tailwind Configにトークンを注入

```javascript
// tailwind.config.js
import { colorTokens } from './tokens/colors'

export default {
  theme: {
    extend: {
      colors: {
        brand: colorTokens.brand,
        neutral: colorTokens.neutral,
        semantic: colorTokens.semantic,
      }
    }
  }
}
```

### ステップ3: セマンティッククラスを使用

```tsx
// ❌ 以前(生のTailwind)
<button className="bg-blue-600 hover:bg-blue-700 text-white">
  Save
</button>

// ✅ 以降(Token駆動Tailwind)
<button className="bg-brand-primary hover:bg-brand-primaryHover text-white">
  Save
</button>
```

**今ブランドカラー変更は1行編集:**

```diff
// tokens/colors.ts
export const colorTokens = {
  brand: {
-   primary: '#4338CA',
+   primary: '#7C3AED',  // 紫に切り替え
  }
}
```

すべてのボタン、リンク、CTAが即座に更新。検索置換ゼロ。

## Spacing、Typographyなど

カラーは始まりにすぎません。トークンは**すべてのデザイン決定**に機能します。

### Spacing Tokens

```typescript
// tokens/spacing.ts
export const spacingTokens = {
  '3xs': '0.125rem',  // 2px
  '2xs': '0.25rem',   // 4px
  'xs': '0.5rem',     // 8px
  'sm': '0.75rem',    // 12px
  'md': '1rem',       // 16px
  'lg': '1.5rem',     // 24px
  'xl': '2rem',       // 32px
  '2xl': '3rem',      // 48px
  '3xl': '4rem',      // 64px
}

// tailwind.config.js
export default {
  theme: {
    spacing: spacingTokens,
  }
}
```

**使用:**
```tsx
// ❌ 以前: 任意値
<div className="px-4 py-3 gap-5">

// ✅ 以降: 名前付き決定
<div className="px-md py-sm gap-lg">
```

### Typography Tokens

```typescript
// tokens/typography.ts
export const typographyTokens = {
  fontFamily: {
    sans: ['Inter', 'system-ui', 'sans-serif'],
    mono: ['JetBrains Mono', 'monospace'],
  },
  fontSize: {
    'xs': ['0.75rem', { lineHeight: '1rem' }],
    'sm': ['0.875rem', { lineHeight: '1.25rem' }],
    'base': ['1rem', { lineHeight: '1.5rem' }],
    'lg': ['1.125rem', { lineHeight: '1.75rem' }],
    'xl': ['1.25rem', { lineHeight: '1.75rem' }],
    '2xl': ['1.5rem', { lineHeight: '2rem' }],
  },
  fontWeight: {
    normal: '400',
    medium: '500',
    semibold: '600',
    bold: '700',
  }
}

// tailwind.config.js
export default {
  theme: {
    extend: {
      fontFamily: typographyTokens.fontFamily,
      fontSize: typographyTokens.fontSize,
      fontWeight: typographyTokens.fontWeight,
    }
  }
}
```

**使用:**
```tsx
<h1 className="font-sans text-2xl font-bold">
  Heading
</h1>
<p className="font-sans text-base font-normal text-neutral-700">
  一貫したline heightの本文テキスト。
</p>
```

## Tokensとのダークモード

トークンが真に輝く場所です。ダークモードカラーを推測する代わりに、**セマンティック役割**を定義:

```typescript
// tokens/colors.ts
export const colorTokens = {
  surface: {
    base: {
      light: '#FFFFFF',
      dark: '#111827',
    },
    elevated: {
      light: '#F9FAFB',
      dark: '#1F2937',
    }
  },
  text: {
    primary: {
      light: '#111827',
      dark: '#F9FAFB',
    },
    secondary: {
      light: '#6B7280',
      dark: '#9CA3AF',
    }
  }
}
```

**CSS変数を使ったTailwind config:**

```javascript
// tailwind.config.js
export default {
  theme: {
    extend: {
      colors: {
        surface: {
          base: 'var(--color-surface-base)',
          elevated: 'var(--color-surface-elevated)',
        },
        text: {
          primary: 'var(--color-text-primary)',
          secondary: 'var(--color-text-secondary)',
        }
      }
    }
  }
}
```

**CSS:**

```css
:root {
  --color-surface-base: #FFFFFF;
  --color-surface-elevated: #F9FAFB;
  --color-text-primary: #111827;
  --color-text-secondary: #6B7280;
}

.dark {
  --color-surface-base: #111827;
  --color-surface-elevated: #1F2937;
  --color-text-primary: #F9FAFB;
  --color-text-secondary: #9CA3AF;
}
```

**コンポーネント:**
```tsx
<div className="bg-surface-base text-text-primary">
  これは自動でlight/dark modeに適応します
</div>
```

`dark:bg-gray-900 dark:text-white`が不要。トークンが処理します。

## FramingUIアプローチ

FramingUIはこのワークフロー全体を自動化します。

### 1. トークンを一度定義

```typescript
// framingui.config.ts
import { defineConfig } from '@framingui/core'

export default defineConfig({
  tokens: {
    colors: {
      brand: {
        primary: '#4338CA',
        secondary: '#7C3AED',
      },
      // ... カラー
    },
    spacing: {
      sm: '0.5rem',
      md: '1rem',
      lg: '1.5rem',
    },
    typography: {
      fontFamily: {
        sans: 'Inter, system-ui',
      }
    }
  }
})
```

### 2. Tailwind Config自動生成

```bash
npx framingui generate
```

出力:
- トークン付き`tailwind.config.js`
- dark mode用CSS変数
- オートコンプリート用TypeScript型

### 3. コンポーネントで使用

```tsx
import { tokens } from '@/framingui'

// Tailwindクラス(オートコンプリート動作)
<button className="bg-brand-primary px-md py-sm">

// またはプログラマティックアクセス
<button style={{ backgroundColor: tokens.colors.brand.primary }}>
```

### 4. トークン変更、即座に再ビルド

```diff
// framingui.config.ts
colors: {
  brand: {
-   primary: '#4338CA',
+   primary: '#DC2626',  // 赤にリブランド
  }
}
```

```bash
npx framingui generate
```

`bg-brand-primary`を使うすべてのコンポーネントが更新。手動検索置換なし。

## 実例: 5分でリブランド

**シナリオ:** スタートアップが買収されました。新しいブランドカラー、新しいタイポグラフィ、新しいスペーシングシステム。

**以前(生のTailwind):**
1. `blue-600`のすべてのインスタンスを検索 → 2時間
2. `px-4`のすべてのインスタンスを見つけて保持するか決定 → 3時間
3. タイポグラフィクラスを更新 → 1時間
4. 見逃したインスタンスをQA → 2時間
**合計: 8時間**

**以降(Tailwind + Tokens):**
1. `framingui.config.ts`更新(10分)
2. `npx framingui generate`実行(5秒)
3. 視覚的にQA(20分)
**合計: 30分**

**節約時間: 7.5時間**

## よくある反論

### 「Tailwindクラスは十分セマンティック」

`bg-blue-600`は**何**(青のshade)を伝えますが、**なぜ**(primaryブランドカラー? リンクカラー? 装飾アクセント?)は伝えません。

`bg-brand-primary`は**意図**を伝えます。コードで見れば役割がわかります。

### 「これは余分な抽象化を追加する」

その通りですが、正しい種類です。**実装**ではなく**決定**を抽象化します。

トークンがあれば気を変えるのは安い。トークンなしでは気を変えるのは高くつきます。

### 「Tailwindの任意値が好き」

ワンオフで保持:
```tsx
<div className="w-[47.5%]">  // ユニークレイアウトには問題なし
```

しかし繰り返される決定(ブランドカラー、スペーシングスケール)にはトークンを使用。

### 「チームが小さすぎてこれには不向き」

小さなチームが最も恩恵を受けます。官僚主義なしで基盤を設定できる完璧なサイズです。

**2人チーム:** あなたと共同創業者が離れるのを防ぐ
**5人チーム:** 新規採用を即座にオンボード
**10人チーム:** 一貫性に不可欠

## マイグレーション戦略: すべてを書き直さずにトークンを追加

一晩でコードベース全体をリファクタリングする必要はありません。

### フェーズ1: トークン定義(1時間)

最も使うカラー、スペーシング、タイポグラフィをトークンに抽出。

### フェーズ2: Tailwind Config拡張(30分)

Tailwindテーマにトークンを追加。古いクラスは依然として動作。

### フェーズ3: 段階的に採用

**新しいコンポーネント:** トークン使用(`bg-brand-primary`)
**古いコンポーネント:** そのまま(`bg-blue-600`)

両方が同時に動作。破壊的変更なし。

### フェーズ4: ホットパスリファクタリング(継続中)

別の理由でファイルを触るとき、クラスをトークン使用に更新。

2〜3ヶ月かけてコードベースが自然に収束。

## まとめ

**Tailwindは速度を与えます。**  
**Design tokensは一貫性を与えます。**  
**一緒なら両方得られます。**

速く動くことと一貫したデザインシステム構築の間で選択する必要はありません。

Tailwindユーティリティクラスの上にセマンティックトークンをレイヤー化します。好きなDXを保持し、必要な構造を追加します。

## 次のステップ

1. **Tailwind使用を監査**: 青のshadeはいくつありますか?
2. **繰り返し値を抽出**: カラー、スペーシング、タイポグラフィ
3. **FramingUIインストール**: `npm install @framingui/core`
4. **トークン定義**: カラーから開始、次にスペーシングとタイポグラフィ追加
5. **Tailwind config生成**: `npx framingui generate`

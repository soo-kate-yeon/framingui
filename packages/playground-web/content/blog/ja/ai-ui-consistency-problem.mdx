## AI一貫性の問題

Cursorでダッシュボードを作成中です。Claudeにリクエスト:

> "名前、アバター、ステータスバッジ付きのユーザープロファイルカードを作って"

Claudeが素晴らしいコードを生成:

```tsx
<div className="bg-white rounded-lg p-6 shadow-md">
  <img src="/avatar.jpg" className="w-16 h-16 rounded-full" />
  <h3 className="text-xl font-bold text-gray-900">John Doe</h3>
  <span className="bg-green-500 text-white px-3 py-1 rounded-full text-sm">
    Active
  </span>
</div>
```

完璧! コードベースにコピーします。

2日後、再度リクエスト:

> "アイコン、メッセージ、タイムスタンプ付きの通知カードを作って"

Claudeが生成:

```tsx
<div className="bg-slate-50 rounded-xl p-4 border border-gray-200">
  <div className="flex items-center gap-3">
    <Bell className="w-5 h-5 text-blue-600" />
    <p className="text-base text-slate-800">New message received</p>
  </div>
  <time className="text-xs text-gray-500">2 minutes ago</time>
</div>
```

これも良さそう! でも待って...

- プロファイルカード: `bg-white`、通知: `bg-slate-50`
- プロファイル: `rounded-lg`、通知: `rounded-xl`
- プロファイル: `p-6`、通知: `p-4`
- プロファイル: `text-gray-900`、通知: `text-slate-800`

**AIが全く異なるデザイン言語で2つのカードを生成しました。**

## なぜこうなるか: AIがUIコードを生成する仕組み

### パターンマッチングの問題

Claude、GPT-4、GeminiなどのLLMはデザインシステムを「記憶」しません。コード生成方法:

1. トレーニングデータ(GitHub、ドキュメント、チュートリアル)に対する**パターンマッチング**
2. 統計的に最も可能性の高い次のトークンを**予測**
3. 独立して「合理的に見える」ものに対する**最適化**

「カードコンポーネント」をリクエストすると、AIは見た数千のカード実装からサンプリング:

- Material Designカード(Googleのパターン)
- Tailwind UIの例
- shadcn/uiコンポーネント
- ランダムなGitHubリポジトリ
- Stack Overflowスニペット

それぞれが異なるものを使用:
- カラーパレット(`blue-500` vs. `indigo-600` vs. `#4F46E5`)
- スペーシングスケール(`p-4` vs. `p-6` vs. `px-5 py-3`)
- Border radius値(`rounded-md` vs. `rounded-lg` vs. `rounded-xl`)
- Typographyスケール(`text-sm` vs. `text-base` vs. `text-lg`)

**AIはどれがあなたのパターンか知る方法がありません。**

### コンテキストウィンドウの制限

デザインシステムをチャットに貼り付けても2つの問題に直面:

**1. コンテキスト減衰**

会話でデザインシステムが後ろにあるほど、AI生成での「重み」が減少:

```
メッセージ1: [デザイントークン - 2000トークン]
メッセージ2-5: [その他の実装詳細]
メッセージ6: "新しいカードコンポーネントを作って"
         ↑
         AIがトークンを「忘れた」
```

**2. トークン予算**

すべてのメッセージがトークンを消費。デザインシステム全体(カラー、スペーシング、タイポグラフィ、コンポーネント)の貼り付けは:
- 包括的なdesign tokensに2,000〜5,000トークン使用
- 実際の実装コンテキストに使える余地が減少
- コストが急速に増加(特にClaude Opus / GPT-4)

### 暗黙の仮定の問題

AIは指定しないと「合理的なデフォルト」で空白を埋めます:

**あなた:** "ボタンを作って"

**AIが仮定:**
- Primary color: おそらく青(`blue-600`)
- Padding: おそらく`px-4 py-2`(Tailwindデフォルト)
- Border radius: おそらく`rounded-md`
- Font weight: おそらく`font-medium`

これらの仮定は**トレーニングデータの統計的頻度**に基づき、あなたのデザイン決定ではありません。

## 実際の一貫性問題の事例

### ケース1: カラードリフト

Claude Codeで2週間分析ダッシュボードを開発:

**1週目のコンポーネント:**
- Primary actionに`blue-600`使用
- 背景に`gray-100`
- Success状態に`green-500`

**2週目のコンポーネント(同じプロンプト、新しいチャット):**
- Primary actionに`indigo-600`使用
- 背景に`slate-50`
- Success状態に`emerald-500`

**結果:** アプリが2人の異なる人がデザインしたように見える。

### ケース2: スペーシングの混乱

CursorでSaaSオンボーディングフロー開発:

```tsx
// Step 1(月曜日生成)
<div className="p-6 space-y-4">

// Step 2(火曜日生成)  
<div className="p-8 space-y-6">

// Step 3(水曜日生成)
<div className="p-4 space-y-3">
```

同じフロー、同じデザイナーの意図、**3つの異なるスペーシングシステム**。

### ケース3: タイポグラフィのジャングル

ChatGPTにフォームコンポーネント生成をリクエスト:

- Input label: `text-sm font-medium text-gray-700`
- Error message: `text-xs text-red-600`
- Helper text: `text-sm text-gray-500`
- Placeholder: `text-base text-gray-400`

サポートテキストに4つの異なるフォントサイズ。システムなし、感覚だけ。

## 現在のソリューションが機能しない理由

### ❌ ソリューション1: 「AIに一貫性を保つよう伝える」

```
プロンプト: "すべてのコンポーネントで一貫したスペーシングとカラーを使用"
```

**問題:** 「一貫した」が曖昧。AIは1つのレスポンス内では内部的に一貫していますが、複数生成にわたっては一貫しません。

### ❌ ソリューション2: 「毎回デザインシステムを貼り付ける」

```
プロンプト: [2000トークンのデザインシステム]
"これらのトークンを使ってボタンを作って"
```

**問題:**
- 面倒で遅い
- トークン制限を消費
- 1〜2人の開発者を超えてスケールしない
- 長い会話ではコンテキスト減衰が依然として発生

### ❌ ソリューション3: 「AI内蔵メモリを使う」

Claude ProjectsとChatGPT Memoryは役立ちますが:
- 限られたメモリサイズ(包括的なデザインシステムを保持できない)
- プログラマティックアクセスなし(「primary colorは何?」とクエリできない)
- 一貫性のない想起(カラーパレットは覚えているがスペーシングスケールは忘れることも)

### ❌ ソリューション4: 「モデルをファインチューン」

技術的に可能ですが:
- ML専門知識が必要
- 高額(数千ドル)
- デザインシステム更新ごとに再トレーニング必要
- 99%のチームには過剰

## Design Tokenソリューション

Design tokensはAIにデザインシステムへの**プログラマティックアクセス**を提供して問題を解決します。

### 仕組み: MCP(Model Context Protocol)

チャットにトークンを貼り付ける代わりに、デザインシステムをAIツールに**接続**します:

```
┌──────────────┐         MCP          ┌─────────────────┐
│   Cursor     │ ◄─────────────────► │  Design Token   │
│   Claude     │  "primaryは何?"      │     Server      │
│   ChatGPT    │ ◄─────────────────► │  (FramingUI)    │
└──────────────┘    返却: #4338CA      └─────────────────┘
```

**以前(コピー&ペーストアプローチ):**
```
ユーザー: "ブランドカラーでボタンを作って"
AI: *推測* → bg-blue-600を生成
```

**以降(MCPアプローチ):**
```
ユーザー: "ブランドカラーでボタンを作って"
AI: *トークンサーバーをクエリ* → primary: '#4338CA'を受信
AI: 生成 → bg-[#4338CA] またはカスタムTailwindクラスbg-brand-primaryを使用
```

### FramingUIアプローチ

FramingUIはdesign tokensをクエリ可能なリソースとして公開するMCPサーバーを提供:

**1. トークンを一度定義**

```typescript
// tokens/config.ts
export const tokens = {
  colors: {
    brand: {
      primary: '#4338CA',
      secondary: '#7C3AED',
    },
    neutral: {
      50: '#F9FAFB',
      900: '#111827',
    }
  },
  spacing: {
    sm: '0.5rem',
    md: '1rem',
    lg: '1.5rem',
  },
  typography: {
    fontFamily: {
      sans: 'Inter, system-ui',
    },
    fontSize: {
      sm: '0.875rem',
      base: '1rem',
      lg: '1.125rem',
    }
  }
}
```

**2. AIが自動でトークンをクエリ**

CursorまたはClaude Codeにコンポーネント生成をリクエストすると:
1. UIコンポーネント作業中であることを検出
2. トークンサーバーをクエリ: 「利用可能なカラーは?」
3. トークン構造を受信
4. 汎用トークンではなくあなたのトークンを使ってコード生成

**3. 毎回一貫した出力**

```tsx
// 最初の生成(1日目)
<button className="bg-brand-primary text-white px-md py-sm rounded-md">
  Primary Button
</button>

// 2回目の生成(30日目、別のチャットセッション)
<button className="bg-brand-primary text-white px-md py-sm rounded-md">
  Another Button
</button>
```

同じクラス、同じデザイン言語、**コピー&ペーストゼロ**。

## トークン対応AIのセットアップ(5分)

### ステップ1: FramingUIインストール

```bash
npm install @framingui/tokens @framingui/mcp-server
```

### ステップ2: トークン定義

```typescript
// lib/tokens.ts
import { createTokens } from '@framingui/tokens'

export const tokens = createTokens({
  colors: {
    brand: {
      primary: '#4338CA',
      secondary: '#7C3AED',
    },
    // ... カラーシステム
  },
  spacing: {
    xs: '0.25rem',
    sm: '0.5rem',
    md: '1rem',
    lg: '1.5rem',
    xl: '2rem',
  }
})
```

### ステップ3: MCPサーバー設定

```json
// claude_desktop_config.json(Claude Code用)
{
  "mcpServers": {
    "framingui": {
      "command": "npx",
      "args": ["@framingui/mcp-server", "--tokens", "./lib/tokens.ts"]
    }
  }
}
```

Cursorの場合、`.cursorrules`に追加:

```
このプロジェクトはFramingUI design tokensを使用しています。
UIコードを生成する前に常に利用可能なトークンをクエリしてください。
ハードコードされた値の代わりにトークン変数を使用してください。
```

### ステップ4: テスト

**プロンプト:** "primaryボタンを作って"

**AI出力(MCP前):**
```tsx
<button className="bg-blue-600 px-4 py-2">Click me</button>
```

**AI出力(MCP後):**
```tsx
<button className="bg-brand-primary px-md py-sm">Click me</button>
```

## AI + Design Tokensのベストプラクティス

### 1. コンポーネントではなくトークンから開始

AIがコンポーネントを生成する前にカラーパレット、スペーシングスケール、タイポグラフィシステムを定義します。そうしないと、AI出力からデザインシステムをリバースエンジニアリングすることになります。

### 2. セマンティックネーミングを使用

```typescript
// ❌ 悪い(AIが意図を推測できない)
colors: {
  blue500: '#3B82F6',
  blue600: '#2563EB',
}

// ✅ 良い(AIが目的を理解)
colors: {
  brand: {
    primary: '#2563EB',
    primaryHover: '#1D4ED8',
  }
}
```

### 3. 使用コンテキストを提供

```typescript
// ✅ AIコンテキスト用のコメントを含める
export const tokens = {
  spacing: {
    sm: '0.5rem',  // コンパクトUI、モバイル用
    md: '1rem',    // ほとんどのコンポーネントのデフォルトスペーシング
    lg: '1.5rem',  // ゆとりのあるレイアウト、マーケティングページ用
  }
}
```

### 4. リンティングと統合

AIの幻覚を自動で検出:

```javascript
// eslint-plugin-design-tokens
{
  "rules": {
    "design-tokens/no-hardcoded-colors": "error",
    "design-tokens/no-arbitrary-spacing": "warn"
  }
}
```

### 5. 定期的に監査

```bash
# コードベースでハードコードされたカラーを見つける
npx @framingui/audit colors

# 出力:
# ❌ src/components/Button.tsx:12 - ハードコード #3B82F6
# ❌ src/pages/Dashboard.tsx:45 - ハードコード rgb(59, 130, 246)
# ✅ 47コンポーネントがトークンを正しく使用
```

## まとめ

AIはUIコード生成には素晴らしいですが、ガードレールなしでは一貫性維持に不向きです。

**問題はAIの知性ではありません。AIのコンテキスト不足です。**

Design tokens + MCPはAIが毎回デザインシステムと一致するコードを生成するのに必要なコンテキストを提供します。

**トークンなし:** AIはスタイルガイドを忘れ続けるジュニア開発者です。

**トークンあり:** AIはデザインシステムを完璧に知るシニア開発者です。

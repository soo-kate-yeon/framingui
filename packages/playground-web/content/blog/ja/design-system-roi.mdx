## 小規模チームのジレンマ

3〜5人でSaaS製品を作っています。デザイナー1人(いれば幸運)、開発者2人、そして残りすべてを担当するファウンダー。1分1秒が重要で、お金も節約が必要です。

誰かが提案します。「デザインシステムを導入してみませんか?」

即座の反応: **「そんな時間ありません。デザインシステムは専属チームのある大企業がやるものです。」**

この記事は実際の数値で、その考えが間違っていることを証明します。

## 要点まとめ

- 小規模チーム(2〜5人)は基本デザインシステム導入後**月15〜25時間節約**
- 平均投資回収期間: 初期セットアップ時間の**2〜4週間**
- ROIは時間とともに増加: ほとんどのチームで初年度300〜500%
- Figma + Storybook + ドキュメントすべては不要—design tokens + コンポーネントライブラリで開始
- **FramingUI**のようなツールでセットアップ時間が数週間から数時間に短縮

## デザインシステムがない場合の実際のコスト

デザインシステムなしで開発する場合に何が起こるか追跡します。実際のチームのデータです。

### シナリオ: 新機能開発

**1週目: 初期実装**

- デザイナーが既存と少し異なるボタンスタイルでモックアップ作成(どれが「正しい」か議論に**2時間**)
- 開発者が他で使われている`#4338CA`ではなく`#4F46E5`で実装(コードレビューで修正に**30分**)
- 間隔が他のアプリと合わない—あるカードは`16px`、別のは`20px`、今回は`18px`(Figmaコメントで議論に**1時間**)

**2週目: イテレーション**

- プロダクトマネージャーがブランド全体のprimary buttonカラー変更をリクエスト
- 開発者が**12ファイル**のボタンを更新(**3時間**)
- 3箇所を見落とし; バグレポート到着(探して修正に**1時間**)

**3週目: 契約社員のオンボーディング**

- 新しい開発者が質問: 「spacing scaleは何ですか?」誰も知らない。(既存コードをリバースエンジニアリングに**2時間**)
- 「どの青を使えばいいですか?」コードベースに青が7種類。(Slackディスカッション**1時間**)

**月間コスト: 3人が無駄にする時間約11時間**。チームの時間単価に応じて**約$1,100〜$2,200**(スタートアップ基準$100〜$200/時間)。

年間: **$13,200〜$26,400の無駄**。

## デザインシステムの実際のコスト

セットアップ時間とメンテナンスについて現実的に見ていきます。

### 初期セットアップ(最新ツール使用)

**オプションA: ゼロから構築**
- Design tokens定義(colors, spacing, typography): **8時間**
- コアコンポーネント構築(Button, Input, Cardなど): **24時間**
- ドキュメント化: **8時間**
- **合計: 40時間(約1週間)**

**オプションB: 基盤活用(FramingUI + shadcn/ui)**
- FramingUI design tokensインストールと設定: **2時間**
- トークンをブランドに合わせてカスタマイズ: **3時間**
- shadcn/uiコンポーネントのインポートと調整: **3時間**
- 基本ドキュメント: **2時間**
- **合計: 10時間(約1.5日)**

### 継続的なメンテナンス

- 月次更新/追加: **2〜4時間**
- 四半期レビュー: **4時間**
- **年間: 約30〜40時間**

## 2年目以降: 複利効果

2年目にはすでにセットアップコストを支払っています。メンテナンスは同程度ですが、メリットは拡大します:

### 2年目以降の新しいメリット

1. **より速い機能開発**: コンポーネントが事前構築・テスト済み
   - 平均実装時間20%短縮
   - 月2機能リリースのチーム基準: **月約15時間節約**

2. **スケーリング容易**: 4人目または5人目のチームメンバー追加
   - オンボーディング時間50%短縮(**新規採用あたり約12時間節約**)
   - 新しい開発者が2週間ではなく初日から生産的

3. **AI駆動開発**: MCPを使うCursor/Claude Codeのようなツール
   - AIが自動でブランドに合ったコード生成
   - 手動トークン置換から**月約8時間節約**

**2年目価値: $36,000以上の時間節約**

## 小規模チームのスイートスポット: 何を作るか

すべてを作る必要はありません。**80/20**に集中します。

### 最小機能デザインシステム(10時間)

**1. Design Tokens(3時間)**
```typescript
// tokens/colors.ts
export const colors = {
  brand: {
    primary: '#4338CA',
    secondary: '#7C3AED',
  },
  neutral: {
    50: '#F9FAFB',
    100: '#F3F4F6',
    // ... 全スケール
    900: '#111827',
  },
  semantic: {
    success: '#10B981',
    error: '#EF4444',
    warning: '#F59E0B',
  }
}

// tokens/spacing.ts
export const spacing = {
  xs: '0.25rem',   // 4px
  sm: '0.5rem',    // 8px
  md: '1rem',      // 16px
  lg: '1.5rem',    // 24px
  xl: '2rem',      // 32px
}
```

**2. コアコンポーネント(5時間)**

最も使用頻度の高い5つのコンポーネントに集中:
- Button(primary, secondary, ghost variants)
- Input
- Card
- Modal
- Select/Dropdown

**3. ドキュメント(2時間)**

README一つに:
- Design tokensの使用方法
- コンポーネント例
- 新しいコンポーネントを作るべきときvs既存コンポーネントを使うとき

### まだ作らないもの

- ❌ Storybook(8人以上に成長したら追加)
- ❌ 自動visual regressionテスト(時期尚早)
- ❌ 包括的なドキュメントサイト(READMEで十分)
- ❌ 自動同期するFigmaライブラリ(良いが必須ではない)

## 意思決定フレームワーク: デザインシステムを作るべきか

### 作るべき場合:

✅ 同じコードベースで働く開発者が2人以上  
✅ 今後6ヶ月で5つ以上の新機能を作る予定  
✅ 「どの青を使う?」と言ったことがある  
✅ 採用または契約社員をオンボーディング予定  
✅ AIコーディングツール(Cursor, Claude Code)を効果的に使いたい

### 作らないべき場合(まだ):

❌ Product/market fit前で2週間ごとにピボット中  
❌ 一人でMVPを作っている(Tailwind + shadcn/uiをそのまま使用)  
❌ プロダクトが純粋に内部ツールでユーザーが自分だけ

### 検討する価値がある場合:

⚠️ 近々資金調達予定(デモで完成度が重要)  
⚠️ 初めてのデザイナー採用予定(先に基盤を設定)  
⚠️ 類似製品/ホワイトラベルが3〜5つ(再利用性 = 高ROI)

## 今週から始める方法

### 1日目: 点検(2時間)

1. 最も使用頻度の高いコンポーネント10個をリストアップ
2. コードに存在するprimary colorの異なるshadeの数を数える
3. 現在のspacing値をドキュメント化(コードベースで`px-`, `p-`, `m-`をgrep)

### 2〜3日目: Tokensセットアップ(4時間)

**オプション1: FramingUI(推奨)**

```bash
npm install @framingui/tokens
```

```typescript
// lib/tokens.ts
import { createTokens } from '@framingui/tokens'

export const tokens = createTokens({
  colors: {
    brand: {
      primary: '#4338CA',
      // ここにカラーを追加
    }
  },
  // ... その他のデザイン決定事項
})
```

**オプション2: CSS Variablesで構築**

```css
/* styles/tokens.css */
:root {
  --color-primary: #4338CA;
  --color-secondary: #7C3AED;
  --spacing-md: 1rem;
  --font-sans: 'Inter', system-ui;
}
```

### 4〜5日目: トップ3コンポーネントのマイグレーション(4時間)

ゼロから再構築しないでください。既存コンポーネントをトークン使用にリファクタリング:

```tsx
// Before
<button className="bg-blue-600 px-4 py-2">
  Click me
</button>

// After
import { tokens } from '@/lib/tokens'

<button style={{
  backgroundColor: tokens.colors.brand.primary,
  padding: `${tokens.spacing.sm} ${tokens.spacing.md}`
}}>
  Click me
</button>

// またはTailwind + FramingUI使用
<button className="bg-brand-primary px-md py-sm">
  Click me
</button>
```

### 2週目: ドキュメント化 & 普及(2時間)

リポジトリに`DESIGN_SYSTEM.md`を作成:

````markdown
# デザインシステム

## Colors
- Primary: `tokens.colors.brand.primary`
- 用途: CTA、リンク、active状態

## Spacing
- 任意の値ではなく`tokens.spacing.*`を使用
- `px-[13]`や任意の数字は使用禁止

## Components
- Button: `<Button variant="primary">` (`/components/Button.tsx`参照)
- Input: `<Input />` (`/components/Input.tsx`参照)

## 不確かなとき
#design-system Slackチャネルで質問するか、既存コンポーネントを先に確認してください。
````

## よくある反論(そしてなぜ間違っているか)

### 「デザインシステムを使うには小さすぎます」

**完璧な**サイズです。大企業は50人から同意を得るのに苦労します。1週間で基盤を設定し、全員が整列できます。

### 「Product/market fit後にします」

木を植えるのに最適な時期は20年前でした。2番目に良い時期は今です。基本デザインシステム(10時間)はロードマップを遅らせませんが、その後すべてを高速化します。

### 「デザイナーがFigmaファイルを管理すればいい」

FigmaファイルはコードのSource of truthではありません。開発者は推測し、ハードコードし、適当に見えるTailwindクラスを使います。プログラマティックトークンが必要です。

### 「Tailwind/Material-UIをそのまま使います」

汎用ライブラリはMVPに適していますが、ブランド差別化が必要な瞬間(資金調達、マーケティング、ユーザー信頼のために必要)が来たらカスタマイズが必要です。今トークンで始めましょう; 後で必要ならコンポーネントを交換します。

## まとめ

3〜5人チームにとってデザインシステムはオーバーヘッドではなく**インフラ**です。

データベース、認証システム、デプロイパイプラインなしでSaaSアプリを作りません。デザインシステムも同じカテゴリの基礎ツールです。

**最小投資: 10時間**  
**一般的な投資回収: 2〜4週間**  
**初年度ROI: 100〜500%**

質問は「デザインシステムを作る余裕はあるか?」ではありません。

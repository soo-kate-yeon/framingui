# Input

An accessible text input component supporting multiple input types (text, email, password, number, etc.) with full Tekton design token integration for consistent styling.

## Overview

The Input component provides a foundation for form data entry. Built with accessibility in mind, it supports all native HTML input types and states while maintaining consistent styling through Tekton tokens.

**Key Features:**
- Supports all HTML input types (text, email, password, number, tel, url, date, etc.)
- Built-in focus and disabled states
- Accessible with proper ARIA attributes
- File input support
- 100% Tekton token compliance
- TypeScript support with full type inference

## Installation

```bash
npm install @framingui/ui
```

```tsx

  return <Input placeholder="Enter text..." />;
```

## Props/API

| Prop | Type | Default | Description |
|------|------|---------|-------------|
| `type` | `'text' \| 'email' \| 'password' \| 'number' \| 'tel' \| 'url' \| 'search' \| 'date' \| 'time' \| 'file'` | `'text'` | HTML input type |
| `placeholder` | `string` | `undefined` | Placeholder text |
| `disabled` | `boolean` | `false` | Disable input interaction |
| `value` | `string` | `undefined` | Controlled value |
| `defaultValue` | `string` | `undefined` | Uncontrolled default value |
| `onChange` | `(event: ChangeEvent) => void` | `undefined` | Change event handler |
| `className` | `string` | `undefined` | Additional CSS classes |
| `id` | `string` | `undefined` | HTML id attribute (for label association) |
| `name` | `string` | `undefined` | Form field name |
| `required` | `boolean` | `false` | Required field validation |
| `aria-invalid` | `boolean` | `false` | Indicates validation error state |
| `aria-describedby` | `string` | `undefined` | Associates error/helper text |

All standard HTML input attributes are supported via rest props.

## Best Practices

### ✅ DO

- Always associate inputs with labels using `htmlFor` and `id` attributes
- Use semantic input types (email, tel, url) for browser validation and mobile keyboard optimization
- Include `aria-invalid` and `aria-describedby` for error states
- Provide clear placeholder text that describes expected format
- Use `required` attribute for required fields to enable browser validation

### ❌ DON'T

- Don't use placeholder text as a replacement for labels (accessibility issue)
- Avoid very wide inputs without width constraints (poor UX)
- Don't forget to handle both controlled and uncontrolled input patterns
- Avoid custom validation without proper ARIA error messaging
- Don't disable inputs without clear visual indication of why

## Accessibility Notes

- Input has proper focus-visible ring styles for keyboard navigation
- Disabled inputs have `cursor-not-allowed` and reduced opacity
- All input types support browser-native keyboard accessibility
- File inputs have proper file button styling
- Placeholder text uses muted foreground color for sufficient contrast
- Focus ring uses 2px ring with offset for clear visibility

## Common Patterns

### Controlled Input

```tsx
import { Input } from '@framingui/ui';
import { useState } from 'react';

export default function ControlledExample() {
  const [value, setValue] = useState('');

  return (
    <Input
      value={value}
      onChange={(e) => setValue(e.target.value)}
      placeholder="Controlled input"
    />
  );
}
```

### Form Integration

```tsx
import { Input } from '@framingui/ui';
import { Label } from '@framingui/ui';
import { Button } from '@framingui/ui';

export default function FormExample() {
  return (
    <form className="grid w-full max-w-sm gap-4">
      <div className="grid gap-2">
        <Label htmlFor="name">Name</Label>
        <Input id="name" placeholder="John Doe" required />
      </div>
      <div className="grid gap-2">
        <Label htmlFor="email">Email</Label>
        <Input id="email" type="email" placeholder="john@example.com" required />
      </div>
      <Button type="submit">Submit</Button>
    </form>
  );
}
```
